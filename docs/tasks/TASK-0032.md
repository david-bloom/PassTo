## Product Goal

Move ID.me trust decisions into a backend-owned flow so credential issuance can require verified IAL2 identity without exposing secrets or relying on browser-side token exchange.

## Technical Scope

- Implement the approved `idme-exchange` backend flow from `P2_ENROLLMENT_PIPELINE_MIGRATION_SPEC.md`.
- Wire the Lovable ID.me callback to call the backend exchange with the authenticated nurse JWT.
- Store ID.me verification status, assurance level, and temporary identity fields needed for data matching.
- Block credential issuance for IAL1, failed, missing, or unexpected ID.me results.
- Write audit/event records for identity verification success/failure where supported.

## Out of Scope

- ID.me production credential procurement.
- Twilio verification.
- License lookup and data matching implementation.
- Wallet issuance.

## Acceptance Criteria

- [ ] ID.me code exchange happens only in a trusted backend context.
- [ ] Lovable does not store ID.me client secrets.
- [ ] IAL2 success advances the nurse to phone verification.
- [ ] IAL1/failure/missing data blocks issuance and routes to retry/support behavior.
- [ ] Temporary identity fields needed for matching are not returned to Lovable.
- [ ] Claude documents tests, files changed, deviations, and risks.

## QA Plan

- Test success, IAL1, failed exchange, and missing-field paths with sandbox/test fixtures.
- Verify Supabase state changes for identity status.
- Verify no ID.me secrets are present in frontend code or public env vars.
