## Technical Scope

### Architecture

- Stand up the isolated demo/UAT stack as a parallel copy of the PassTo
  production architecture (Supabase project, Vercel routes, Lovable build) with
  separate credentials, identifiers, domains, and analytics.
- Enforce fail-closed environment isolation at boot and at every Edge Function
  entry: refuse to start or refuse the request if the configuration contains a
  production project ref, production wallet identifier, production provider
  credential, or production domain. Failures must log a structured event and
  return a safe error rather than fall through.
- Drive boot and per-request isolation checks from a checked-in
  non-secret environment manifest (see Environment Isolation Manifest below),
  not from ad-hoc string matching.
- Tag every session record with an immutable `mode = demo | uat` value and
  every storage object with an immutable `environment = demo_uat` value.
  Application code must reject any attempt to mutate either field.

### Schema

- Add a demo/UAT-only schema overlay (separate Postgres schemas or a clearly
  prefixed set of tables) so demo data cannot collide with production-shaped
  rows in the same project.
- Add a `demo_sessions` table keyed by `session_id` with `mode`, `created_at`,
  `created_by`, `presenter_phone_id`, `participant_phone_id`, `selfie_id`,
  `state`, and the immutable `environment` tag.
- Add a `demo_session_participants` table that binds a Supabase Auth user to a
  session: `(session_id, auth_user_id, role, created_at, revoked_at)`.
  - `role` is one of `participant`, `presenter`, `observer`.
  - Used as the join target for every demo RLS policy. RLS policies must
    derive authorization from `auth.uid()` joined to this table, not from a
    client-supplied `session_id`.
  - Role assignments and revocations are audit logged.
- Add a `demo_presenters` allowlist table: `(auth_user_id, granted_at,
  granted_by, revoked_at)`. Presenter-only Edge Functions require both an
  active `demo_session_participants` row with `role = presenter` AND an
  active `demo_presenters` row, so a regular nurse cannot self-assign the
  presenter role.
- Add a `demo_selfie_access_tokens` ledger:
  `(token_hash, session_id, caller_context, issued_at, expires_at,
  consumed_at, consumed_via)`. Tokens are stored as a server-side hash only;
  raw tokens never persisted. See Selfie Short-TTL Delivery Contract.
- Add a `demo_share_tokens` ledger (the share-link counterpart):
  `(token_hash, session_id, issued_at, expires_at, first_used_at,
  revoked_at, verifier_session_id)`. The share token is single-use:
  `first_used_at` is set atomically when the verifier first opens it,
  and the row carries a pointer to the `demo_verifier_sessions` row
  the share-token use spawned. After consumption the share token
  cannot mint another verifier session.
- Add a `demo_verifier_sessions` table:
  `(verifier_session_id, demo_session_id, share_token_hash, opened_at,
  expires_at, closed_at, ip_hash, ua_fingerprint)`. This is the
  bounded server-side session that the verifier holds while viewing
  the credential. TTL is short (default 15 minutes). Subresource
  fetches (selfie, future PDF, future additional images) require an
  active `demo_verifier_sessions` row, not the original share token.
- Add a `demo_entitlements` table keyed by `session_id` that holds the
  server-controlled demo entitlement state for the session. This table is
  the **only** persistent store of demo entitlement state. Demo code is
  prohibited from writing the existing production tables
  `profiles.subscription_tier`, `subscriptions`, `payments`, or
  `stripe_events`, or from calling production payment/entitlement Edge
  Functions. See Demo Entitlement Contract below.
- Add a `demo_recordings` metadata-only table:
  `(recording_id, session_id, mode, captured_at, captured_by,
  storage_location_label, state, redacted_at, deleted_at, attestation_by,
  attestation_at)`.
  - `state` enum: `raw`, `redacted`, `deleted`.
  - `storage_location_label` is a free-text descriptor of where the
    recording physically lives (e.g., "presenter Mac password manager",
    "Zoom cloud", "shared drive folder X"). The recording bytes are not
    persisted in Supabase.
  - Cleanup is a manual attestation pattern; the scheduled job reports
    retention drift from this metadata, not from automatic file inspection.
- Add audit tables for simulator inputs/outputs, OTP delivery, selfie capture,
  wallet issuance, share creation, verifier view, session reset, presenter
  role grants/revocations, and isolation-validator failures.

### RLS

- All demo/UAT tables must have RLS enabled before any row is written.
- Default deny on every table.
- Participant access rules derive from `auth.uid()` joined to
  `demo_session_participants`, never from a client-supplied `session_id`:
  - Authenticated participant: read/write own fields on rows whose
    `session_id` is bound to `auth.uid()` via an active
    `demo_session_participants` row with `role = participant`.
  - Authenticated presenter: read participant-scope fields on bound sessions
    plus presenter-scope mutations gated additionally by an active
    `demo_presenters` row for `auth.uid()`.
- Public verifier access uses no Supabase JWT (see Edge Function
  authorization table). The public verifier path never reads tables
  directly; it goes through Edge Functions that look up a hashed share
  token, validate consumption state, and return only the safe-display
  projection.
- Service role usage stays inside Edge Functions; the demo/UAT frontend
  never receives the service-role key.
- Role assignment, revocation, and presenter allowlist changes are
  audit-logged via a database trigger so RLS bypass attempts via service
  role are traceable.

### Storage

- Create a separate Supabase storage bucket for demo/UAT selfies under the
  demo/UAT project. The bucket name must be distinct from any production
  bucket name (recommend `demo-selfies`).
- Bucket policy denies public reads. Selfie access is mediated by a
  backend-authorized short-TTL contract (see Selfie Delivery Contract).
- Object metadata must include `environment = demo_uat`, `mode`, `session_id`,
  and `captured_at`. Objects without these tags must be rejected by upload
  policy.

### Edge Functions / Backend

Edge Functions are classified by authorization model. Each function declares
one of three models and enforces all guards listed for that model.

**Authenticated functions** (`verify_jwt = true`): require a valid Supabase
JWT, resolve `auth.uid()`, and additionally enforce the
`demo_session_participants` binding (and `demo_presenters` allowlist for
presenter-only functions) before performing any work. Production-shape data
boundaries apply (Demo Entitlement Contract; production tables never written).

**Public-token functions** (`verify_jwt = false`): no Supabase JWT. Caller
must present a token. The function must:

- Hash the supplied token server-side and look it up in the corresponding
  ledger (`demo_share_tokens` or `demo_selfie_access_tokens`).
- Reject if not found, revoked, expired, already consumed, or bound to a
  session in the wrong `mode` / `environment`.
- Enforce per-IP and per-token rate limits.
- Enforce a tight CORS allowlist drawn from the Environment Isolation
  Manifest (only `demo.passtodigital.com` plus the Lovable preview origin
  during implementation).
- Return only the documented safe-display projection. Never expose storage
  paths, raw token hashes, internal IDs, service-role-derived fields, or
  non-display audit data.

**Scheduled/service-role functions:** invoked by cron or other Edge
Functions, not by browsers. Must validate the calling context and load the
isolation manifest before any work.

| Function | Model | Guards |
|---|---|---|

### Selfie Short-TTL Delivery Contract

This is a hard pre-cohort-1 gate (TASK-0073, `CR-0073-03`, `CR2-0073-02`).
The contract uses a streaming proxy with a one-time token ledger. A normal
Supabase signed URL is never exposed to the frontend, because a signed URL
placed in an `<img>` source is reachable from the DOM and is not inherently
single-use after first fetch.

The contract splits into two paths with different authorization but the
same streaming-proxy + ledger primitives.

#### Nurse-app path (single Edge Function, one round trip)

1. The authenticated nurse frontend calls `demo-selfie-fetch` with its
   Supabase JWT.
2. The function validates the caller via `demo_session_participants`,
   then atomically does the following inside one transaction-then-stream:
   - Insert a `demo_selfie_access_tokens` row with `caller_context = 'nurse_app'`, `session_id` derived from the binding,
     `issued_at`, `expires_at = issued_at + 60s`, `consumed_at = now()`.
   - Stream the image bytes from Supabase storage server-to-server with
     `Cache-Control: no-store` and `Content-Disposition: inline`.
3. The frontend `<img src="...demo-selfie-fetch">` calls the function
   directly. Each render attempt produces a new consumed-on-issue ledger
   row, so replay forensics are intact.
4. Acceptable alternative for the nurse-app path: function returns
   image bytes; frontend uses `URL.createObjectURL(blob)` and revokes
   immediately after `onload`. Ledger row still required.

#### Verifier path (two Edge Functions, share token then selfie token)

The share token is consumed once on first verifier page open and mints
a bounded verifier session token. The selfie subresource is fetched with
a separate, single-use selfie token scoped to that verifier session. Use
of the original share token outside the first `demo-verifier-view` call
is rejected.

1. The verifier opens the share link
   `https://demo.passtodigital.com/v/<raw_share_token>`.
2. The frontend calls `demo-verifier-view` with the raw share token.
3. `demo-verifier-view` hashes the share token, looks it up in
   `demo_share_tokens`, and atomically (single transaction):
   - rejects if not found, revoked, expired, or already consumed
     (`first_used_at IS NOT NULL`);
   - marks `first_used_at = now()`;
   - inserts a `demo_verifier_sessions` row with
     `verifier_session_id`, `demo_session_id`, the share token hash,
     `opened_at`, `expires_at = opened_at + 15 minutes`, `ip_hash`,
     `ua_fingerprint`;
   - writes the `verifier_session_id` back onto the share-token row;
   - inserts a freshly minted `demo_selfie_access_tokens` row with
     `caller_context = 'verifier'`, `verifier_session_id`,
     `session_id` from the demo session, `issued_at`,
     `expires_at = issued_at + 60s`, `consumed_at = NULL`. The raw
     selfie token is returned in the response; only its hash is
     persisted.
4. `demo-verifier-view` returns the safe-display projection plus a
   one-time Edge Function selfie URL of the shape
   `https://<edge>/demo-verifier-view-selfie?st=<raw_selfie_token>`.
   The URL does not contain the share token. The Supabase storage URL
   is never returned.
5. The frontend `<img src="...demo-verifier-view-selfie?st=...">` calls
   `demo-verifier-view-selfie`.
6. `demo-verifier-view-selfie` hashes the selfie token, looks it up in
   `demo_selfie_access_tokens`, and atomically:
   - rejects unless the selfie-token row exists, has not expired, is
     not yet consumed, and is bound to a `demo_verifier_sessions` row
     that is still active (`expires_at > now()`, `closed_at IS NULL`);
   - marks `consumed_at = now()` BEFORE streaming;
   - streams image bytes from storage with `Cache-Control: no-store`.
7. Re-fetch with the same selfie token returns `410 Gone`. Re-use of
   the original share token at `demo-verifier-view` returns `410 Gone`
   because `first_used_at` is set. If the verifier needs another image
   render before `demo_verifier_sessions.expires_at`, the frontend
   asks `demo-verifier-view` for a fresh selfie token via a
   verifier-session-scoped sub-endpoint (`demo-verifier-mint-selfie`)
   that requires the verifier session cookie/header, not the original
   share token. Spec adds this endpoint as an Authenticated-equivalent
   public-token function scoped to the verifier session.

#### Universal rules

- TTL is server-enforced (<=60s for selfie tokens; <=15 min for the
  verifier session) and single-use on the share token and selfie token
  is server-enforced. Client guarantees are not relied upon.
- The Supabase storage URL is never returned to the client under any
  path.
- Missing or deleted selfie returns `Selfie not provided` per
  TASK-0073; the safe-display projection includes that label rather
  than a 404.
- The production verifier safe-display contract is updated in parallel
  and uses the same shape; demo/UAT is the first production-ready
  instance.
- The contract receives Codex security review before Stage A internal
  UAT and re-review before Stage B nurse cohort 1.
- QA must include replay tests covering:
  - Reuse of the raw share token at `demo-verifier-view` after first
    open returns `410 Gone`.
  - Reuse of a copied selfie token returns `410 Gone` after first
    consumption.
  - Selfie fetch with a valid selfie token whose verifier session has
    expired or closed is rejected.
  - Selfie fetch with a selfie token from a different verifier session
    is rejected.
  - Copying the Edge Function selfie URL does not allow a second image
    render.

### Demo Entitlement Contract

Demo/UAT runs do not flow through Stripe or production entitlement code paths.
The contract is explicit and audit-checked.

**Demo code MUST:**

- Read entitlement state only from `demo_entitlements`, keyed by `session_id`.
- Write entitlement state only to `demo_entitlements`.
- Resolve "is this session allowed to issue a credential / install a wallet
  pass / share" via a dedicated `demo-entitlement-resolve` Edge Function (or
  inline equivalent) that touches only `demo_entitlements` and audit tables.

**Demo code MUST NOT:**

- Read or write any row in `profiles.subscription_tier`, `subscriptions`,
  `payments`, `stripe_events`, or any other production-shape entitlement,
  subscription, or payment table.
- Call any production entitlement or payment Edge Function, including any
  Stripe webhook receiver or Stripe-API-touching function.
- Persist a literal `demo` value into the production `subscription_tier`
  column (the value is not a member of the production enum and would be a
  schema regression).
- Emit any Stripe customer, subscription, invoice, or webhook event.

**QA evidence required (production-shape diff):**

- After a full Stage 1 engineering validation run, a production-shape diff
  check must show: zero new rows in the production-shape entitlement,
  subscription, payment, and Stripe tables; zero invocations recorded in
  Stripe Dashboard test or live mode; zero `demo` literal value present
  anywhere in production-shape entitlement columns. The diff result is part
  of the evidence package.

### Cleanup Mechanics

- Participant phone/contact data clears automatically at session close via
  `demo-cleanup-phone`. Cleanup is invocable from the presenter console and
  as a scheduled job; manual SQL is not part of normal operation.
- Recordings are not stored in Supabase. The `demo_recordings` metadata
  table records the existence, physical location, attestation state, and
  redaction/deletion timestamps of every session recording. Cleanup of the
  recording bytes is a manual operation performed by the recording redactor
  role (see TASK-0073 Session Ownership) and attested in the table.
- Raw recordings retention: presenter or recording redactor must mark the
  recording `redacted` or `deleted` in `demo_recordings` within 30 days of
  capture. The scheduled `demo-retention-report` Edge Function reports any
  `state = raw` row older than 30 days as a drift event.
- Redacted recordings/clips retention: must be marked `deleted` in
  `demo_recordings` or carry a renewed `attestation_at` timestamp within
  12 months. Drift report includes both conditions.
- Selfies: retain until replaced or manually deleted at participant
  request. The selfie token ledger is purged of consumed rows older than
  the operational audit window (default 90 days).
- The retention drift report is delivered to David and Codex on a weekly
  cadence and is part of the evidence package for Stage A and Stage B.

## Codex Architecture and Security Review - 2026-06-16

**Reviewer:** Codex
**Review Type:** Architecture and security review before David execution approval
**Verdict:** Revisions required before execution approval

The TASK-0074 direction is sound and should remain the implementation vehicle
for TASK-0073. However, the current spec has several ambiguity/security gaps
that should be corrected before David grants execution approval. None of these
findings reject the demo/UAT program; they tighten the implementation contract
so Claude, Lovable, and Codex can execute and QA the platform without unsafe
interpretation.

### CR-0074-01 - Selfie short-TTL delivery says "single use" but still returns a URL

**Severity:** P1 - must fix before execution approval
**Area:** Selfie short-TTL delivery contract

The current contract requires a "one-time, short-TTL URL" and says the URL must
not be stored in the DOM or local storage. In normal browser rendering, an image
URL placed in an `<img>` source is present in the DOM, and ordinary Supabase
signed URLs are TTL-bound but not inherently one-time after first fetch. That
creates a spec/implementation mismatch for the hard pre-cohort-1 selfie gate.

**Required correction:** Specify the actual delivery pattern:

- Preferred: backend proxy endpoint streams the image after validating a
  one-time server-side token, then immediately marks the token consumed.
- Acceptable alternative: Edge Function returns image bytes/blob to the
  frontend, and the frontend renders an object URL that is revoked immediately
  after render.

If a Supabase signed URL is used internally, it must remain server-side or be
wrapped by the one-time proxy/token ledger. The QA plan must include replay
tests proving that a copied selfie URL/token cannot be reused after first
render or after TTL expiry.

### CR-0074-02 - Verifier functions cannot all require Supabase JWT

**Severity:** P1 - must fix before execution approval
**Area:** Edge Function authorization model

TASK-0074 says required Edge Functions use `verify_jwt = true` unless justified.
The verifier route is intentionally opened by an external verifier through a
share token; MVP verifiers do not have PassTo accounts or Supabase JWTs. If
`demo-verifier-view` requires Supabase JWT, the verifier demo path will not
work. If it drops JWT without a stricter token model, it risks anonymous data
exposure.

**Required correction:** Explicitly classify each function by auth model:

- Presenter/nurse functions: `verify_jwt = true` plus server-side role/session
  authorization.
- Public verifier token functions: `verify_jwt = false`, but require hashed
  token lookup, status/expiry/first-use checks, environment/mode checks,
  rate limiting, safe CORS, and safe-display projection only.

The public verifier functions must never expose storage paths, raw token
hashes, internal IDs, service-role-derived fields, or non-display audit data.

### CR-0074-03 - RLS needs explicit session-to-user binding, not only `session_id`

**Severity:** P1 - must fix before execution approval
**Area:** Schema, RLS, presenter authorization

The RLS section allows authenticated participants to read/write rows tagged
with their own `session_id`, but the spec does not define how a Supabase Auth
user is cryptographically bound to that `session_id`. If the client can supply
or guess a session ID, RLS based only on `session_id` is not a security
boundary. The presenter console is also described as service-role-only backend
work, but it still needs an authenticated presenter authorization model.

**Required correction:** Add explicit tables/claims:

- `demo_session_participants(session_id, auth_user_id, role, created_at,
  revoked_at)` or equivalent.
- RLS must join `auth.uid()` to the participant/session binding, not trust a
  client-provided session ID.
- Presenter actions must require an authenticated presenter allowlist or custom
  claim before invoking service-role work.
- Session ownership and role assignments must be audit logged.

### CR-0074-04 - Environment isolation needs an allowlist manifest and negative test artifact

**Severity:** P1 - must fix before execution approval
**Area:** Environment isolation / fail-closed boot validation

The fail-closed requirement is correct, but implementation teams need a precise
source of truth for allowed demo values and disallowed production values.
Without an explicit manifest, boot validation can drift into string matching
or miss a production endpoint.

**Required correction:** Require a checked-in non-secret environment manifest
or config contract listing:

- allowed demo Supabase project ref: `atnmcjkjshyqcttnmzkq`
- allowed demo domain/origin: `demo.passtodigital.com`
- allowed Lovable preview origin during implementation only
- allowed demo Apple pass type ID
- allowed demo Google project/service account/business profile/issuer/class
  identifiers as they become available
- disallowed production project refs, domains, wallet IDs, and provider
  identifiers

The acceptance criteria must require a negative-test evidence packet showing
the app/functions refuse to boot or refuse requests when production refs,
domains, wallet IDs, or provider credentials are injected.

### CR-0074-05 - Demo entitlement bypass needs a separate contract and explicit non-use of production functions

**Severity:** P1 - must fix before execution approval
**Area:** Demo entitlement bypass / production subscription-tier contract

The spec creates `demo_entitlements`, which is the right direction, but it also
says demo share and wallet functions mirror production semantics. That leaves
room for a developer to reuse production entitlement readers or write demo
values into production-shaped `profiles.subscription_tier` state.

**Required correction:** Add a demo entitlement contract:

- Demo/UAT entitlement checks read only `demo_entitlements` by `session_id`.
- Demo code must not write `profiles.subscription_tier`, `subscriptions`,
  `payments`, or `stripe_events`.
- Demo code must not call production payment/entitlement Edge Functions.
- QA must include a production-shape diff proving no `demo` tier value, fake
  subscription row, fake payment row, or Stripe event is created.

### CR-0074-06 - Source reconciliation must gate verifier/selfie contract exposure

**Severity:** P1 - must fix before execution approval
**Area:** Source reconciliation gating

TASK-0074 correctly lists the five source docs that conflict with the new
selfie/verifier direction. The current wording gates reconciliation before
Stage B, but the implementation also says the production safe-display contract
must be updated in parallel. If the external verifier/selfie shape is built
before the source docs are reconciled and approved, Claude/Lovable may build
against a contract that security docs still forbid.

**Required correction:** Split execution sequencing:

- Engineering-only Stage 1 may build isolated internal prototypes and
  fail-closed tests.
- Any nurse-visible or verifier-visible selfie display, organization/purpose
  capture, or production-shape safe-display contract must wait until the five
  source docs are reconciled, Codex-reviewed, and David-approved.
- Stage A may not be treated as product acceptance for the verifier/selfie
  contract until reconciliation is complete.

### CR-0074-07 - Recording/retention controls lack a concrete data location

**Severity:** P2 - should fix before execution approval
**Area:** Retention and cleanup

The spec requires scheduled retention drift reports for raw recordings and
redacted clips, but it does not define where those recordings/clips live or how
the scheduled job can inspect them. Native phone recordings, Zoom recordings,
and local files will not automatically be visible to a Supabase scheduled job.

**Required correction:** Define the recording evidence location and retention
authority:

- either store recording metadata in a `demo_recordings` table and make the
  scheduled job report on metadata/attestation,
- or store recordings/clips in a controlled bucket/location the cleanup job can
  inspect,
- or explicitly make recording deletion a manual post-session checklist item
  with required attestation instead of an automated cleanup claim.
