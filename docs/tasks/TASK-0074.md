## Technical Scope

### Architecture

- Stand up the isolated demo/UAT stack as a parallel copy of the PassTo
  production architecture (Supabase project, Vercel routes, Lovable build) with
  separate credentials, identifiers, domains, and analytics.
- Enforce fail-closed environment isolation at boot and at every Edge Function
  entry: refuse to start or refuse the request if the configuration contains a
  production project ref, production wallet identifier, production provider
  credential, or production domain. Failures must log a structured event and
  return a safe error rather than fall through.
- Tag every session record with an immutable `mode = demo | uat` value and
  every storage object with an immutable `environment = demo_uat` value.
  Application code must reject any attempt to mutate either field.

### Schema

- Add a demo/UAT-only schema overlay (separate Postgres schemas or a clearly
  prefixed set of tables) so demo data cannot collide with production-shaped
  rows in the same project.
- Add a `demo_sessions` table keyed by `session_id` with `mode`, `created_at`,
  `created_by`, `presenter_phone_id`, `participant_phone_id`, `selfie_id`,
  `state`, and the immutable `environment` tag.
- Add a `demo_entitlements` table that holds the server-controlled demo
  entitlement state per session. This must not persist a `demo` value into the
  production `subscription_tier` contract.
- Add audit tables for simulator inputs/outputs, OTP delivery, selfie capture,
  wallet issuance, share creation, verifier view, and session reset.

### RLS

- All demo/UAT tables must have RLS enabled before any row is written.
- Default deny. Allow only:
  - Authenticated participant: read/write rows tagged with their own
    `session_id` and limited fields.
  - Presenter (David) via service role on backend-only Edge Functions.
  - Verifier short-TTL token: read-only access to the safe-display projection.
- Service role usage stays inside Edge Functions; the demo/UAT frontend never
  receives the service-role key.

### Storage

- Create a separate Supabase storage bucket for demo/UAT selfies under the
  demo/UAT project. The bucket name must be distinct from any production
  bucket name (recommend `demo-selfies`).
- Bucket policy denies public reads. Selfie access is mediated by a
  backend-authorized short-TTL contract (see Selfie Delivery Contract).
- Object metadata must include `environment = demo_uat`, `mode`, `session_id`,
  and `captured_at`. Objects without these tags must be rejected by upload
  policy.

### Edge Functions / Backend

Required Edge Functions in the demo/UAT project, all using `verify_jwt = true`
unless explicitly justified:

- `demo-session-prepare` - allocates a new `session_id`, generates `mode`,
  clears prior participant phone and contact data, restores synthetic license
  fixture, runs sync verification, and returns presenter-console state.
- `simulator-identity` - returns the acknowledged simulated identity result
  after the acknowledgment event is recorded. Never imitates ID.me branding.
- `simulator-license` - accepts any reasonable alphanumeric license number,
  echoes it into the synthetic record, and labels the result synthetic.
- `demo-selfie-upload` - validates `mode`, `session_id`, and consent flags,
  writes the storage object with required metadata.
- `demo-selfie-fetch` - backend-authorized short-TTL retrieval (see Selfie
  Delivery Contract). Verifier path uses an issued short-TTL token; nurse-app
  path uses the authenticated nurse session.
- `demo-wallet-issue` - mirrors production `wallet-issue` but signs with the
  demo Apple/Google identifiers, writes to demo `wallet_passes` table, and
  enforces unique souvenir serial/object IDs per session.
- `demo-share-create` - mirrors production share semantics with short-lived,
  first-use behavior and `environment = demo_uat`.
- `demo-verifier-view` - returns the safe-display projection including the
  short-TTL selfie URL; never exposes storage paths or the durable signed URL.
- `demo-session-reset` - per `Prepare New Demo` runbook step 5; preserves
  session-scoped audit history.
- `demo-cleanup-phone` - cleans participant phone/contact data at session
  close. Must be invocable from the presenter console and as a scheduled
  job; must not require manual SQL.

### Identity and License Simulators

- Identity simulator returns a deterministic `simulated_success` result tied
  to `session_id` after the participant or presenter has acknowledged the
  simulation copy approved in TASK-0073.
- License simulator returns an `Active - Simulated` record echoing the
  participant-entered number. Reserved sentinel values may produce
  deterministic failure scenarios for engineering testing only and are not
  used in moderated cohort sessions.
- Simulators record the inputs, outputs, and acknowledgment event so the
  evidence package can show simulation provenance.

### Selfie Short-TTL Delivery Contract

This is a hard pre-cohort-1 gate (TASK-0073, `CR-0073-03`, `CR2-0073-02`).

- The frontend (nurse app and verifier) never receives a durable signed URL or
  storage path for any selfie.
- The frontend calls `demo-selfie-fetch` (nurse-app path) or the verifier
  Edge Function (verifier path) with the appropriate auth context.
- The Edge Function validates the caller, generates a one-time, short-TTL URL
  (target: <=60 seconds, single use), and returns the URL to the caller.
- The URL is consumed immediately by the rendering surface and not stored in
  the DOM or local storage.
- Missing or deleted selfie returns `Selfie not provided` per TASK-0073.
- The production verifier safe-display contract must be updated in parallel so
  the same mechanism can support production with no change in shape; demo/UAT
  is the first production-ready instance of this contract.
- The contract receives a Codex security review before Stage A internal UAT
  and a re-review before Stage B nurse cohort 1.

### Frontend (Lovable scope)

Driven by specs from Claude/Codex; Lovable executes.

- Demo/UAT login and acknowledgment screens with approved copy.
- Simulated-identity acknowledgment screen replacing the production ID.me
  button as approved in TASK-0073.
- License entry screen accepting any alphanumeric value; no instruction to
  randomize unless asked, per the UAT protocol.
- OTP entry, selfie capture, credential review, wallet install, share
  creation, verifier entry, verifier view, and nurse activity view, all
  consuming demo/UAT backend contracts.
- Permanent demo treatment, labels, and copy required by TASK-0073 and the
  UAT protocol.
- Presenter console screens for `Prepare New Demo`, session reset, fallback
  artifacts, and cleanup, gated by an authenticated presenter role.

### Cleanup Mechanics

- Participant phone/contact data clears automatically at session close via
  `demo-cleanup-phone`.
- Raw recordings retention: delete within 30 days of capture (TASK-0073
  retention baseline).
- Redacted recordings/clips retention: up to 12 months, then review for
  deletion or renewed need.
- Selfies: retain until replaced or manually deleted at participant request.
- A scheduled job verifies retention bounds and reports drift.
