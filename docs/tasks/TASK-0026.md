#### Migration A — Remove `p_phone` from `update_own_profile_basic()`

**Caller-impact check (required before applying):** Search the Lovable P2 and P3 enrollment project source for any call to `update_own_profile_basic` that passes `p_phone`. If found, remove or replace the caller before applying this migration. The Phase 2 flow is expected to be safe (uses only `p_first_name` and `p_last_name`), but this must be confirmed in the live Lovable codebase before applying.

```sql
-- Migration A: Remove p_phone from update_own_profile_basic()
-- Purpose: profiles.phone becomes backend-controlled; only phone-verify-otp (service-role) may write it.
-- Risk: Any code calling update_own_profile_basic with p_phone will break. Perform caller-impact check first.
-- Expected safe for Phase 2 flow: only p_first_name and p_last_name are passed in production.

DROP FUNCTION IF EXISTS public.update_own_profile_basic(text, text, text, text, text, jsonb);

CREATE OR REPLACE FUNCTION public.update_own_profile_basic(
  p_first_name              text    DEFAULT NULL::text,
  p_last_name               text    DEFAULT NULL::text,
  p_display_name            text    DEFAULT NULL::text,
  p_preferred_name          text    DEFAULT NULL::text,
  p_notification_preferences jsonb  DEFAULT NULL::jsonb
)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $function$
declare
  v_profile_id uuid;
begin
  select id into v_profile_id
  from public.profiles
  where auth_user_id = auth.uid();

  if v_profile_id is null then
    raise exception 'Profile not found for current user';
  end if;

  update public.profiles
  set
    first_name               = coalesce(p_first_name,               first_name),
    last_name                = coalesce(p_last_name,                last_name),
    display_name             = coalesce(p_display_name,             display_name),
    preferred_name           = coalesce(p_preferred_name,           preferred_name),
    notification_preferences = coalesce(p_notification_preferences, notification_preferences),
    updated_at               = now()
  where id = v_profile_id;
end;
$function$;
```

#### Migration C — Atomic phone verification completion function

```sql
-- Migration C: Create complete_phone_verification() — atomic profile update + audit insert
-- Called by phone-verify-otp Edge Function (service-role) on OTP approved.
-- Both UPDATE and INSERT execute in a single transaction; either both commit or both roll back.

CREATE OR REPLACE FUNCTION public.complete_phone_verification(
  p_profile_id uuid,
  p_phone      text
)
RETURNS void
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path TO 'public'
AS $function$
begin
  -- Advance profile: set phone and onboarding_step = 'license'
  -- Guard: only update if currently in 'phone' step (idempotency-safe)
  UPDATE public.profiles
  SET
    phone           = p_phone,
    onboarding_step = 'license',
    updated_at      = now()
  WHERE id = p_profile_id
    AND onboarding_step = 'phone';

  IF NOT FOUND THEN
    -- Profile not in 'phone' step — may already be at 'license' (idempotency)
    -- Caller handles this via idempotency check before calling this function
    RAISE EXCEPTION 'profile_not_in_phone_step: %', p_profile_id;
  END IF;

  -- Write audit row in same transaction
  INSERT INTO public.audit_events (actor_id, action, resource_type, resource_id, change_after)
  VALUES (
    p_profile_id,
    'phone.verified',
    'profile',
    p_profile_id,
    jsonb_build_object('onboarding_step', 'license')
    -- phone omitted from change_after to avoid PII in audit log
  );
END;
$function$;
```

## Twilio Service Choice

**Use Twilio Verify** (not Twilio Programmable SMS).

Twilio Verify handles OTP code generation, delivery, expiry (10 minutes default), and validation server-side. PassTo does not manage code storage, hashing, or expiry logic.

Twilio Verify rate limits apply automatically: 5 incorrect attempts before the verification is cancelled; 1 pending verification per phone number at a time.

## Edge Function: `phone-send-otp`

### Purpose

Accepts a phone number from the nurse, sends an OTP via Twilio Verify, and writes a `notification_events` row.

### Input

| Field | Type | Source | Required |
|---|---|---|---|
| `phone` | string | Nurse-entered phone number from Lovable | Yes |
| JWT | string | Auto-sent in Authorization header by Supabase client | Yes |

Phone must be provided in E.164 format (`+1XXXXXXXXXX`). If Lovable collects a raw number, it must normalize to E.164 before calling the Edge Function. If the number fails E.164 validation, the Edge Function returns `{ sent: false, error: 'invalid_phone' }`.

### Process

```
1. Validate input: phone must be present, non-empty, E.164 format
2. Verify JWT → auth_user_id
3. Look up profile: .eq('auth_user_id', user.id).maybeSingle()
   if not found → return 404 { error: 'profile_not_found' }
4. Verify onboarding_step = 'phone'
   if not 'phone' → return 400 { error: 'invalid_step' }
   (prevents OTP spam if nurse has already advanced)
   Also verify id_verification_status = 'verified' AND id_verification_level = 'IAL2'
   if either check fails → return 400 { error: 'identity_not_verified' }
   (defensive: nurse must have completed Phase 3.1 before reaching Phase 3.2)
5. Call Twilio Verify:
     POST https://verify.twilio.com/v2/Services/{TWILIO_VERIFY_SERVICE_SID}/Verifications
       To=<phone>&Channel=sms
     Auth: Basic <TWILIO_ACCOUNT_SID>:<TWILIO_AUTH_TOKEN>
   if Twilio error → log, write notification_events (status='failed'), return 200 { sent: false, error: 'provider_error' }
6. Write notification_events row (service-role):
     profile_id:          profile.id
     related_entity_type: 'phone_verification'  (Migration B required)
     channel:             'sms'
     provider:            'twilio'
     direction:           'outbound'
     status:              'sent'
     external_message_id: Twilio Verification SID (not MessageSID — Verify uses VerificationSID)
7. Write audit_events row:
     action:        'phone.otp_sent'
     actor_id:      profile.id
     resource_type: 'profile'
     resource_id:   profile.id
     change_after:  { phone_e164: <last 4 digits masked>, channel: 'sms' }
8. Return 200 { sent: true }
```

### Failure Cases and Return Values

| Condition | HTTP status | Body |
|---|---|---|
| `phone` missing or invalid E.164 | 400 | `{ error: 'invalid_phone' }` |
| JWT invalid or missing | 401 | `{ error: 'unauthenticated' }` |
| Profile not found | 404 | `{ error: 'profile_not_found' }` |
| `onboarding_step` is not `'phone'` | 400 | `{ error: 'invalid_step' }` |
| `id_verification_status` ≠ `'verified'` or `id_verification_level` ≠ `'IAL2'` | 400 | `{ error: 'identity_not_verified' }` |
| Twilio Verify error | 200 | `{ sent: false, error: 'provider_error' }` |
| Twilio rate limit hit | 200 | `{ sent: false, error: 'rate_limited' }` |
| Success | 200 | `{ sent: true }` |

## Edge Function: `phone-verify-otp`

### Purpose

Accepts the nurse's phone number and OTP code, checks with Twilio Verify, and on approval writes `profiles.phone` and advances `onboarding_step` to `'license'`.

### Input

| Field | Type | Source | Required |
|---|---|---|---|
| `phone` | string | Same E.164 phone submitted to `phone-send-otp` | Yes |
| `code` | string | 6-digit OTP entered by nurse | Yes |
| JWT | string | Auto-sent in Authorization header by Supabase client | Yes |

### Process

```
1. Validate input: phone (E.164) and code (6 digits) must be present
2. Verify JWT → auth_user_id
3. Look up profile: .eq('auth_user_id', user.id).maybeSingle()
   if not found → return 404 { error: 'profile_not_found' }
4. Verify onboarding_step = 'phone'
   if not 'phone' → check idempotency (see below)
   Also verify id_verification_status = 'verified' AND id_verification_level = 'IAL2'
   if either check fails → return 400 { error: 'identity_not_verified' }
5. Check code with Twilio Verify:
     POST https://verify.twilio.com/v2/Services/{TWILIO_VERIFY_SERVICE_SID}/VerificationCheck
       To=<phone>&Code=<code>
     Auth: Basic <TWILIO_ACCOUNT_SID>:<TWILIO_AUTH_TOKEN>
6. Evaluate result:
   - status = 'approved':
       a. Call complete_phone_verification(profile.id, phone) RPC (Migration C — atomic)
          This updates profiles.phone + onboarding_step = 'license' AND inserts audit_events
          in a single Postgres transaction. Either both commit or neither does.
          If RPC throws → return 200 { verified: false, error: 'provider_error' }
          (Profile remains in 'phone' step; nurse can retry)
       b. Return 200 { verified: true }
   - status = 'pending' (wrong code, attempts remain):
       a. Write audit_events: action 'phone.otp_failed' (swallow audit error — do not block)
       b. Return 200 { verified: false, error: 'invalid_code' }
   - status = 'canceled' (max attempts exceeded or expired):
       a. Write audit_events: action 'phone.otp_expired' (swallow audit error)
       b. Return 200 { verified: false, error: 'code_expired' }
   - Twilio error / unexpected:
       a. Write audit_events: action 'phone.otp_failed' (swallow audit error)
       b. Return 200 { verified: false, error: 'provider_error' }
```

### Idempotency

If `phone-verify-otp` is called after the nurse is already at `onboarding_step = 'license'`:
- If `profiles.phone` matches the submitted `phone` → return `{ verified: true }` immediately without re-calling Twilio
- If `profiles.phone` does NOT match the submitted `phone` → return `{ verified: false, error: 'already_verified_different_phone' }` (prevents phone substitution after verification)
- This handles re-submission of the verify page

### Failure Cases and Return Values

| Condition | HTTP status | Body |
|---|---|---|
| `phone` or `code` missing/invalid | 400 | `{ error: 'invalid_input' }` |
| JWT invalid or missing | 401 | `{ error: 'unauthenticated' }` |
| Profile not found | 404 | `{ error: 'profile_not_found' }` |
| `onboarding_step` not `'phone'` and not `'license'` | 400 | `{ error: 'invalid_step' }` |
| `id_verification_status` ≠ `'verified'` or `id_verification_level` ≠ `'IAL2'` | 400 | `{ error: 'identity_not_verified' }` |
| Already at `'license'`, phone matches | 200 | `{ verified: true }` |
| Already at `'license'`, phone does NOT match | 200 | `{ verified: false, error: 'already_verified_different_phone' }` |
| Wrong OTP code (attempts remain) | 200 | `{ verified: false, error: 'invalid_code' }` |
| Code expired / max attempts exceeded | 200 | `{ verified: false, error: 'code_expired' }` |
| Twilio error | 200 | `{ verified: false, error: 'provider_error' }` |
| `complete_phone_verification()` RPC throws (atomic rollback) | 200 | `{ verified: false, error: 'provider_error' }` |

## Lovable Changes: `/verify-phone` Page

### Current state

`/verify-phone` is a placeholder (no Twilio wiring). It exists as a route in App.tsx (expected from Phase 2 routing).

### Two-step UI

**Step 1 — Phone collection:**
- Input: phone number (free text; Lovable must format to E.164 before calling Edge Function)
- Button: "Send Code"
- On submit: call `supabase.functions.invoke("phone-send-otp", { body: { phone } })`
- On `{ sent: true }`: advance to Step 2
- On `{ sent: false, error: 'rate_limited' }`: show rate-limit message; offer retry after delay
- On `{ sent: false, error: 'invalid_phone' }`: show inline validation error
- On `{ sent: false, error: 'provider_error' }`: show provider-unavailable message; offer retry

**Step 2 — OTP entry:**
- Input: 6-digit code
- Button: "Verify Code"
- On submit: call `supabase.functions.invoke("phone-verify-otp", { body: { phone, code } })`
- On `{ verified: true }`: navigate to `/license-lookup`
- On `{ verified: false, error: 'invalid_code' }`: show "Incorrect code" inline error; allow retry
- On `{ verified: false, error: 'code_expired' }`: show "Code expired" message; offer to re-send (back to Step 1)
- On `{ verified: false, error: 'provider_error' }`: show provider error; offer retry
- "Resend code" option: returns to Step 1 (calls `phone-send-otp` again)

### Phone number formatting

Lovable must normalize the nurse-entered phone to E.164 before passing to the Edge Function.  
For US numbers: strip non-digits, prepend `+1` if 10 digits.  
If the number cannot be normalized to a valid E.164 format, show inline validation before calling the Edge Function.

### Route guard

Before rendering the phone entry form, call `supabase.from('profiles').select('onboarding_step').maybeSingle()`. If `onboarding_step` is not `'phone'`, call `routeByOnboardingStep()` to re-route appropriately. This prevents a nurse from reaching `/verify-phone` out of sequence.

## Audit Event Requirements

All audit events written by Edge Functions (service-role). Format: `resource.verb` (OD-1).

| Trigger | `action` | `change_after` |
|---|---|---|
| OTP sent successfully | `phone.otp_sent` | `{ channel: 'sms' }` (phone masked) |
| OTP send failure | `phone.otp_failed` | `{ error_key: 'provider_error' }` |
| OTP verified (approved) | `phone.verified` | `{ onboarding_step: 'license' }` |
| OTP wrong code | `phone.otp_failed` | `{ error_key: 'invalid_code' }` |
| OTP expired / cancelled | `phone.otp_expired` | `{ error_key: 'code_expired' }` |

`actor_id` and `resource_id` are `profile.id` for all rows. Write `phone.otp_failed` for all profile-known failures after step 3.

**Phone number must not appear in `change_after` or any audit log field.** Log only masked indicator (e.g. last 4 digits) or omit entirely.
