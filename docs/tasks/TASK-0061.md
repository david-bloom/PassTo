## Product Goal

Give nurses a safe MVP path for subscription management and cancellation without letting Lovable alter payment truth directly.

## Technical Scope

- Decide whether MVP subscription management uses Stripe Customer Portal or a documented manual support path.
- If Customer Portal is approved, specify the backend function or route that creates portal sessions.
- Ensure cancellation/lapse behavior remains webhook-driven.
- Define Lovable dashboard/account behavior for active, canceled, past_due, unpaid, and lapsed subscriptions.
- Document user-facing copy for "manage subscription" and cancellation/lapse states.
- Document any Stripe configuration required before production.

- Stripe live-mode cutover.

## Codex QA Review

**Date:** 2026-06-04
**Reviewer:** Codex
**Result:** Blocked - needs Claude remediation before this task can pass QA.

### Findings

- **P1 - `subscription-portal-session` queries a dropped MVP table.** The function reads from `stripe_customers` at `supabase/functions/subscription-portal-session/index.ts:95-99`, but ADR-0001 states that `stripe_customers` was not created for MVP and implementations must use `profiles.stripe_customer_id`, `subscriptions`, `payments`, and `stripe_events`. Result: portal-session creation will fail against the approved v4 schema instead of returning a portal URL.
- **P1 - `dashboard-status` also derives `can_manage_subscription` from the dropped `stripe_customers` table.** The dashboard function reads `stripe_customers` at `supabase/functions/dashboard-status/index.ts:91-95`. Because that table is not part of the accepted MVP schema, `can_manage_subscription` is not reliable and may produce backend errors or false UI state.
- **P1 - subscription payment rows use action types outside the v4 schema constraint.** `stripe-webhook` inserts `payments.action_type = "subscription_start"` at `supabase/functions/stripe-webhook/index.ts:219-222` and `"subscription_renewal"` at `supabase/functions/stripe-webhook/index.ts:377-380`; v4 allows only `share_token`, `refresh`, `pdf_export`, and `additional_license` in `docs/architecture/V4_MIGRATION_SQL.md:524-525`. This means the documented payment history path is not actually supported by the current schema/function pair.

### Required Remediation

- Replace `stripe_customers` reads with `profiles.stripe_customer_id` or a documented schema-approved source.
- Recompute `can_manage_subscription` from the approved MVP schema.
- Reconcile subscription payment event recording with the approved `payments.action_type` constraint, either by updating the schema through an approved migration or by writing schema-valid payment/action records.
- Re-run source QA and, if available, live/test-mode portal-session and dashboard checks.
