## Product Goal

Give nurses a safe MVP path for subscription management and cancellation without letting Lovable alter payment truth directly.

## Technical Scope

- Decide whether MVP subscription management uses Stripe Customer Portal or a documented manual support path.
- If Customer Portal is approved, specify the backend function or route that creates portal sessions.
- Ensure cancellation/lapse behavior remains webhook-driven.
- Define Lovable dashboard/account behavior for active, canceled, past_due, unpaid, and lapsed subscriptions.
- Document user-facing copy for "manage subscription" and cancellation/lapse states.
- Document any Stripe configuration required before production.

## Out of Scope

- Stripe live-mode cutover.
- Live product/price changes.
- Annual plans, coupons, trials, refunds, disputes, institutional billing, employer billing.
- Direct Lovable writes to payment/subscription truth.

## Claude Implementation Spec - 2026-06-04

**Deliverable:** `docs/tasks/TASK-0061-SUBSCRIPTION-MANAGEMENT-SPEC.md`

**Summary:**
- **Approach:** Stripe Customer Portal (secure, standard, safe)
- **Backend:** New `subscription-portal-session` function (service-role only, no secrets exposed)
- **Lapse behavior:** Webhook-driven immediate downgrade to Free (credentials/wallet preserved)
- **Dashboard states:** Documented copy for active, past_due, canceled, unpaid, incomplete
- **Lovable updates:** Subscription card with "Manage Subscription" button → portal session
- **Stripe config:** Listed as hard gate before production

**Key Decisions:**
- Customer Portal for MVP (vs manual support path) — safer, standard, requires no PassTo infrastructure beyond function
- Immediate downgrade on cancellation (Option A) — simpler than waiting for period end; free features available immediately
- Portal session function is service-role only — no Stripe secrets in Lovable

**Files to create/modify:**
- `subscription-portal-session` (new Edge Function)
- `stripe-webhook` (update lapse handling)
- `dashboard-status` (update subscription status fields)
- Lovable `/dashboard` (update subscription card UI)

**Tests to run:**
- Portal session creation for valid/invalid users
- Subscription canceled → profile downgraded
- Credential remains active after downgrade
- Dashboard reflects correct status and copy

**Risks identified:**
- Stripe portal not enabled in test account (LOW — verify in dashboard)
- Webhook failure to downgrade (MEDIUM — Phase 7 reconciliation job mitigates)
- User expects refund (MEDIUM — clear dashboard copy; support owns refunds)

**Deviations:** None; approach aligns with task scope and PRD.
