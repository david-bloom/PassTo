## Product Goal

Give nurses a safe MVP path for subscription management and cancellation without letting Lovable alter payment truth directly.

## Technical Scope

- Decide whether MVP subscription management uses Stripe Customer Portal or a documented manual support path.
- If Customer Portal is approved, specify the backend function or route that creates portal sessions.
- Ensure cancellation/lapse behavior remains webhook-driven.
- Define Lovable dashboard/account behavior for active, canceled, past_due, unpaid, and lapsed subscriptions.
- Document user-facing copy for "manage subscription" and cancellation/lapse states.
- Document any Stripe configuration required before production.

## Out of Scope

- Stripe live-mode cutover.
- Live product/price changes.
- Annual plans, coupons, trials, refunds, disputes, institutional billing, employer billing.
- Direct Lovable writes to payment/subscription truth.

## Routes Affected

- `/dashboard`
- Account/subscription management route if created.

## Components Affected

- Lovable dashboard/account subscription controls.
- Optional backend portal-session function or route.
- Stripe webhook state readers.

## Database/Tables Affected

- `subscriptions`
- `payments`
- `stripe_events`
- `audit_events`

## Integrations Affected

- Stripe Customer Portal if approved.
- Stripe webhooks.
- Supabase Edge Functions or Vercel route if needed.

## Security/RLS Impact

Security Impact: High

Subscription status and entitlement must remain backend-derived. Lovable may request a portal session or show status, but must not write subscription state.

## Acceptance Criteria

- [ ] MVP subscription management path is documented and approved.
- [ ] Cancellation/lapse behavior is webhook-driven and does not delete credentials or wallet records.
- [ ] Lovable dashboard/account states are specified for active, canceled, past_due, unpaid, and lapsed subscriptions.
- [ ] Any backend portal-session route is service-side and does not expose Stripe secrets.
- [ ] Any required Stripe configuration is documented as a David hard gate before production.
- [ ] Claude documents files changed, tests run, deviations, and risks.

## QA Plan

- Review source/prompt/route spec for direct payment-truth writes.
- Verify customer portal session creation, if implemented, returns only safe URL data.
- Verify cancellation/lapse webhook events continue to drive state changes.
- Verify dashboard copy for lapsed/unpaid states does not imply paid entitlement remains active.
