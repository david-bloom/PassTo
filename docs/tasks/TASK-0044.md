## Product Goal

Create safe, repeatable dev-only test accounts that David, Codex, and Claude can use to log in and exercise the MVP journey: account access, onboarding states, pass generation, dashboard review, share-link creation, verifier viewing, refresh behavior, entitlement boundaries, and suspended/closed account handling.

This task must make test access reliable without introducing production data, real nurse PII, real credentials, or committed secrets.

## Required Approval State

David approval is required before Claude executes this task.

Execution may create Supabase Auth users, profile rows, credential records, wallet pass rows, subscription/payment state, verification token test state, or seed scripts. Those are security-sensitive changes and must not proceed until David explicitly approves the task in GitHub or chat and the approval is recorded in GitHub.

## Technical Scope

- Create an idempotent dev-only seed harness, such as `scripts/seed-dev-test-personas.ts`, or the equivalent established project pattern.
- Seed Supabase Auth users server-side through the Admin API or Supabase dashboard/API using privileged credentials only in a trusted local/server context.
- Never expose `service_role`, secret keys, seed passwords, or privileged credentials in Lovable/client code or committed files.
- Seed matching application rows required for the personas, including the appropriate subset of:
  - `profiles`
  - `licenses`
  - `credentials`
  - `wallet_passes`
  - `subscriptions`
  - `payments`
  - `license_lookups`
  - `verification_tokens` only when needed and only with hashed/stored-token-safe behavior
  - audit/event rows used by ops and failure-state review
- Ensure the harness refuses to run unless it is pointed at the approved development Supabase project.
- Use fake/sandbox data only. Do not use real nurse identities, real license numbers, real payment data, or production wallet credentials.
- Keep generated passwords/secrets out of GitHub. Use local environment variables or generate and display credentials only in the local execution output.
- Provide cleanup/reset instructions or an idempotent reseed mode so repeated QA runs do not corrupt state.

- Account-status enforcement for active, suspended, and closed users.
- Credential/pass-ready dashboard state.
- Wallet pass generation or pass-ready display using sandbox/fake pass data until production signing is approved.
- Share-link creation for entitled accounts.
- Verifier token flow for a test share link once TASK-0041 and TASK-0042 are implemented.
- Manual refresh/status-refresh behavior where implemented. Scheduled automated refresh remains deferred unless David separately reopens it.
- Subscription/entitlement boundaries for free, Standard, Premier, lapsed, suspended, and closed states.
- Ops visibility for all seeded states.

## Security Requirements

- The seed harness must not run against production.
- The seed harness must include an explicit environment/project guard and fail closed on mismatch.
- No real PII, real nurse credentials, real license numbers, live Stripe data, live wallet signing credentials, or production tokens may be used.
- Passwords and raw share tokens must not be committed to GitHub.
- If raw tokens are generated for test use, they must be displayed once locally and stored only as hashes where the product design requires hashing.
- Seeded data must not weaken RLS or bypass account-status enforcement.
- Any `SECURITY DEFINER` function or privileged backend helper introduced for seeding must be isolated from client access and reviewed by Codex before use.

## Out of Scope

- Production test nurse creation.
- Live Stripe payments.
- Live Apple/Google wallet signing unless separately approved.
- Real ID.me, Twilio, RapidAPI, Propelus, or Postmark production calls.
- Scheduled automated refresh implementation.
- Reopening deferred PRD journeys.

## Codex Spec Review — 2026-06-01

**Reviewer:** Codex  
**Review Type:** Pre-implementation spec review  
**Result:** Direction approved with required revisions before execution  

### Summary

TASK-0044 is a good and useful task. Dev personas will make David/Codex/Claude QA much less fragile. It should remain dev-only and may be implemented incrementally, but the current persona list and dependency posture need to be updated for the ID.me-first flow and for safety around privileged seed operations.

### Required Spec Changes Before Approval

#### P1 — Add stronger dev-project and seed-scope guards

This task will use privileged Supabase Admin/service-role capabilities. A mistake here can damage real users or create test data in the wrong project.

Required revision:

- The harness must hard-fail unless the Supabase project ref is exactly the approved dev ref.
- Require an explicit local opt-in such as `PASSTO_ALLOW_DEV_SEED=true` plus an expected project ref env var.
- Refuse to run if any production-looking URL/project ref/env var is detected.
- Default to `--dry-run`; require an explicit `--apply` or equivalent to write.
- Never run from Lovable/client code or unauthenticated public backend routes.
- Do not include seed credentials, service-role keys, raw share tokens, or raw OTP/session tokens in GitHub.

#### P1 — Cleanup/reset must be allowlist-only

The Security Model discourages hard deletes for normal account closure, and the seed harness should not normalize destructive behavior.

Required revision:

- Cleanup/reset may only touch rows carrying a deterministic seed marker, for example `seed_persona_key`, seed email domain, or app metadata marker.
- Use a dedicated fake email domain such as `@passtodigital.test` or another reserved internal test domain.
- Never delete or mutate rows that lack the seed marker.
- If Auth users are deleted during dev cleanup, document that this is dev-seed cleanup only and not an account-closure pattern.

#### P2 — Update personas for the ID.me-first route order

The current required personas still reflect the older phone-before-license mental model:

```text
Identity verified -> phone gate
Phone verified -> license gate
```

ID.me-first now expects:

```text
ID.me/account linked -> license-info -> phone-check -> plan/payment -> selfie -> success
```

Required revision:

- Replace or supplement the persona list with ID.me-first states:
  - New/pre-ID.me account or start state
  - ID.me verified/contact confirmed, license pending
  - License lookup failed
  - License found but identity/license match failed
  - License matched, phone pending
  - Phone verified, plan pending
  - Free selected, selfie pending
  - Paid selected, payment pending
  - Paid active, selfie/pass pending
  - Credential/pass pending
  - Credential/pass ready
  - Suspended account
  - Closed account
- Keep old-flow personas only if they are explicitly for regression testing legacy routes.

#### P2 — Seed only implemented or intentionally stubbed states

TASK-0044 depends on many future tasks. It should not pretend unimplemented provider behavior is real.

Required revision:

- Support incremental seed modes, for example `auth-only`, `onboarding-gates`, `credential-ready`, and `full-smoke`.
- For flows not yet implemented, seed clearly labeled fake/sandbox status rows only.
- Do not create records that imply live Stripe payment, live wallet issuance, live ID.me verification, or live license provider success unless those integrations were actually exercised in sandbox/test mode.
- Mark fake/sandbox rows so QA copy and ops views cannot confuse them with real provider results.

#### P2 — Add deterministic seeded-data identifiers

Required revision:

- Every persona should have a stable key, email, expected route, expected allowed actions, and expected blocked actions.
- Store seed metadata in an approved non-user-editable location where possible, such as Auth app metadata or a seed-only column/table if approved.
- The harness should print a concise persona table after seeding, including email, intended password delivery method, expected route, and state summary.

#### P3 — Add harness ergonomics

Recommended revision:

- Include `--list`, `--dry-run`, `--apply`, and `--reset-seed-data` modes.
- Include a post-seed verification query/check.
- Add a short "How David logs in" section without committing passwords.
- Add a "Known blocked personas" section for personas that depend on TASK-0045/TASK-0046/TASK-0047 or later tasks.

### Approval Guidance

Codex supports executing TASK-0044 after the revisions above are made and David approves. It does not need to wait for every launch feature to be complete if the harness is incremental and honest about which personas are real, fake/sandbox, or blocked by unfinished tasks.
