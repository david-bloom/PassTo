## Product Goal

Ensure subscription, entitlement, payment, and lapse states are visible enough for MVP ops and safe enough that paid features cannot remain active after lapse.

## Technical Scope

- Define the minimum Supabase dashboard/query views or documented inspection checklist for subscription/payment state.
- Confirm ops can inspect profile tier intent, active subscription truth, payment history, webhook events, and failed/unprocessed Stripe events.
- Confirm lapsed/canceled subscriptions remove paid entitlements while preserving credential and wallet records.
- Confirm dashboard/status endpoints expose safe user-facing lapse/paid-state information.
- Identify whether Postmark or another alert path is required for unresolved Stripe webhook failures.
- Document manual recovery boundaries and audit requirements.

## Out of Scope

- Lovable admin UI.
- Deep analytics.
- Stripe live-mode cutover.
- Live product/price changes.
- Production launch.

- `/dashboard`

## Security/RLS Impact

Security Impact: Medium

Ops visibility must not weaken RLS or expose payment details to nurses/verifiers. Any manual intervention must be auditable.

## Acceptance Criteria

- [ ] Ops inspection path exists for profile tier intent, active subscription truth, payments, webhook events, and failed Stripe processing.
- [ ] Lapse/cancellation removes paid entitlements without deleting credential or wallet records.
- [ ] Dashboard/status surfaces safe user-facing subscription/lapse state.
- [ ] Critical unresolved Stripe processing failures have an alerting or documented manual-monitoring path.
- [ ] Manual recovery boundaries and audit requirements are documented.
- [ ] Claude documents files changed, tests run, deviations, and risks.

## QA Plan

- Review Supabase policies/views/queries for safe ops access.
- Test canceled/lapsed subscription state against entitlement readers.
- Test `stripe_events.processed = false` inspection path.
- Verify no client route can restore paid entitlement without Stripe/server truth.
