## Product Goal

Create an honest, repeatable way to test and demonstrate PassTo when fictional
identities cannot complete real ID.me verification and fictional nursing
licenses cannot be looked up through RapidAPI/Propelus.

The program must show the real PassTo product journey without representing
simulated provider results as genuine verification.

## Approved Planning Direction

- Maintain a separate demo/UAT environment with its own Supabase project,
  provider credentials, wallet identifiers, domains, and synthetic data.
- Use two modes on the isolated platform:
  - Demo mode: presenter-controlled Avery Demo golden path.
  - UAT mode: moderated testing, deterministic scenarios, evidence, and review.
- Begin with internal UAT, then David personally recruits nurses for moderated
  UAT using the fictional Avery Demo identity.
- Demonstrate the full journey from onboarding through wallet installation and
  time-limited sharing.
- Use real SMS OTP where available, to David's phone or the participant's phone.
- Require a fresh selfie as part of the trust cycle.
- Use an acknowledgment screen before simulated identity verification.
- Allow any reasonable alphanumeric license number in the synthetic golden path.
- Let participants keep a clearly marked souvenir wallet pass.
- Keep launch-list opt-in separate from demo identity and UAT records.

- Using real nurse licenses or representing synthetic records as authentic.
- Promoting demo/UAT records into production.
- Production launch approval.

## Environment Boundary

The isolated platform must:

- Use a Supabase project that is not the production PassTo project.
- Use separate Apple pass type and Google Wallet class identifiers.
- Use separate credentials, secrets, storage, logs, and analytics.
- Use reserved synthetic email identities and synthetic licenses.
- Reject production project refs, production provider credentials, and
  production wallet identifiers.
- Mark every record with an immutable environment/session classification.
- Prevent any demo/UAT credential from being promoted into production.
- Require real production onboarding before issuing a useful professional pass.

## Canonical Demo Identity

```text
first_name: Avery
last_name: Demo
license identity: synthetic
license number: participant-entered alphanumeric value
default license type: RN
default state: New York
default status: Active - Simulated
```

The participant's or presenter's selfie may represent Avery Demo for the
session. It does not prove that the photographed person is a nurse or that
ID.me verified Avery Demo.

## Demo Artifact Rules

Every portable demo credential must permanently and prominently display:

```text
DEMO
NOT A VALID PROFESSIONAL CREDENTIAL
Avery Demo
Synthetic license data
```

Additional rules:

- No permanent verifier QR or long-lived verification token.
- Unique serial/object ID per demo session so souvenir passes do not overwrite
  one another.
- No conversion or migration into a production credential.
- Share links retain normal short-lived, first-use behavior.
- A general launch-list link may appear in informational pass content, but it
  must not function as credential verification.

## Selfie and Verifier Direction

The agreed target behavior is:

- Selfie remains a required production issuance gate.
- Selfie appears in the PassTo app credential experience.
- Verifier view displays the latest selfie available at access time.
- Verifier copy describes it as a credential-holder-provided current likeness,
  not ID.me biometric verification.
- If a selfie is missing or manually deleted, display `Selfie not provided`.
- A concerned UAT participant may ask David to manually delete the selfie.
- Wallet-pass photo rendering remains a separate implementation decision and is
  not required by this protocol.

## Verifier and Activity Direction

- Demo verifier details may be fictional or real.
- Production verifiers must provide accurate information and accept Terms.
- PassTo must not claim verifier identity is independently verified unless a
  later authentication feature actually does so.
- Nurse activity history should show verifier-provided name, organization,
  purpose, and access time where those fields are approved and implemented.
- Such data must be labeled `Verifier-provided information - not independently verified`.

## Claude Review Comments - 2026-06-15

**Reviewer:** Claude
**Status:** Suggestions and clarifying questions; disposition pending Codex / David
**Scope:** Comments on TASK-0073, the UAT Protocol, and the Live Demo Runbook as
a whole. Doc-specific items are also captured in the respective files.

### CR-0073-01 - Split DEMO and UAT into modes at the artifact level

The runbook currently mixes presenter-polishing guidance ("memorable synthetic
value") with UAT-observation guidance ("don't tell the nurse it can be random").
Without an explicit mode switch tagged at session creation
(`mode: DEMO | UAT`), drift between intent and execution is likely. Recommend
the isolated platform stamp every session record with an immutable `mode` value
and expose distinct presenter-console controls so operators cannot accidentally
apply the wrong instruction to the wrong session.

### CR-0073-02 - Spec reconciliation should gate UAT spin-up, not run alongside it

`Source Reconciliation Required` already lists five docs that contradict the
selfie-required and verifier-visible direction
(`docs/flows/IDME_FIRST_ONBOARDING.md`, `docs/features/PASS_MANAGEMENT.md`,
`docs/design_system/WALLET_PASS_DISPLAY_SPEC.md`,
`docs/flows/VERIFIER_CREDENTIAL_VIEW.md`, `docs/architecture/SECURITY_MODEL.md`).
The UAT protocol and runbook already assume the new behavior. If UAT runs
against an unreconciled spec, findings cannot be cleanly attributed to "product
confusing" vs. "spec changed under us." Recommend either (a) close the
reconciliation task before cohort 1 starts, or (b) explicitly scope cohort 1 as
exploratory-only with non-binding exit criteria.

### CR-0073-03 - Selfie-in-verifier-view backend contract must be scoped before cohort 1

The protocol and runbook depend on a short-TTL, backend-authorized selfie
delivery that does not expose private storage paths (per `SECURITY_MODEL.md`).
TASK-0073 acknowledges this must be designed, but the implementation task has
not been created. If the delivery contract lands mid-cohort, sessions test
different products. Recommend the implementation task define this contract
before cohort 1.

### CR-0073-04 - Apple pass serial-number uniqueness across many sessions

Souvenir-pass-per-session implies a high serial count over time on the demo
Apple cert. Confirm the demo Apple pass type ID and signing environment are
dimensioned for this volume and that rotation is not required mid-cohort.

### CR-0073-05 - Ownership during a UAT session

The protocol does not state who runs the presenter console, triggers session
reset, redacts the recording, or reviews the evidence package when David is the
moderator. If David is moderating, console operation cannot reliably also be
David in real time. Recommend a one-line ownership matrix per session role:
moderator, console operator, evidence reviewer, recording redactor.

### CR-0073-06 - Retention policy for participant PII and recordings

Protocol section 15 says "preserve non-sensitive evidence under session ID" but
does not bound PII retention. Recommend explicit caps now: raw recordings
retained N days then deleted or redacted; redacted clips retained indefinitely;
participant phone cleared at session close (already stated); selfie retained
until replaced or until manual deletion request, whichever comes first.
