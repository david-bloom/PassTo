## Design Question Answers

### DQ-1: Does `handle_new_user()` fully initialize a usable profiles row?

**Yes, with one gap.**

The v4 trigger:
```sql
create or replace function public.handle_new_user()
returns trigger language plpgsql security definer set search_path = public as $$
begin
  insert into public.profiles (auth_user_id, email)
  values (new.id, new.email)
  on conflict (auth_user_id) do nothing;
  return new;
end;
$$;
create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();
```

After `supabase.auth.signUp()` the trigger immediately creates a `profiles` row with:
- `auth_user_id` = Supabase Auth user UUID
- `email` = from auth.users
- All column defaults: `onboarding_step = 'identity'`, `account_status = 'active'`, `subscription_tier = 'free'`, `id_verification_status = 'unverified'`

**No additional `profile-init` Edge Function is required for Phase 2** to create the row.

**The gap:** `first_name`, `last_name`, and other nurse-editable fields are `NULL` after trigger creation. These are populated by calling `update_own_profile_basic()` — a nurse-callable SECURITY DEFINER RPC.

**Email confirmation setting:** Supabase Auth email confirmation is **ON** for `wvzjfxacykgsaffskgtr`. This means `authData.session` will be `null` immediately after `signUp()` — the nurse is not yet authenticated until they confirm their email. `update_own_profile_basic()` cannot be called at signUp time. It must be called after email confirmation and first login.

### DQ-3: What is the authoritative frontend identifier after signup?

**`supabase.auth.getUser().data.user.id`** — this is the Supabase Auth UUID.

This equals `profiles.auth_user_id` in the database. It does **not** equal `profiles.id`.

The v4 schema has two distinct UUIDs:
- `profiles.auth_user_id` — FK to `auth.users(id)`, has UNIQUE constraint, equals `auth.uid()` in RLS
- `profiles.id` — the profiles table PK, its own `gen_random_uuid()` UUID, different from `auth_user_id`

**Critical correction from P2 enrollment pipeline spec:**
`P2_ENROLLMENT_PIPELINE_MIGRATION_SPEC.md` Note 2 states `profiles.id UUID REFERENCES auth.users(id)`, implying `profiles.id = auth.uid()`. This is **wrong in the v4 schema**. The RLS policy confirms:
```sql
using (auth_user_id = (select auth.uid()))
```
`auth.uid()` = `auth_user_id`, not `profiles.id`.

**Lovable code should use:**
- `supabase.auth.getUser().data.user.id` for nurse identification, Storage paths, and Edge Function JWT payloads
- `profile.auth_user_id` or `profile.id` when needed in DB queries, with care about which one is being referenced
- Storage path `${user.id}/selfie.jpg` where `user.id` = `auth.uid()` = `profiles.auth_user_id` — already correct in TASK-0020

### DQ-4: How should Lovable fetch the current user's profile after login/signup?

```js
// After signUp or signIn — fetch the current nurse's profile
const { data: profile, error } = await supabase
  .from('profiles')
  .select('*')
  .maybeSingle()
```

The `nurse_select_own_profile` RLS policy (`auth_user_id = (select auth.uid())`) ensures this returns only the current user's own row. No explicit `.eq()` filter needed.

Use `.maybeSingle()` (not `.single()`) so a missing profile row returns `null` rather than an error — enabling graceful null-profile handling (see DQ-6).

`profile === null` (no error, null data) and an `error` object are different states and must be handled separately:
- `profile === null, error === null` → row does not exist — null-profile recovery path (see DQ-6)
- `error` present → network, auth, or Supabase failure — show retry UI; do not route to enrollment steps

**When to fetch:**
- On app mount / route guard after authentication confirmed
- After signUp (gives router the initial `onboarding_step = 'identity'`)
- After any step that may have changed `onboarding_step` (to re-route if needed)

### DQ-6: What should happen if authenticated but no profile row exists?

The `handle_new_user()` trigger fires on every `auth.users` INSERT, so this should not occur for new signups after the v4 migration is live. However it **can** occur for:
- Auth users created before the v4 trigger was applied (legacy accounts)
- Edge cases where the trigger failed silently (very unlikely but possible)

**Required handling:**
Lovable should detect `profile === null` after a successful auth session and:
1. Show a non-blocking "Setting up your account..." state
2. Call a recovery RPC or Edge Function that re-runs the profile initialization (inserts the missing row)
3. On success, re-fetch the profile and proceed to the router

**Phase 2 approved behavior: show retry/support only.**

```js
if (!profile) {
  // Show recovery UI — do not proceed to onboarding router
  // Log error for ops visibility
  // Show a "contact support" message with a retry/refresh button
  // Do NOT attempt auto-recovery or call a backend function
}
```

A dedicated `profile-recovery` Edge Function is **not in TASK-0022 scope**. The risk is low for new signups — the `handle_new_user()` trigger fires synchronously and the null case is primarily a legacy pre-v4 account concern. The MVP-safe approach is to surface the state to support rather than auto-recover. A recovery Edge Function can be added in a later phase if ops data shows it is needed.

### DQ-7: What if multiple profile rows match the same auth user?

Impossible by schema constraint. The v4 profiles table has:
```sql
auth_user_id uuid not null unique references auth.users(id)
```
The UNIQUE constraint prevents duplicate profiles for the same auth user at the database level. The `handle_new_user()` trigger also uses `ON CONFLICT (auth_user_id) DO NOTHING`. No error path needed beyond the constraint violation itself.

## 4. Phase 2.2 — Profile Initialization Design

### 4.1 Trigger-based profile creation (no Edge Function needed)

When a nurse signs up via `supabase.auth.signUp()`, the `on_auth_user_created` trigger fires synchronously and inserts the profiles row. The row exists before `signUp()` returns — but **email confirmation is ON**, so `authData.session` will be `null`. The nurse is not authenticated until they confirm their email.

**`CreateAccount.tsx` is a full replacement**, not a patch. Real file inspection (2026-05-27) showed the existing file inserts into an `enrollments` table that does not exist in v4 and stores `sessionStorage.enrollmentId`. Both must be removed. The v4 form collects: first name, last name, email, password. No license fields — license lookup is Phase 3.3 backend work.

**`update_own_profile_basic()` RPC params (verified from live schema, 2026-05-27):**
`p_first_name`, `p_last_name`, `p_phone`, `p_display_name`, `p_preferred_name`, `p_notification_preferences` — all `DEFAULT NULL`. Passing only `p_first_name` and `p_last_name` is valid.

**Signup handler:**

```js
const { data: authData, error: signUpError } = await supabase.auth.signUp({
  email: email.trim(),
  password,
})

if (signUpError || !authData?.user) {
  // show error, do not proceed
  return
}

// handle_new_user() trigger has created the profiles row.
// authData.session is null — email confirmation is ON. This is the expected path.

if (authData.session) {
  // Unexpected: email confirmation may be off. Call RPC immediately.
  await supabase.rpc('update_own_profile_basic', {
    p_first_name: firstName.trim(),
    p_last_name: lastName.trim(),
  })
  navigate('/verify-identity')
} else {
  // Expected default: persist name for post-login RPC call.
  sessionStorage.setItem('pendingName', JSON.stringify({
    firstName: firstName.trim(),
    lastName: lastName.trim(),
  }))
  navigate('/confirm-email')
}
```

**Post-login name completion** (run in the post-login flow before routing):

```js
const pending = sessionStorage.getItem('pendingName')
if (pending) {
  const { firstName, lastName } = JSON.parse(pending)
  await supabase.rpc('update_own_profile_basic', {
    p_first_name: firstName,
    p_last_name: lastName,
  })
  sessionStorage.removeItem('pendingName')
}
// then call routeByOnboardingStep()
```

### 4.4 `profiles.id` vs `profiles.auth_user_id` — usage guide for Lovable

| Use case | Correct value | Why |
|---|---|---|
| Identify current nurse after login | `supabase.auth.getUser().data.user.id` | This is `auth.uid()` = `profiles.auth_user_id` |
| Storage upload path | `${user.id}/selfie.jpg` | `user.id` = `auth.uid()` = `auth_user_id` |

## 5. Phase 2.3 — Onboarding Step Routing Baseline

### 5.1 Route map

| `profiles.onboarding_step` | Route | Condition to advance |
|---|---|---|
| `identity` | `/verify-identity` | ID.me verification completes (Phase 3.1) |
| `phone` | `/verify-phone` | Twilio phone verification completes (Phase 3.2) |
| `license` | `/license-lookup` | License + data match + selfie complete (Phase 3.3-3.5) |
| `pass` | `/pass-ready` | Wallet issuance complete (Phase 4) |
| `complete` | `/dashboard` | — |

**Phase 2 implements the routing only** — routes display "coming soon" / placeholder screens for all steps except the existing UI. Trust gates, backend verification, and `onboarding_step` advancement are all Phase 3+.

### 5.2 Router logic (pseudocode for Lovable)

```js
// Called after login and on app load
async function routeByOnboardingStep() {
  const { data: profile } = await supabase
    .from('profiles')
    .select('onboarding_step, account_status')
    .maybeSingle()

  if (!profile) {
    // No profile — show recovery state (see Section 6)
    return navigate('/account-recovery')
  }

  if (profile.account_status === 'suspended') {
    return navigate('/account-suspended')
  }

  if (profile.account_status === 'closed') {
    return navigate('/account-closed')
  }

  switch (profile.onboarding_step) {
    case 'identity':  return navigate('/verify-identity')
    case 'phone':     return navigate('/verify-phone')
    case 'license':   return navigate('/license-lookup')
    case 'pass':      return navigate('/pass-ready')
    case 'complete':  return navigate('/dashboard')
    default:          return navigate('/verify-identity')  // safe fallback
  }
}
```

### 5.3 Route guard

Authenticated routes should run `routeByOnboardingStep()` on mount. Unauthenticated users attempting to reach any `/verify-*`, `/dashboard`, or `/pass-ready` route are redirected to `/` (login/signup) by the auth guard.

### 5.4 Step advancement

`onboarding_step` is a backend-controlled field. Lovable **must not** write it directly. The nurse's `onboarding_step` advances only when a backend Edge Function confirms the relevant trust gate has passed. Phase 2 does not implement any advancement logic — the field remains `'identity'` for all nurses until Phase 3 backend work begins.

## 6. Error and Recovery States

| State | Detection | Lovable behavior |
|---|---|---|
| Auth session valid but no profile row | `profile === null` after `.maybeSingle()` | Navigate to `/account-recovery`; show "Setting up your account" message; offer support contact |
| signUp succeeds but `update_own_profile_basic` fails | RPC returns error | Log silently; do not block routing; name/phone will be null but profile row exists |
| `account_status = 'suspended'` | `profile.account_status === 'suspended'` | Navigate to `/account-suspended`; show explanation; offer support contact |
| `account_status = 'closed'` | `profile.account_status === 'closed'` | Navigate to `/account-closed`; account closure confirmation screen |
| Network error on profile fetch | Supabase client returns error | Show retry state; do not route to enrollment steps |
| Session expired mid-enrollment | `supabase.auth.getSession()` returns no session | Redirect to login; preserve intended destination for post-login redirect |
| signUp email already registered | `signUpError.message` indicates duplicate | Show "account already exists" prompt; offer login link |

### 7.2 What Lovable must not do in Phase 2

| Forbidden action | Reason |
|---|---|
| `supabase.from('profiles').update({ onboarding_step: ... })` | Trust-gate field — service-role only |
| `supabase.from('profiles').update({ account_status: ... })` | Admin field — service-role only |
| `supabase.from('profiles').update({ id_verification_status: ... })` | Trust-gate field — service-role only |
| `supabase.from('profiles').update({ selfie_storage_path: ... })` | Trust-gate field — service-role only |
| Direct INSERT on `profiles` | Handled by `handle_new_user()` trigger — no nurse INSERT policy |
| Direct DELETE on `profiles` | No nurse DELETE policy — soft delete only |
| Use `service_role` key in Lovable | Must never appear in Vite env or Lovable code |

### 7.3 RLS policies active in Phase 2

| Policy | Table | Effect |
|---|---|---|
| `nurse_select_own_profile` | `profiles` | SELECT where `auth_user_id = (select auth.uid())` |
| `update_own_profile_basic()` SECURITY DEFINER RPC | `profiles` (via RPC) | Only updates name/phone/display/preferred/notification_preferences |
| `handle_new_user()` trigger | `auth.users` → `profiles` | Creates profiles row on signUp — nurse never has direct INSERT |

No INSERT or UPDATE policy exists for the `authenticated` role directly on `profiles`. All writes go through the trigger or the RPC.
