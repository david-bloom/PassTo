## Known Constraints

- Supabase project: `wvzjfxacykgsaffskgtr`.
- Lovable may use only publishable/browser-safe keys and authenticated user sessions.
- Service-role key must never be exposed to Lovable.
- Durable profile/account state belongs in Supabase.
- Trust fields remain backend-controlled.
- `profiles.id` is a profile UUID. `profiles.auth_user_id` references `auth.users(id)`. Do not assume `profiles.id = auth.uid()` unless live schema is intentionally changed later.
- Existing P2 enrollment docs may contain stale `profiles.id = auth.uid()` assumptions; Claude must correct for the v4 schema.
- `TASK-0018` R4 `licenses.normalized_status` fix is not required for Phase 2, but remains required before Phase 3.3 license lookup work.

## Onboarding Routing Baseline

Use the v4 `profiles.onboarding_step` allowed values as the starting point:

```text
identity
phone
license
pass
complete
```

Expected baseline mapping to confirm or refine:

| `profiles.onboarding_step` | Intended route / state | Notes |
|---|---|---|
| `identity` | ID.me start or identity verification route | Default after profile creation. Phase 2 routes only; does not implement ID.me. |
| `phone` | Phone verification route | Phase 2 routes only; does not implement Twilio. |
| `license` | License lookup / data match precondition route | Phase 2 routes only; does not implement lookup. |
| `pass` | PassReady / wallet issuance state | Phase 2 routes only; does not issue credential/wallet. |
| `complete` | Dashboard route | Phase 2 routes only; dashboard implementation is later. |

If Claude recommends changing the routing names, document why and whether a schema migration would be required. Do not apply one.
