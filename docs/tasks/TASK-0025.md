## Product Goal

Issue Apple and Google wallet passes automatically after credential creation and surface success/failure through the PassReady experience.

## Technical Scope

- Wire credential creation completion to wallet issuance.
- Use approved Vercel wallet signing routes where Node/certificate/provider requirements make Vercel the right boundary.
- Persist wallet provider state in Supabase.
- Return safe wallet URLs or status to Lovable.
- Implement PassReady success, pending, and failure states.
- Write audit/event records for issuance success/failure.

## Out of Scope

- Wallet production certificate approval.
- Manual dashboard-triggered issuance as the primary MVP path.
- Share-link verifier flow.

## Acceptance Criteria

- [ ] Wallet issuance is triggered automatically after credential creation.
- [ ] Apple and Google wallet pass paths are wired or explicitly documented if one provider is blocked.
- [ ] `wallet_passes` state is persisted.
- [ ] PassReady reflects success, pending, and failure states.
- [ ] Failure does not erase the credential and provides ops recovery signal.
- [ ] Claude documents tests, files changed, deviations, and risks.

## QA Plan

- Test successful wallet issuance with provider sandbox/test mode where available.
- Test signing route failure and retry/recovery behavior.
- Verify wallet state rows and audit/event records.
