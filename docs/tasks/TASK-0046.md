## Product Goal

Create the backend workflow for `/license-info`: collect the nurse's primary license details, run license lookup, normalize source status, and bind the license result to the verified ID.me identity before plan selection or payment.

## Technical Scope

- Implement or update the backend endpoint/Edge Function used by `/license-info`.
- Accept primary license inputs from Lovable:
  - `license_number`
  - `license_state`
  - `license_type`
- Require a verified ID.me-first onboarding attempt/profile before lookup.
- Call the approved license lookup source server-side only.
- Store lookup source result and normalized status fields using canonical naming.
- Run deterministic ID.me/license data matching after lookup succeeds.
- Persist match result and failure state.
- Advance onboarding only when lookup and match pass.
- Block `/account-select` and payment if lookup or match fails.
- Write audit/failure events for lookup success, lookup failure, status failure, and identity/license mismatch.

- Twilio phone verification.
- Stripe payment.

## David / Codex Decision — DOB Matching — 2026-06-01

**Decision:** Approved for MVP: `dob_match_mode = name_only`

Rationale:

- FLOW-LICENSE-003 originally specified ID.me DOB vs. license DOB matching when DOB is available.
- The live v4 `licenses` table does not contain a DOB field.
- Adding DOB storage now would increase sensitive PII handling and require a separate schema/privacy decision.
- The safe MVP path is conservative name-only matching using the ID.me-verified legal name and the license holder name returned by the approved license lookup source.

Implementation constraints:

- Do not store ID.me DOB or license DOB for MVP unless David separately approves a schema/privacy change.
- The match source of truth must be ID.me-verified `profiles.first_name` and `profiles.last_name`, not browser-edited names.
- License holder names must come from the server-side license lookup result, not the browser.
- Persist match metadata with `dob_match_mode = 'name_only'`.
- A name-only match may pass only when the license lookup succeeds, normalized license status is issuable, ID.me is verified at IAL2, and conservative deterministic name matching passes.
- If first name or last name mismatches, block progression and do not route to phone/plan/payment.
- Audit/failure metadata must clearly show that DOB was not matched and the MVP mode was name-only.

#### P1 — `complete_license_verification` is a public SECURITY DEFINER bypass

**Function/schema:** `public.complete_license_verification`  
**Pattern:** `supabase/migrations/migration_g_license_match_fields.sql`, lines 23-32; live database confirms `anon_exec = true` and `authenticated_exec = true`.

What is wrong:

- The RPC is in exposed schema `public`, is `SECURITY DEFINER`, and has default PUBLIC execute privileges.
- Because it bypasses RLS, a client can call the RPC directly through PostgREST instead of going through `license-lookup`.
- The RPC accepts caller-provided `p_profile_id`, `p_license_id`, `p_match_result`, and `p_dob_match_mode`; setting `p_match_result = 'passed'` updates `licenses.data_match_passed = true` and advances `profiles.onboarding_step` to `phone` when the profile is at `license`.
- This bypasses the Edge Function trust gates, the RapidAPI lookup, status normalization, and ID.me/license name matching.

Required fix:

- Move the function to a private/non-exposed schema or explicitly revoke execution from `PUBLIC`, `anon`, and `authenticated`.
- Grant execute only to `service_role`.
- Add `SET search_path = public, pg_temp` or fully lock the search path for the security-definer function.
- Validate inputs inside the RPC: allowed `p_match_result` values, `p_dob_match_mode = 'name_only'`, profile is active/verified/IAL2 and still at `license`, license belongs to that profile, license is currently `Active`, and wallet pass treatment is issuable.
- Raise if the profile step update affects zero rows on a supposed pass.

#### P1 — Deployed provider path does not prove active/issuable license status

**Function:** `license-lookup`  
**Pattern:** `supabase/functions/license-lookup/index.ts`, lines 407-413 and 483-484.

What is wrong:

- The deployed RapidAPI contract comment says the endpoint returns only `{ license_number, full_name, license_type }` and has no status field.
- The adapter then synthesizes `rawStatus = "Registered"` when no status exists.
- `STATUS_FALLBACK` maps `"Registered"` to `normalized_status = 'Active'`, `status_intent = 'credential_valid'`, and `wallet_pass_treatment = 'valid'`.
- This means existence under a last-name search is treated as an active/issuable license, so expired, suspended, revoked, inactive, or otherwise non-issuable licenses can pass if the endpoint cannot report their status.

Required fix:

- Do not synthesize an issuable status from a provider that does not return status.
- Either switch to an approved source/path that returns current license status and expiration/status metadata, or treat missing status as `Unknown` / `do_not_issue`.
- Keep plan/payment/credential routing blocked until the provider can prove an issuable status.

#### P2 — Raw provider payload is stored without a privacy allowlist

**Function/table:** `license-lookup`, `public.licenses.lookup_response`  
**Pattern:** `supabase/functions/license-lookup/index.ts`, lines 282, 311, and 493; RLS allows authenticated nurses to select their own `licenses` rows.

What is wrong:

- TASK-0046 explicitly approved `dob_match_mode = name_only` and says not to store ID.me DOB or license DOB for MVP.
- The function stores `providerResult.rawPayload` verbatim in `licenses.lookup_response`.
- If this or a future provider returns DOB, address, disciplinary details, or other unnecessary PII, the system will persist it and expose it through the nurse's own `licenses` SELECT policy.

Required fix:

- Replace raw payload storage with an allowlisted, redacted payload containing only fields needed for audit/debug.
- Explicitly strip DOB/date-of-birth and unrelated PII fields before writing `lookup_response`.
- Document the retained fields in TASK-0046 or FLOW-LICENSE-002.

#### P2 — Existing passed match state is not cleared before ineligible/failure outcomes

**Function:** `license-lookup`  
**Pattern:** update existing license at lines 269-285; ineligible return at lines 329-340; RPC only runs after `credentialEligible = true`.

What is wrong:

- When an existing license row is updated, the function updates status fields but does not clear `data_match_passed` or `dob_match_mode`.
- If a license previously passed and a later lookup returns inactive/expired/revoked or otherwise non-eligible, the function returns before calling the RPC and can leave `data_match_passed = true` on the row.
- Any downstream gate that treats `data_match_passed` as the durable match flag could read stale truth.

Required fix:

- On every lookup refresh, set `data_match_passed = false` and `dob_match_mode = 'name_only'` or `null` until both current status and current match pass.
- Credential issuance must require both `data_match_passed = true` and current issuable status fields (`normalized_status = 'Active'`, wallet treatment issuable, recent `status_checked_at`).

#### P2 — Lookup/audit writes are best-effort and incomplete on failure paths

**Function:** `license-lookup`  
**Pattern:** `writeLookupRecord` and `writeAudit`, lines 586-617; missing-required-field and license-number-mismatch branches at lines 203-230.

What is wrong:

- `writeLookupRecord` and `writeAudit` only log insert errors and allow the flow to continue.
- Some failure branches write a lookup row but no audit event.
- TASK-0046 requires audit/failure events for lookup failure, status failure, and identity/license mismatch; these records are part of the compliance trail for a credential gate.

Required fix:

- Make required lookup/audit writes fail closed for terminal outcomes, or move the terminal persistence into an RPC/transaction that records the failure and returns the user-safe response.
- Add audit events for missing required provider fields and license-number mismatch.

#### P3 — Implementation docs do not match deployed provider configuration

**Docs/function:** TASK-0046 implementation notes and `license-lookup`  
**Pattern:** TASK-0046 says `RAPIDAPI_HOST` and `RAPIDAPI_LICENSE_URL` are read from secrets; `license-lookup` hardcodes `RAPIDAPI_URL` and `RAPIDAPI_HOST`.

What is wrong:

- The implementation notes and function header describe environment-driven provider configuration, but deployed code hardcodes the provider URL and host.

Required fix:

- Either make URL/host secrets authoritative or update the docs to state that the provider path is hardcoded for this MVP deployment.

### Edge Function: `license-lookup` (`verify_jwt: true`)

**Trust gates checked before any lookup:**
- `profiles.account_status = 'active'`
- `profiles.id_verification_status = 'verified'`
- `profiles.id_verification_level = 'IAL2'`
- `profiles.onboarding_step = 'license'`
- `profiles.first_name` and `profiles.last_name` present (ID.me-verified identity anchor)

**RapidAPI adapter:**
- Reads `RAPIDAPI_KEY`, `RAPIDAPI_HOST`, `RAPIDAPI_LICENSE_URL` from Supabase secrets
- 10-second timeout; classifies 429 and 401/403 separately from general errors
- Flexible field extraction (multiple field name alternatives); pin exact names after first sandbox run
- License number normalized (strip non-alphanumeric, uppercase) before comparison — hard safety gate

**Status normalization:**
- Queries `license_status_mappings` table first; falls back to hardcoded MVP map
- Unknown raw status → `normalized_status = 'Unknown'`, `wallet_pass_treatment = 'do_not_issue'`, lookup fails
- Non-Active normalized status → lookup record written, progression blocked, no match attempted

**Name matching (`dob_match_mode = name_only`):**
- Normalizes both names: uppercase, strip punctuation, replace hyphens with spaces
- First-name first-token must match exactly
- Last-name last-token must match exactly
- Middle names/initials on either side are allowed to differ
- No fuzzy matching libraries — deterministic token comparison only

**On match pass:** calls `complete_license_verification()` RPC atomically

**Response codes:**
- `{ success: true, next_step: "phone" }` — match passed, step advanced
- `{ success: false, error: "identity_license_mismatch" }` — name mismatch
- `{ success: false, error: "not_found" }` — lookup failed / number mismatch
- `{ success: false, error: "license_status_unrecognized" }` — unknown raw status
- `{ success: false, credential_eligible: false }` — found but inactive/expired
