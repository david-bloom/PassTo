## Product Goal

Create the backend workflow for `/license-info`: collect the nurse's primary license details, run license lookup, normalize source status, and bind the license result to the verified ID.me identity before plan selection or payment.

## Technical Scope

- Implement or update the backend endpoint/Edge Function used by `/license-info`.
- Accept primary license inputs from Lovable:
  - `license_number`
  - `license_state`
  - `license_type`
- Require a verified ID.me-first onboarding attempt/profile before lookup.
- Call the approved license lookup source server-side only.
- Store lookup source result and normalized status fields using canonical naming.
- Run deterministic ID.me/license data matching after lookup succeeds.
- Persist match result and failure state.
- Advance onboarding only when lookup and match pass.
- Block `/account-select` and payment if lookup or match fails.
- Write audit/failure events for lookup success, lookup failure, status failure, and identity/license mismatch.

## Out of Scope

- ID.me-first callback/account-link implementation.
- Twilio phone verification.
- Stripe payment.
- Selfie upload.
- Credential creation and wallet issuance.
- Additional/second license flow.

### Summary

The product direction is correct: `/license-info` should perform backend-owned license lookup and bind the returned license holder to the verified ID.me identity before plan selection or payment. Do not execute this task yet. TASK-0045 currently has unresolved P1 trust-boundary findings, and TASK-0046 depends directly on the final 0045 account/session/attempt model.

### Required Spec Changes Before Approval

#### P1 — Do not build license lookup on an unresolved TASK-0045 trust model

TASK-0046 says the backend should "require a verified ID.me-first onboarding attempt/profile before lookup." That is directionally right, but too vague while TASK-0045 is blocked.

Required revision:

- If TASK-0045 creates a Supabase Auth session before `/license-info`, the license lookup function must require `verify_jwt = true`, validate the authenticated `profiles` row, and require:
  - `id_verification_status = 'verified'`
  - `id_verification_level = 'IAL2'`
  - `id_me_subject` present
  - `onboarding_step = 'license'`
- Do not make `/license-info` a `verify_jwt = false` function unless Codex separately approves a new attempt-bound token model.
- Do not trust `attempt_id`, profile ID, name, DOB, email, or ID.me subject from the browser.

#### P1 — Reconcile FLOW-LICENSE docs with the live v4 schema

The task references FLOW-LICENSE-002 and FLOW-LICENSE-003, but those docs still contain older naming and fields that do not fully match the live v4 schema.

Observed live schema / docs mismatch:

- FLOW-LICENSE-002 uses `license_lookup_attempts`; live v4 uses `license_lookups`.
- FLOW-LICENSE-002 uses `license_status_map`; live v4 uses `license_status_mappings`.
- FLOW-LICENSE-003 expects holder name/DOB fields such as `holder_name` and `holder_dob`; live `licenses` has `first_name`, `last_name`, and no verified DOB field.
- TASK-0046 lists `license_lookups if present/approved`; it is present in v4 and should be named explicitly.

Required revision:

- Add a "Schema Mapping" section before implementation.
- Use live v4 table names unless Claude proposes a migration and pauses for David approval.
- If DOB matching remains required, specify where ID.me DOB and license DOB are stored, how they are encrypted/minimized, and how they are cleared or retained for audit.
- If the provider does not return DOB or the schema will not store DOB, explicitly approve a conservative name-only MVP path and record `dob_match_mode = unavailable_name_only`.

#### P2 — Provider/path decision is still an execution gate

TASK-0046 depends on an "approved license lookup provider/path", but the spec does not pin the provider contract.

Required revision:

- Identify the approved provider for first implementation, endpoint shape, required response fields, timeout, retry behavior, rate-limit behavior, and sandbox/test mode.
- Do not hardcode old RapidAPI coverage lists.
- Use mocked provider responses in unit tests; live provider calls belong only in manual/sandbox integration QA.

#### P2 — Make the state transition explicit

For the ID.me-first route order, successful lookup plus successful data match should advance the nurse from `onboarding_step = 'license'` to `onboarding_step = 'phone'`.

Required revision:

- State exactly which backend operation performs this transition.
- Ensure lookup success without data-match pass does not advance.
- Ensure lookup failure, inactive/invalid status, source `Unknown`, or identity/license mismatch leaves the nurse blocked from `/phone-check`, `/account-select`, payment, credential creation, and wallet issuance.
- Prefer an atomic RPC for "license lookup + match pass + audit + onboarding step advance" if multiple writes must succeed together.

#### P2 — Persist a durable match gate

TASK-0046 says "persist match result and failure state" but does not say where.

Required revision:

- Specify whether the durable current-state gate lives on `licenses`, `profiles`, a new identity/license match table, or another approved table.
- Credential creation must be able to check a durable `passed` result without reconstructing the match from audit logs alone.
- Audit rows are evidence, not the only state gate.
