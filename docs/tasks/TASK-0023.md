## Proposed Trust-Gate Order

### Gate 1 — ID.me Identity Verification

**Build Sequence:** Phase 3.1  
**Owner:** Supabase Edge Function preferred; Lovable starts/displays flow.  
**Purpose:** Confirm nurse identity at required assurance level before license ownership can be trusted.

Dependencies:

- Phase 2 complete.
- Authenticated Supabase user exists.
- `profiles` row exists and is owned by `auth_user_id = auth.uid()`.
- ID.me credentials/config available for the target environment.

Expected writes:

- `profiles.id_verification_status`
- `profiles.id_verification_level`
- `profiles.id_me_subject`
- `profiles.onboarding_step` advance to `phone` on success
- `audit_events` identity result

Failure states:

- IAL1 returned: block issuance, route retry/support.
- Failed verification: block issuance, route retry/support.
- Missing/invalid callback code: safe error, no state advancement.
- Session expired on callback: re-auth required before exchange.
- Duplicate callback/code reuse: handle idempotently where possible.

### Gate 2 — Twilio Phone Verification

**Build Sequence:** Phase 3.2  
**Owner:** Supabase Edge Functions preferred; Lovable collects/enters code.  
**Purpose:** Confirm phone possession before license lookup/data match.

Dependencies:

- Phase 2 complete.
- ID.me gate passed.
- `profiles.onboarding_step = 'phone'` or equivalent route state.
- Twilio credentials/config available.
- Launch fallback decision if Twilio A2P 10DLC is not ready.

Expected writes:

- `profiles.phone`
- phone verification state field if present or approved in implementation spec
- `profiles.onboarding_step` advance to `license` on success
- `audit_events` phone code sent / verified / failed
- `notification_events` if SMS event logging is used for provider attempts

Failure states:

- Code send failure: allow retry, log event, alert ops if provider issue.
- Code mismatch/expired: allow retry within safe limits.
- Twilio unavailable: block production launch unless David approves fallback.

### Gate 3 — License Lookup

**Build Sequence:** Phase 3.3  
**Owner:** Supabase Edge Function preferred.  
**Purpose:** Retrieve authoritative license data and normalize license status.

Dependencies:

- ID.me gate passed.
- Phone gate passed.
- R4 complete. This is already done: `licenses.normalized_status` supports `Pending`.
- License source credentials/config available.
- Nurse-provided license lookup inputs captured by Lovable.

Expected writes:

- `licenses`
- `license_lookups`
- normalized status fields based on `license_status_mappings`
- `audit_events` lookup started / succeeded / failed
- `profiles.onboarding_step` remains or advances according to data-match handoff design

Failure states:

- No result: block issuance, route support/remediation.
- Provider unavailable: safe retry, log event, ops alert for outage.
- Ambiguous/multiple results: block automated progression, route support.
- Unknown/unmapped status: write safe failure/caution state, do not issue.

### Gate 4 — Data Match

**Build Sequence:** Phase 3.4  
**Owner:** Supabase Edge Function preferred.  
**Purpose:** Confirm verified identity belongs to the license record.

Dependencies:

- License lookup succeeded.
- Identity data required for matching available in backend context only.
- License record exists.
- Matching rules approved.

Expected writes:

- data-match result field(s), if present or approved in implementation spec
- account/credential blocking state on failure, if present or approved
- temporary sensitive matching fields cleared when no longer needed
- `profiles.onboarding_step` remains/advances to selfie-ready state according to schema constraints
- `audit_events` `data_match.passed` or `data_match.failed`

Failure states:

- Name mismatch: block issuance, flag support/remediation.
- Missing identity/license fields: block issuance, support path.
- Matching function error: safe retry or ops alert depending cause.

### Gate 5 — Selfie Confirmation

**Build Sequence:** Phase 3.5  
**Owner:** Lovable direct upload to Supabase Storage plus backend confirmation Edge Function.  
**Purpose:** Confirm selfie asset exists after data match and set trust-gate fields.

Dependencies:

- Data match passed.
- Private `selfies` bucket exists.
- Exact-path Storage policies exist.
- `profiles.selfie_storage_path` and `profiles.selfie_captured_at` exist.

Expected writes:

- `profiles.selfie_storage_path`
- `profiles.selfie_captured_at`
- `profiles.onboarding_step` advance to `pass` only after backend confirms file exists and belongs to user
- `audit_events` `selfie.captured` or failure event

Failure states:

- Upload failed: retry UI, no profile state advancement.
- Upload succeeded but backend confirmation failed: retry confirmation, no state advancement.
- Path mismatch: security event, block progression.
- Unsupported MIME/oversize: retry with clear message.

## Cross-Gate Rules

- No gate may trust client-side state alone.
- Each gate must verify the authenticated user and profile ownership.
- Each gate must be idempotent or have a defined duplicate-call behavior.
- Each gate must fail safely: do not issue or present a credential as valid when a required trust signal is missing.
- Each gate must write enough event/audit state for support and ops.
- Lovable may display route state and collect inputs but cannot decide whether a gate passed.
- Service-role keys stay only in trusted backend contexts.
