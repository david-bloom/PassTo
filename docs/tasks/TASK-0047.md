### Summary

The intended route order is right for the ID.me-first flow, but TASK-0047 needs a sharper backend transition matrix before Claude executes it. The biggest issue is that the already-approved phone verification backend was originally built for the older order and advances phone success to `onboarding_step = 'license'`. In the new flow, phone success must advance to plan selection.

### Required Spec Changes Before Approval

#### P1 â€” Update the phone-success transition for ID.me-first

Current live phone verification behavior is still compatible with the older order:

```text
identity -> phone -> license
```

ID.me-first requires:

```text
identity/confirm -> license -> phone -> plan
```

Required revision:

- TASK-0047 must explicitly update the `complete_phone_verification()` RPC or replacement backend operation so successful Twilio verification advances to `onboarding_step = 'plan'`, not `license`.
- Update `phone-verify-otp` idempotency logic so the already-verified state is `plan` or later, not only `license`.
- Update `phone-send-otp` / `phone-verify-otp` route names and UX contract if the frontend route is now `/phone-check` rather than `/verify-phone`.
- Confirm no old router still sends `onboarding_step = 'license'` users back to the prior phone/license order.

#### P1 â€” Do not implement routing until TASK-0045 and TASK-0046 define durable gates

Route guards depend on trusted state from TASK-0045 and TASK-0046.

Required revision:

- `/phone-check` must require authenticated profile + verified ID.me + durable license lookup/data-match pass.
- `/account-select` must require Twilio phone verification and the durable license match gate.
- `/payment` must require a paid plan selection recorded server-side.
- `/upload-selfie` must require either Free plan selection or server-confirmed Stripe success for paid plans.
- `/success` must not be treated as proof of credential issuance; it should read server-derived credential/wallet status.

#### P2 â€” Add an explicit route/state matrix

The spec currently lists routes and acceptance criteria, but not the authoritative state machine.

Required revision:

Add a table such as:

| `profiles.onboarding_step` | Allowed route | Required backend facts |
|---|---|---|
| `license` | `/license-info` | ID.me verified/account linked |
| `phone` | `/phone-check` | License lookup + data match passed |
| `plan` | `/account-select` | Phone verified |
| `selfie` | `/upload-selfie` | Free selected or paid entitlement confirmed |
| `pass` / `complete` | `/success` | Credential/pass status read server-side |

Claude should adapt exact values to the final approved schema, but the implementation must have one source of truth and must not rely on UI-only route hiding.

#### P2 â€” Separate Stripe integration from route gating

TASK-0047 depends on Stripe readiness, but TASK-0040 is the actual Stripe subscription/payment state task.

Required revision:

- If TASK-0040 is not complete, TASK-0047 may define route guards and server-state expectations but must not invent a parallel Stripe implementation.
- Paid plans must not activate entitlements from client navigation or Stripe return URLs.
- Only webhook/server-confirmed payment/subscription state may advance paid-plan users past payment.

#### P2 â€” Clarify `/success` versus credential/wallet issuance

TASK-0047 says `/success` can show wallet readiness, but credential creation/wallet issuance are mostly out of scope.

Required revision:

- `/success` should be a status surface, not an issuance decision.
- It must support pending/delayed wallet issuance, provider failure, and support states.
- Add-another-license should remain hidden/disabled until the selected plan and entitlement state support it.
- If `/pass-ready` is retained, define whether it redirects to `/success`, aliases it, or is deprecated after a compatibility window.
