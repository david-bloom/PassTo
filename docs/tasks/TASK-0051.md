## Product Goal

Persist wallet pass issuance state in Supabase so nurses, ops, and verifier-facing systems can rely on durable backend state rather than transient provider responses.

## Technical Scope

- Implement or revise backend writes to `wallet_passes`.
- Store provider, status, pass URL or provider reference, issued/updated timestamps, and error state where applicable.
- Link wallet pass records to the credential record.
- Support Apple and Google as separate provider states or document the approved single-row/provider-variant model.
- Record audit/event state for issuance success, pending, failure, and retry.
- Ensure status readers can distinguish:
  - no wallet attempt yet
  - pending
  - issued
  - failed
  - blocked/do_not_issue

## Claude Implementation Notes

**Date:** 2026-06-02

### What Was Already Done (TASK-0050)

`wallet-issue` already writes to `wallet_passes` with:
- One row per provider (`apple_wallet`, `google_wallet`)
- `unique(credential_id, provider)` enforced by schema
- Status: `pending` â†’ `issued` or `error` after provider call
- Provider response stored in `provider_response` jsonb
- Audit events: `wallet.apple_issued`, `wallet.apple_failed`, `wallet.google_issued`, `wallet.google_failed`

### New Work â€” `success-status` Update

`success-status` previously read only the most recent single `wallet_passes` row. Updated to:
- Read all `wallet_passes` rows for the credential in one query
- Return separate `wallet.apple` and `wallet.google` status objects
- Distinguish: `not_attempted` (no row), `pending`, `issued`, `error`, `voided`
- Return safe URLs only when `status = 'issued'`
- Keep legacy `wallet_status` / `wallet_pass_url` / `wallet_provider` fields for backward compatibility
- Add `wallet.any_issued` boolean for Lovable display decisions

### Status Values Returned

| State | `wallet.apple.status` / `wallet.google.status` |
|---|---|
| No wallet attempt yet | `not_attempted` |
| Pending provider call | `pending` |
| Successfully issued | `issued` |
| Provider failed | `error` |
| Pass voided | `voided` |

### Ops Alerting â€” Deferred

`notification_events` table is for outbound SMS/email tracking (Twilio/Postmark). A wallet failure ops alert would require an email/SMS send mechanism not yet set up. Wallet failures are observable via `audit_events` with `action = 'wallet.apple_failed'` or `wallet.google_failed`, and via `wallet_passes.status = 'error'`. Ops monitoring query deferred to Phase 5.
