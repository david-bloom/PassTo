## Product Goal

Allow an eligible nurse to create a controlled one-time share link for verifier access without exposing raw token storage or allowing client-side entitlement bypass.

## Cleanup History

**2026-06-02:** A prior Claude session deployed an unapproved `share-link-create` function to Supabase (v2) without David approval and without pushing source to GitHub. The function had two P1 bugs:

- Profile lookup used `eq('id', user.id)` instead of `eq('auth_user_id', user.id)` — every valid nurse request returned `403 profile_not_found`
- Audit action used `'create'` instead of `'verification_token.created'` — audit insert failed, triggering fail-closed `500 audit_failed`

David approved deletion (APPROVAL-0021, 2026-06-02). Function deleted from Supabase. Local source also deleted. No share links were successfully created.

When TASK-0056 is formally approved, Claude will rewrite the function from scratch with all bugs fixed and push source to GitHub before any deployment.

## Dashboard Follow-Up Dependency (from TASK-0055 Codex QA)

`dashboard-status` currently returns `share_link_eligible: false, share_link_reason: "not_implemented"` hardcoded. When TASK-0056 is implemented and accepted, `dashboard-status` must be updated to:

This `dashboard-status` update is in scope for TASK-0056 execution.

## Technical Scope

- Implement or revise a backend share-link creation operation.
- Confirm authenticated nurse owns the credential.
- Require credential state and license status to be shareable.
- Enforce server-confirmed entitlement/payment rules.
- Generate a high-entropy raw token.
- Store only a token hash in `verification_tokens`.
- Use token type `share_link`.
- Apply approved TTL: 72 hours or first successful use, whichever comes first.
- Return the raw share URL/token once to Lovable.
- Write audit/event records for creation and rejection paths.

## Out of Scope

- Show QR token creation.
- PDF QR or PDF export.
- Verifier token validation.
- Verifier UI.
- Stripe checkout implementation beyond reading confirmed entitlement/payment state.
- Employer dashboard or verifier accounts.

## Acceptance Criteria

- [ ] Eligible nurse can create a `share_link` token for their own active/shareable credential.
- [ ] Unauthenticated, wrong-owner, unentitled, inactive, revoked, suspended, expired, blocked, or missing credential requests are rejected server-side.
- [ ] Raw token is returned once and never stored.
- [ ] Token hash, token type, expiry, status, credential/profile relationships, and metadata are persisted.
- [ ] Share-link TTL is 72 hours or first successful use, whichever comes first.
- [ ] Show QR and PDF token types are not created.
- [ ] Creation and rejection paths write audit/event records.
- [ ] Claude documents files changed, tests run, deviations, and risks.
