## Product Goal

Allow an eligible nurse to create a controlled one-time share link for verifier access without exposing raw token storage or allowing client-side entitlement bypass.

## Cleanup History

**2026-06-02:** A prior Claude session deployed an unapproved `share-link-create` function to Supabase (v2) without David approval and without pushing source to GitHub. The function had two P1 bugs:

- Profile lookup used `eq('id', user.id)` instead of `eq('auth_user_id', user.id)` — every valid nurse request returned `403 profile_not_found`
- Audit action used `'create'` instead of `'verification_token.created'` — audit insert failed, triggering fail-closed `500 audit_failed`

David approved deletion (APPROVAL-0021, 2026-06-02). Function deleted from Supabase. Local source also deleted. No share links were successfully created.

When TASK-0056 is formally approved, Claude will rewrite the function from scratch with all bugs fixed and push source to GitHub before any deployment.

`dashboard-status` currently returns `share_link_eligible: false, share_link_reason: "not_implemented"` hardcoded. When TASK-0056 is implemented and accepted, `dashboard-status` must be updated to:

This `dashboard-status` update is in scope for TASK-0056 execution.

## Technical Scope

- Implement or revise a backend share-link creation operation.
- Confirm authenticated nurse owns the credential.
- Require credential state and license status to be shareable.
- Enforce server-confirmed entitlement/payment rules.
- Generate a high-entropy raw token.
- Store only a token hash in `verification_tokens`.
- Use token type `share_link`.
- Apply approved TTL: 72 hours or first successful use, whichever comes first.
- Return the raw share URL/token once to Lovable.
- Write audit/event records for creation and rejection paths.

## Out of Scope

- PDF QR or PDF export.
- Verifier token validation.
- Verifier UI.
- Stripe checkout implementation beyond reading confirmed entitlement/payment state.
- Employer dashboard or verifier accounts.

## Claude Implementation Notes

**Date:** 2026-06-02

**Deliverable:** `share-link-create` Edge Function (new) + `dashboard-status` updated with real share-link eligibility (TASK-0056 dependency from TASK-0055 Codex QA).

**Files created/updated:**

| File | Change |
|---|---|
| `supabase/functions/share-link-create/index.ts` | New POST function — token creation, gate chain, audit |
| `supabase/functions/dashboard-status/index.ts` | Updated — `share_link_eligible` now derived from real credential/license/entitlement state |

**Fixes from prior unapproved source (documented in Cleanup History above):**

| Prior bug | Fix applied |
|---|---|
| `eq('id', user.id)` profile lookup | Corrected to `eq('auth_user_id', user.id)` |
| `action: 'create'` audit format | Corrected to `'verification_token.created'` (OD-1 resource.verb format) |
| No rejection-path audit events | All rejection gates now write `verification_token.creation_rejected` best-effort audit |

**Gate chain:**
1. Auth — `supabaseAuth.getUser()` + service-role for all writes
2. Profile — `account_status = 'active'`, IAL2, terminal `onboarding_step`
3. Credential — `status = 'active'`, owned by profile (most recent)
4. License — `normalized_status = 'Active'`, `data_match_passed = true`, `credential.license_id` not null, `profile_id` ownership confirmed
5. Entitlement — free tier always passes; paid tier requires active `subscriptions` row

**Token:**
- 32-byte `crypto.getRandomValues()` → hex raw token
- SHA-256 hash stored in `verification_tokens`; raw token never stored
- TTL: 72 hours; `token_type: share_link`
- `SHARE_LINK_BASE_URL` env var (default: `https://passtodigital.com/v`)

**`dashboard-status` eligibility logic:**
- `eligible: true` when credential active + license Active + data_match_passed + entitlement confirmed
- `eligible: false` with specific reason: `credential_not_active` | `license_not_active` | `license_match_not_passed` | `entitlement_not_confirmed`

**Tables written:**
- `verification_tokens` — `profile_id`, `credential_id`, `token_hash`, `token_type`, `status`, `expires_at`
- `audit_events` — creation path (fail-closed) + all rejection paths (best-effort)

**Tests run:** Both functions deployed to Supabase (project wvzjfxacykgsaffskgtr). Source in GitHub. Codex QA required before production use.

**Deviations from spec:**
- `payments` table not read — entitlement gated on `subscriptions.status = active` only, consistent with established pattern across all functions.
- `license_not_linked` rejection added for `credential.license_id = null` case — not in spec but required defensively given `license_id` is nullable in schema.

**Risks / open items:**
- `SHARE_LINK_BASE_URL` secret not yet set in Supabase — defaults to `https://passtodigital.com/v`. Must be set before production use.
- First-use enforcement (mark token `used` on first verifier access) deferred to TASK-0057.
