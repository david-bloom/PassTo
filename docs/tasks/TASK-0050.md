## Product Goal

Define the backend contract for issuing Apple Wallet and Google Wallet passes from an approved credential without exposing signing secrets or provider keys to Lovable.

## Technical Scope

- Decide and document the signing boundary for MVP:
  - Supabase Edge Function where practical.
  - Vercel route where Node-specific libraries, certificates, or provider SDKs make Vercel the safer path.
- Define request/response contract between credential creation and wallet signing.
- Define the safe credential payload used to generate wallet passes.
- Define Apple and Google provider handling, including blocked-provider fallback if one provider is not launch-ready.
- Define signing secret/certificate storage requirements.
- Define audit/event behavior for signing success, pending, and failure.
- Define retry/idempotency behavior.

## Out of Scope

- Production wallet certificate/key procurement.
- Live wallet provider configuration changes without David approval.
- Full wallet lifecycle refresh/revocation automation.
- Permanent QR in wallet pass.
- Show QR and PDF export.

### Signing Boundary Decision

**Apple Wallet → Vercel** (`api/sign-apple.js`): Node.js required for `passkit-generator` and PEM certificate handling. Existing route retained and hardened.  
**Google Wallet → Vercel** (`api/sign-google.js`): JWT RS256 signing. Existing route retained and hardened.  
**Orchestration → Supabase Edge Function** (`wallet-issue`): loads from DB, calls both Vercel routes, writes `wallet_passes`, activates credential.

### Key Changes from Prior Signing Code

- Both Vercel routes now reject direct client calls (internal auth token required)
- All pass data loaded from Supabase via service role — no raw nurse data from request body
- Serial number / object ID are deterministic from `credential_id` for idempotency
- Google pass no longer embeds a QR barcode (product decision: no permanent verification QR)
- `wallet_passes` rows written by `wallet-issue` after each provider call
- `credentials.status` advanced to `active` when at least one provider succeeds

### Required Secrets (not yet set — David must provide)

- `VERCEL_SIGN_GOOGLE_URL` — `https://your-app.vercel.app/api/sign-google`

**Vercel env vars** (set in Vercel dashboard):

### Deviations

- `wallet_passes.external_pass_id` is used to store the Google Wallet `save_url`. The column comment implies a PassKit-assigned ID, but for Google Wallet the save URL is the durable reference.
- `wallet-issue` returns 200 even when both providers fail — the credential remains `pending` and the nurse sees a "preparing" state. A 5xx would cause Lovable to show an error and the nurse might retry excessively.

### Codex QA v1 Findings

**P1 — Vercel signing routes not runnable as checked in:**
- `api/sign-apple.js` and `api/sign-google.js` import `@supabase/supabase-js` but `package.json` was missing the dependency
- `api/sign-apple.js` required `icon.png`, `icon@2x.png`, `logo@2x.png`; only `logo.png` existed (others had macOS " copy" suffix)

**P1 — `wallet-issue` wallet durable-state writes not fail-closed:**
- `wallet_passes` upsert/update errors unchecked — provider success could be returned while DB persistence failed
- Credential activation error unchecked — `credential_status: active` could be returned when DB not yet updated

### Claude Remediation (2026-06-02, commit a691000)

| Finding | Fix |
|---|---|
| P1 `@supabase/supabase-js` missing | Added `^2.39.0` to `package.json` |
| P1 asset files with " copy" suffix | Copied `icon.png`, `icon@2x.png`, `logo@2x.png` to correct names in `api/assets/` |
| P1 `wallet_passes` pending upsert unchecked | Returns 500 if pending-row init fails — cannot proceed without durable state |
| P1 `wallet_passes` update after provider success unchecked | Checks error; on failure marks row as `error` (not `issued`), logs for ops |
| P1 credential activation unchecked | Checks error; on failure returns `credential_status: pending, wallet_activation_partial: true` instead of overstating activation |
| P2 response reflected provider success not DB state | Response now uses `applePersisted`/`googlePersisted` flags — only returns `issued` after DB write confirmed |

**P2 hardening applied before verdict (commit f2b012a):**
- `wallet-issue` activation failure returns distinct `wallet_activation_partial: true` partial state
- `success-status` read errors now fail closed (503 on DB error)
- Legacy `wallet_pass_url`/`wallet_provider` status-gated on `status = 'issued'`
