#### Tests Run

No automated test suite. Codex QA required with Stripe test-mode event replay.

#### Risks

- `stripe-webhook` returns 200 even on processing errors to prevent Stripe retries from looping. Events with `processed = false` require manual replay.
- `customer.subscription.deleted` lapse downgrades `subscription_tier` to `free` immediately. If Stripe retries the delete event, the downgrade is idempotent.
- `subscription_tier` on profiles is plan intent; `subscriptions.status` is the confirmed entitlement source.
- Lovable must poll `payment-status` on `/upload-selfie` page load — the Stripe `success_url` redirect may arrive before the webhook fires and advances `onboarding_step` from `payment` to `selfie`. Lovable should show a loading/pending state until `payment-status` confirms the step has advanced.

#### Additional Migration

### Codex QA v1 Verdict — Blocked (2026-06-02)

Three P1 findings:

**P1 — `stripe-webhook` deployed with `verify_jwt: true`:** Stripe cannot call the endpoint. All webhook delivery rejected before signature verification.

**P1 — Entitlement counts wrong:** Code had `standard: 2, premier: 5`. DECISION-0010 canonical values are `standard: 1, premier: 2`. `success-status` uses `license_entitlement_count > 1` for `can_add_license`, so wrong values would incorrectly grant Standard nurses the additional-license entitlement.

**P1 — `handleSubscriptionUpsert` cannot reliably upsert:** `subscriptions.profile_id` is NOT NULL, but the handler didn't include it. Also `stripe-checkout-create` only set Checkout Session metadata, not `subscription_data.metadata`, so Subscription-object events had no `profile_id`.

### Claude Remediation (2026-06-02, commit a57b25a)

| Finding | Fix |
|---|---|
| verify_jwt | `stripe-webhook` redeployed with `--no-verify-jwt` |
