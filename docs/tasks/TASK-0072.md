## Product Goal

Move PassTo from wallet-signing scaffolding to actual Apple Wallet and Google Wallet issuance so a verified nurse can receive real add-to-wallet actions after the credential is created.

## Background

TASK-0050 defined and implemented the wallet signing boundary:

- `wallet-issue` Supabase Edge Function orchestrates issuance and writes `wallet_passes`.
- `api/sign-apple.js` signs Apple `.pkpass` files on Vercel.
- `api/sign-google.js` signs Google Wallet save JWTs on Vercel.
- `/success` calls `credential-create`, then `wallet-issue`, then reads `success-status`.

That work passed with a documented deferral: real Apple/Google signing was not testable because provider credentials, signing secrets, Vercel environment variables, and production route configuration were not yet completed.

This task closes that deferral.

## Security/RLS Impact

Security Impact: High

Wallet signing certificates, private keys, Google service account JSON, service-role keys, and internal signing secrets must remain server-side only. Lovable must not call Apple/Google provider routes directly and must not receive signing material.

## Design System Impact

Design System Impact: Medium
Design Source: `docs/design_system/README.md`

The wallet credential design must be built in the provider pass payloads, not only represented by screenshot concepts. Apple Wallet uses the PassKit field layout in `api/sign-apple.js`; Google Wallet mirrors that same canonical field set in `api/sign-google.js`. Only wallet action display states may need Lovable UI correction. No new design system primitives are expected.
