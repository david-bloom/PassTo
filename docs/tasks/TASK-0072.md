## Product Goal

Move PassTo from wallet-signing scaffolding to actual Apple Wallet and Google Wallet issuance so a verified nurse can receive real add-to-wallet actions after the credential is created.

## Background

TASK-0050 defined and implemented the wallet signing boundary:

- `wallet-issue` Supabase Edge Function orchestrates issuance and writes `wallet_passes`.
- `api/sign-apple.js` signs Apple `.pkpass` files on Vercel.
- `api/sign-google.js` signs Google Wallet save JWTs on Vercel.
- `/success` calls `credential-create`, then `wallet-issue`, then reads `success-status`.

That work passed with a documented deferral: real Apple/Google signing was not testable because provider credentials, signing secrets, Vercel environment variables, and production route configuration were not yet completed.

This task closes that deferral.

## Codex QA Review

### Codex QA Blocked - 2026-06-15

**Decision:** Fail / remediation required. TASK-0072 is not ready for David Done review.

#### P1 Findings

1. **`wallet-issue` does not pass type-checking and uses unsupported PostgREST `.catch()` calls.**
   - `deno check supabase/functions/wallet-issue/index.ts` reports 12 errors.
   - The audit inserts at lines 106-110, 167-171, 175-179, 188-192, 213-217, 221-225, 234-238, 260-264, and 274-278 call `.catch()` on a `PostgrestFilterBuilder`, which does not expose that method in the deployed client contract.
   - The first occurrence is before wallet row initialization, so an authenticated non-idempotent invocation can fail before creating `wallet_passes`.
   - Required remediation: replace these calls with explicit awaited insert results and non-fatal error handling, run `deno check`, deploy the corrected function under the existing approval, and exercise it with an authenticated disposable nurse.

2. **The documented test credential has no end-to-end wallet issuance evidence.**
   - Live credential `c855fe7f-db98-4e79-884b-227194922a92` is `active`, but `pass_template_data` is `null`.
   - Live SQL returned zero `wallet_passes` rows and zero wallet/credential audit rows for that credential.
   - Therefore, the direct Vercel signing calls did not verify the required `wallet-issue` orchestration, durable provider persistence, backend-controlled activation invariant, `/success` integration, or required display payload.
   - Required remediation: use a disposable credential created through the current `credential-create` path, confirm non-null canonical `pass_template_data`, invoke `wallet-issue` with that nurse's authenticated JWT, then read back both provider rows, credential status, audit events, and `/success` behavior.

3. **Google Wallet class ID is constructed incorrectly for the documented production configuration.**
   - The configured value is documented as the full class ID `3388000000023110660.passto_nurse_license_v1`.
   - `api/sign-google.js` sets `classId` to `${issuerId}.${classId}`, producing a duplicated issuer prefix.
   - Google requires class IDs in the single-prefix form `issuerID.identifier`: https://developers.google.com/wallet/reference/rest/v1/genericobject
   - The route only signs a JWT locally and returns `success: true`; it does not prove that Google accepted or rendered the referenced object.
   - Required remediation: define `GOOGLE_WALLET_CLASS_ID` consistently as either a full class ID used directly or a suffix prefixed exactly once, then verify the save link through Google Wallet on a real/preview device.

4. **`do_not_issue` is treated as a valid-looking wallet pass.**
   - Both signing routes default `deriveTreatment()` to `valid`.
   - `wallet_pass_treatment = do_not_issue` is neither rejected nor mapped to an invalid treatment, contrary to `WALLET_PASS_DISPLAY_SPEC.md`.
   - Required remediation: fail closed before signing when treatment is `do_not_issue` or required canonical identity/license fields are absent. Add focused tests for valid, caution, invalid, and do-not-issue states.
