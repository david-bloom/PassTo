## Account-Status Authorization Note — 2026-05-29

Lovable reported that current login handling checks `profiles.account_status` client-side after `signInWithPassword()` succeeds. The current flow redirects suspended users to `/account-suspended` and closed users to `/account-closed`, but the user already has a valid Supabase session/JWT at that point.

Codex classification: this is not true entitlement or authorization enforcement. It is route/navigation behavior only.

Required implementation posture:

- UI redirects may remain for user experience.
- Suspended/closed users must not retain useful client data access merely because they obtained a Supabase session.
- RLS policies and/or backend functions must check account status before returning or mutating sensitive data.
- Any Edge Function that performs privileged work must check account status before acting.
- The client should call sign-out immediately after detecting `suspended` or `closed`, but sign-out is defense-in-depth, not the primary security control.
- If Claude proposes a Supabase Auth hook or custom JWT claim path, Claude must pause and route the design for Codex/David approval before implementation because this changes auth architecture.
