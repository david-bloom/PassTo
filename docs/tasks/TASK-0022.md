## Code Reference

### 2.2-5 through 2.2-7 — Updated signUp flow in `CreateAccount.tsx`

Remove the `create-airtable-record` call and Airtable record ID. The `handle_new_user()` trigger creates the profiles row automatically when `signUp()` succeeds. Call `update_own_profile_basic()` immediately after to populate name and phone.

```js
// Step 1: Create Supabase Auth user
const { data: authData, error: signUpError } = await supabase.auth.signUp({
  email,
  password,
})

if (signUpError || !authData?.user) {
  // handle error — show message, do not proceed
  return
}

// Step 2: Check for active session before calling the RPC.
// If Supabase email confirmation is enabled, authData.session may be null.
// The RPC requires an authenticated session — do not call it without one.
if (authData.session) {
  // handle_new_user() trigger already created the profiles row.
  // update_own_profile_basic() only updates: first_name, last_name, phone,
  // display_name, preferred_name, notification_preferences.
  const { error: profileError } = await supabase.rpc('update_own_profile_basic', {
    p_first_name: firstName,
    p_last_name: lastName,
    p_phone: phone,
  })

  if (profileError) {
    // Log error — non-blocking. Profile row exists; name/phone will be null.
    console.error('update_own_profile_basic failed:', profileError)
  }

  // Step 3: Store Supabase Auth UUID as nurse identifier (replaces recXXXXXX)
  // authData.user.id === profiles.auth_user_id — NOT profiles.id
  update({ nurseId: authData.user.id })

  // Step 4: Navigate — onboarding_step defaults to 'identity'
  navigate('/verify-identity')
} else {
  // No session — email confirmation required.
  // RPC will be called after the nurse confirms email and logs in.
  navigate('/confirm-email')
}
```

### Profile fetch pattern (use after login and on app load)

```js
const { data: profile, error } = await supabase
  .from('profiles')
  .select('*')
  .maybeSingle()

// These are different states — handle them separately:
// error present          → network/auth/Supabase failure → show retry UI, do not route
// error null, profile null → row does not exist → show recovery/support UI (see null-profile guard)
// error null, profile set  → normal case → pass profile.onboarding_step to router
```

The `nurse_select_own_profile` RLS policy ensures this returns only the current nurse's row. No `.eq()` filter required.

### 2.2-8 through 2.2-9 — Onboarding router

```js
async function routeByOnboardingStep() {
  const { data: profile, error } = await supabase
    .from('profiles')
    .select('onboarding_step, account_status')
    .maybeSingle()

  if (error) {
    // Network or auth failure — show retry state
    showRetry()
    return
  }

  if (!profile) {
    // Authenticated but no profile row — show recovery state
    return navigate('/account-recovery')
  }

  if (profile.account_status === 'suspended') {
    return navigate('/account-suspended')
  }

  if (profile.account_status === 'closed') {
    return navigate('/account-closed')
  }

  switch (profile.onboarding_step) {
    case 'identity':  return navigate('/verify-identity')
    case 'phone':     return navigate('/verify-phone')
    case 'license':   return navigate('/license-lookup')
    case 'pass':      return navigate('/pass-ready')
    case 'complete':  return navigate('/dashboard')
    default:          return navigate('/verify-identity')  // safe fallback
  }
}
```

Call `routeByOnboardingStep()` on:
- App mount when a valid auth session is detected
- Immediately after `signIn()` returns without error
- Any time the enrollment app returns from an external flow (not needed in Phase 2 — placeholder for Phase 3)

## Error States

| State | Detection | Required Lovable behavior |
|---|---|---|
| Authenticated, no profile row | `error === null` and `profile === null` after `.maybeSingle()` | Show retry/support UI at `/account-recovery`; offer support contact and refresh option; do NOT attempt auto-recovery or call a backend function |
| Profile fetch failure | `error` object returned from Supabase client | Show retry UI; do not route to enrollment steps; do not treat as null-profile state |
| `account_status = 'suspended'` | `profile.account_status === 'suspended'` | Navigate to `/account-suspended`; show explanation; offer support contact |
| `account_status = 'closed'` | `profile.account_status === 'closed'` | Navigate to `/account-closed` |
| `onboarding_step` unexpected value | `default:` branch of router switch | Navigate to `/verify-identity` as safe fallback; log unexpected value |
| Auth session expired mid-enrollment | `supabase.auth.getSession()` returns no session | Redirect to login; preserve intended destination for post-login redirect if possible |
| signUp email already registered | `signUpError` indicates duplicate email | Show "account already exists" message; offer login link |
| `update_own_profile_basic()` fails at signUp | RPC returns error | Log error; do not block navigation — profile row exists, name/phone will be null |

## Security Boundaries

### What Lovable may do in Phase 2

| Action | Mechanism |
|---|---|
| Read own profile | `supabase.from('profiles').select('*').maybeSingle()` — RLS returns own row only |
| Update own name / phone / display_name / preferred_name / notification_preferences | `supabase.rpc('update_own_profile_basic', { ... })` |
| Sign up | `supabase.auth.signUp()` |
| Sign in | `supabase.auth.signInWithPassword()` |
| Sign out | `supabase.auth.signOut()` |
| Get current session | `supabase.auth.getSession()` / `supabase.auth.getUser()` |

### What Lovable must never do

| Forbidden action | Reason |
|---|---|
| `supabase.from('profiles').update({ onboarding_step: ... })` | Trust-gate field — service-role only, no authenticated UPDATE RLS policy |
| `supabase.from('profiles').update({ account_status: ... })` | Admin field — service-role only |
| `supabase.from('profiles').update({ id_verification_status: ... })` | Trust-gate field — service-role only |
| `supabase.from('profiles').update({ selfie_storage_path: ... })` | Trust-gate field — service-role only |
| `supabase.from('profiles').insert({ ... })` | Handled by `handle_new_user()` trigger — no authenticated INSERT policy |
| Use service-role key in Lovable | Must never appear in any Lovable project or Vite env var |
| Call `submit-enrollment` to write trust-gate state | Phase 3.1+ scope only — not Phase 2 |

No direct UPDATE, INSERT, or DELETE policy exists for the `authenticated` role on `profiles`. All authenticated nurse writes go through `update_own_profile_basic()` (RPC) or the `handle_new_user()` trigger (automatic).
