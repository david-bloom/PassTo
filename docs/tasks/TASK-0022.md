### 2.2-8 through 2.2-9 ‚Äî Onboarding router

```js
async function routeByOnboardingStep() {
  const { data: profile, error } = await supabase
    .from('profiles')
    .select('onboarding_step, account_status')
    .maybeSingle()

  if (error) {
    // Network or auth failure ‚Äî show retry state
    showRetry()
    return
  }

  if (!profile) {
    // Authenticated but no profile row ‚Äî show recovery state
    return navigate('/account-recovery')
  }

  if (profile.account_status === 'suspended') {
    return navigate('/account-suspended')
  }

  if (profile.account_status === 'closed') {
    return navigate('/account-closed')
  }

  switch (profile.onboarding_step) {
    case 'identity':  return navigate('/id-verification')
    case 'phone':     return navigate('/verify-phone')
    case 'license':   return navigate('/license-lookup')
    case 'pass':      return navigate('/pass-ready')
    case 'complete':  return navigate('/dashboard')
    default:          return navigate('/id-verification')  // safe fallback
  }
}
```

Call `routeByOnboardingStep()` on:
- App mount when a valid auth session is detected
- Immediately after `signIn()` returns without error
- Any time the enrollment app returns from an external flow (not needed in Phase 2 ‚Äî placeholder for Phase 3)

## Error States

| State | Detection | Required Lovable behavior |
|---|---|---|
| Authenticated, no profile row | `error === null` and `profile === null` after `.maybeSingle()` | Show retry/support UI at `/account-recovery`; offer support contact and refresh option; do NOT attempt auto-recovery or call a backend function |
| Profile fetch failure | `error` object returned from Supabase client | Show retry UI; do not route to enrollment steps; do not treat as null-profile state |
| `account_status = 'suspended'` | `profile.account_status === 'suspended'` | Navigate to `/account-suspended`; show explanation; offer support contact |
| `account_status = 'closed'` | `profile.account_status === 'closed'` | Navigate to `/account-closed` |
| `onboarding_step` unexpected value | `default:` branch of router switch | Navigate to `/id-verification` as safe fallback; log unexpected value |
| Auth session expired mid-enrollment | `supabase.auth.getSession()` returns no session | Redirect to login; preserve intended destination for post-login redirect if possible |
| signUp email already registered | `signUpError` indicates duplicate email | Show "account already exists" message; offer login link |
| signUp routes to `/confirm-email` | `authData.session === null` (email confirmation ON) | Expected ‚Äî do not treat as error |
| Post-login RPC fails | `update_own_profile_basic()` returns error | Log error; do not block routing ‚Äî profile row exists, name will be null until next login |
| `sessionStorage.pendingName` absent at login | Nurse logs in without a pending signup | Skip RPC call; route normally |

## Security Boundaries

### What Lovable may do in Phase 2

| Action | Mechanism |
|---|---|
| Read own profile | `supabase.from('profiles').select('*').maybeSingle()` ‚Äî RLS returns own row only |
| Update own name / phone / display_name / preferred_name / notification_preferences | `supabase.rpc('update_own_profile_basic', { ... })` |
| Sign up | `supabase.auth.signUp()` |
| Sign in | `supabase.auth.signInWithPassword()` |
| Sign out | `supabase.auth.signOut()` |
| Get current session | `supabase.auth.getSession()` / `supabase.auth.getUser()` |

### What Lovable must never do

| Forbidden action | Reason |
|---|---|
| `supabase.from('profiles').update({ onboarding_step: ... })` | Trust-gate field ‚Äî service-role only, no authenticated UPDATE RLS policy |
| `supabase.from('profiles').update({ account_status: ... })` | Admin field ‚Äî service-role only |
| `supabase.from('profiles').update({ id_verification_status: ... })` | Trust-gate field ‚Äî service-role only |
| `supabase.from('profiles').update({ selfie_storage_path: ... })` | Trust-gate field ‚Äî service-role only |
| `supabase.from('profiles').insert({ ... })` | Handled by `handle_new_user()` trigger ‚Äî no authenticated INSERT policy |
| Use service-role key in Lovable | Must never appear in any Lovable project or Vite env var |
| Call `submit-enrollment` to write trust-gate state | Phase 3.1+ scope only ‚Äî not Phase 2 |

No direct UPDATE, INSERT, or DELETE policy exists for the `authenticated` role on `profiles`. All authenticated nurse writes go through `update_own_profile_basic()` (RPC) or the `handle_new_user()` trigger (automatic).
