## Product Goal

Revise the onboarding backend so license lookup and identity binding happen **before** the nurse confirms their contact info, and so Lovable can offer a name-only fallback search when the nurse doesn't have their license number.

New flow:

```
/id-verification   ‚Üí ID.me identity verification
/license-info      ‚Üí Enter license details + start lookup
/license-checking  ‚Üí Loading / candidate selection (search fallback only)
/confirm-info      ‚Üí Confirm matched nurse + license + contact info
/phone-check       ‚Üí Twilio OTP
/account-select    ‚Üí Plan selection
/payment           ‚Üí Stripe (paid plans only)
/upload-selfie     ‚Üí Optional selfie
/success           ‚Üí Credential status
```

## New Edge Functions (not deployed ‚Äî awaiting approval)

### `license-lookup-start`

**verify_jwt:** true | **Method:** POST | **Gate:** `onboarding_step = 'license'`, IAL2, active

**Request body:**
```json
{
  "license_state":  "TX",
  "license_type":   "RN",
  "license_number": "751234"   // optional ‚Äî triggers search if omitted
}
```

**Behavior:**

1. Check `attempt_count` for this profile's current `license` step. If ‚â• 3, return `{ error: "max_attempts_reached", message_code: "contact_support" }` without calling RapidAPI.
2. Determine `search_mode`: `'verify'` if `license_number` provided, `'search'` if not.
3. If `search_mode = 'search'` and `license_type` is not in the RapidAPI-supported set, return `{ error: "search_not_supported_for_type" }` ‚Äî Lovable should have hidden the option, but gate it server-side too.
4. Create `license_lookups` row: `result = 'pending'`, `search_mode`, submitted fields, `triggered_by = 'onboarding'`.
5. Increment `attempt_count` on this lookup record.

**Verify path:** Call RapidAPI POST `/verify`. Apply full TASK-0046 validation (license number match, holder name required, `license_type` exact uppercase binding, status normalization, `complete_license_verification()` RPC):
- Success ‚Üí `license_lookups.result = 'success'`, `completed_at` ‚Üí advance `onboarding_step` to `'confirm'` ‚Üí return `{ result: 'success', next_step: 'confirm' }`
- Any failure ‚Üí `license_lookups.result = <failure_code>`, `completed_at` ‚Üí stay at `'license'` ‚Üí return failure with `message_code`

**Search path:** Call RapidAPI POST `/search` with `{ state, query: profile.last_name }`:
- Zero results ‚Üí `result = 'not_found'` ‚Üí stay at `'license'`
- One result ‚Üí resolve via POST `/verify` ‚Üí apply full TASK-0046 validation ‚Üí advance to `'confirm'` on success
- Multiple results ‚Üí `result = 'needs_selection'`, store allowlisted `candidate_data` ‚Üí advance `onboarding_step` to `'license_checking'` ‚Üí return `{ result: 'needs_selection', candidates: [...] }`
- Provider error ‚Üí `result = 'provider_error'` ‚Üí stay at `'license'`

**Candidate data format** (full name per David's decision; no discipline/DOB/raw payload):
```json
[
  {
    "license_number": "751234",
    "license_type":   "RN",
    "holder_name":    "Jane Smith"
  }
]
```

**Audit events:** `license.lookup_started`, `license.lookup_success`, `license.lookup_failed`, `license.search_needs_selection`, `license.search_resolved`, `license.provider_error`, `license.not_found`, `license.max_attempts_reached`.

## RapidAPI Behavior

| Input | Call | Resolution |
|---|---|---|
| License number provided | POST `/verify` | Synchronous full validation |
| No number, 1 search result | POST `/search` ‚Üí POST `/verify` | Two calls; full validation |
| No number, multiple results | POST `/search` ‚Üí `needs_selection` | Nurse selects ‚Üí `license-lookup-select` ‚Üí POST `/verify` |
| No number, 0 results | POST `/search` ‚Üí not_found | Stay at `license` |
| Any provider error | Fail closed | Stay at `license`; audit event |

**Search fallback hidden in Lovable UI for unsupported license types. Also blocked server-side in `license-lookup-start`.**

**3-attempt limit enforced server-side via `license_lookups.attempt_count`. After 3 attempts, `license-lookup-start` returns `max_attempts_reached` without calling RapidAPI.**

**A search result is never treated as sufficient for binding.** All binding goes through `/verify` with full TASK-0046 validation.
