## Findings

### P1 — `licenses.normalized_status` CHECK constraint not updated to match `license_status_mappings`

**Files:** `docs/architecture/V4_MIGRATION_SQL.md` (base migration lines 221–232; R3 remediation)
**Severity:** P1 — will cause a runtime CHECK violation when the license lookup Edge Function is built
**Blocked task:** Phase 3, item 3.3 (license lookup Edge Function)

**Issue:**

R3 correctly expanded `license_status_mappings.normalized_status` to 8 values including `'Pending'`:

```sql
-- R3: license_status_mappings CHECK
check (normalized_status = any (array[
  'Active','Inactive','Expired','Surrendered',
  'Revoked','Suspended','Pending','Unknown'
]))
```

R3 also inserted 5 new seed rows mapping raw statuses like `'Pending'`, `'Application Pending'`, `'Renewal Pending'` to `normalized_status = 'Pending'`.

However, the `licenses` table CHECK constraint was **not updated** by R3 and still only allows 7 values:

```sql
-- licenses table: still 7-value, no 'Pending'
normalized_status text
  check (normalized_status in (
    'Active', 'Inactive', 'Expired',
    'Surrendered', 'Revoked', 'Suspended', 'Unknown'
  )),
```

**Impact:**

When the license lookup Edge Function reads from `license_status_mappings` and gets `normalized_status = 'Pending'` for a raw input like `'Application Pending'`, then attempts to write that value to `licenses.normalized_status`, the database will throw:

```
ERROR: new row for relation "licenses" violates check constraint "licenses_normalized_status_check"
```

This is a silent data integrity gap that will not surface until Phase 3 Edge Function development begins.

**Required fix (new migration needed before Phase 3.3):**

```sql
alter table public.licenses
  drop constraint licenses_normalized_status_check;
alter table public.licenses
  add constraint licenses_normalized_status_check
  check (normalized_status = any (array[
    'Active','Inactive','Expired','Surrendered',
    'Revoked','Suspended','Pending','Unknown'
  ]));
```

The same 8-value array used in the R3 `license_status_mappings` constraint.

**Timing:** This fix is NOT blocking Phase 2 work (profile/auth/onboarding). It is blocking Phase 3, item 3.3. Apply before the license lookup Edge Function spec is executed.

### P2 — PRD Section 4 has two schema inconsistencies

**Files:** `docs/prd/PRD_SECTION_04_DATA_RLS_BACKEND.md`
**Severity:** P2 — non-blocking; PRD is stale relative to approved v4 design
**Impact:** Implementation tasks built from PRD Section 4 may reference wrong table names or incorrect scope

**Issue A — Section 4.4 lists `stripe_customers` as canonical:**

PRD Section 4.4 table list includes `stripe_customers` with `Launch Critical: Yes`. The v4 migration drops this table. The row should be removed from the table and replaced with a note that `stripe_customer_id` lives on `profiles`.

**Issue B — Section 4.11 marks `show_qr` as deferred but schema allows it:**

PRD Section 4.11 states:

```
MVP token types: share_link
Deferred token types: show_qr, pdf_qr
```

But the v4 schema allows `show_qr` in the `verification_tokens.token_type` CHECK constraint (originally per TASK-0007 OD-4 which approved `show_qr` as form-gated). No implementation guard exists in the build sequence, Edge Function specs, or task files preventing an Edge Function from creating `show_qr` tokens during Phase 3 build work.

**Recommendation:** PRD Section 4.11 should be updated to reflect the schema-includes-but-defers-implementation pattern. The `MVP_LAUNCH_CRITICAL_BUILD_SEQUENCE.md` should include an explicit guard: no Edge Function may create `show_qr` tokens until David reopens the scope.

### P3 — Hard-delete cascade chain is an accepted retained risk

**Files:** `docs/architecture/V4_MIGRATION_SQL.md`, `docs/activity_log/DECISION-0016-V4-MIGRATION-DIRECT-APPROVAL.md`
**Severity:** P3 — accepted risk; documented in DECISION-0016; no new action required now

**Issue (retained from prior Codex QA):**

Deleting an `auth.users` row cascades through:

```
auth.users → profiles (CASCADE) → licenses (CASCADE)
                                 → credentials (CASCADE)
                                 → verification_tokens (CASCADE)
                                 → verifiers (CASCADE)
                                 → verification_events (CASCADE)
                                 → license_lookups (CASCADE)
```

The only database-level hard-delete protection is `ON DELETE RESTRICT` on `payments.profile_id`, `pdf_exports.profile_id`, and `subscriptions.profile_id`. A free-tier nurse who has never paid anything has **no RESTRICT protection** — a hard delete of their `auth.users` row would silently destroy all license, credential, token, and verification history.

**Accepted risk context:** David accepted this risk via DECISION-0016. The agreed mitigation is app-level discipline: production code must use `profiles.deleted_at` for all normal account closures and must not hard-delete `auth.users` rows outside a formal purge process.

**Recommendation:** Add an explicit comment or procedure note in `SECURITY_MODEL.md` or an ADR: "Hard delete of auth.users is not an approved account-closure operation. Only soft-delete via profiles.deleted_at is permitted. Hard delete may only be invoked by a dedicated purge process enforced at 7+ years post-closure."
