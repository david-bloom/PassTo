## Product Goal

Revise the onboarding backend so license lookup and identity binding happen **before** the nurse confirms their contact info, and so Lovable can offer a name-only fallback search when the nurse doesn't have their license number handy.

New flow:

```
/id-verification   → ID.me identity verification
/license-info      → Enter license details + start lookup
/license-checking  → Loading / candidate selection (search fallback only)
/confirm-info      → Confirm matched nurse + license + contact info
/phone-check       → Twilio OTP
/account-select    → Plan selection
/payment           → Stripe (paid plans only)
/upload-selfie     → Optional selfie
/success           → Credential status
```

## New Onboarding Step Sequence

| `profiles.onboarding_step` | Route | Gate |
|---|---|---|
| `identity` | `/id-verification` | none (entry) |
| `license` | `/license-info` | ID.me verified, IAL2, account active |
| `license_checking` | `/license-checking` | same + search lookup pending candidate selection |
| `confirm` | `/confirm-info` | same + lookup complete + data_match_passed = true |
| `phone` | `/phone-check` | same + confirm step completed |
| `plan` | `/account-select` | + phone verified |
| `payment` | `/payment` | + paid plan selected |
| `selfie` | `/upload-selfie` | + free or confirmed paid entitlement |
| `pass` | `/success` | status surface only |

**`license_checking` step semantics:**  
This step is entered only when the nurse did not supply a license number and the RapidAPI `/search` returned multiple candidates requiring nurse selection. It is NOT entered for the fast-path `/verify` call (which resolves synchronously). It is also not entered when search returns zero or one result — zero → stay at `license` for retry; one → resolve via `/verify` then advance to `confirm`.

**Do not apply without David approval.**

```sql
-- ============================================================
-- Migration J: Extend license_lookups for TASK-0048 async
-- lookup lifecycle and search-mode tracking
--
-- What this adds:
--   1. submitted_license_number — null when search mode (no number)
--   2. submitted_state          — state submitted by nurse

-- What this fixes:
--   lookup_source default 'propelus' → null (set by Edge Function)

alter table public.license_lookups
  add column if not exists submitted_license_number text,
  add column if not exists submitted_state           text,
  add column if not exists submitted_license_type    text,
  add column if not exists search_mode               text,
  add column if not exists candidate_data            jsonb,
  add column if not exists completed_at              timestamptz;

-- Fix stale default on lookup_source
alter table public.license_lookups
  alter column lookup_source set default null;

comment on column public.license_lookups.submitted_license_number
  is 'Null when search_mode = search (nurse did not provide license number).';
comment on column public.license_lookups.search_mode
  is 'verify = POST /verify with license_number; search = POST /search with name+state.';
comment on column public.license_lookups.candidate_data

```

**Why this migration and not a new table:**  
`license_lookups` already tracks the lookup lifecycle with `result = pending/success/failed`. Extending it is smaller than creating a parallel `license_lookup_attempts` table. That table can be added in a later task when data-driven routing is implemented.

## New Edge Functions (not deployed — awaiting approval)

### `license-lookup-start`

**Purpose:** Nurse submits license info; backend starts lookup synchronously; advances step.

**verify_jwt:** true  
**Method:** POST  
**Gate:** `onboarding_step = 'license'`, ID.me verified, IAL2, account active

**Request body:**
```json
{
  "license_state":  "TX",
  "license_type":   "RN",
  "license_number": "751234"   // optional — triggers search if omitted
}
```

**Behavior:**

1. Validate inputs (state, license_type required; license_number optional).
2. Determine `search_mode`:
   - `license_number` provided → `'verify'` (fast path)
   - `license_number` absent → `'search'` (name fallback)
3. Create `license_lookups` row: `result = 'pending'`, `search_mode`, `submitted_*` fields, `triggered_by = 'onboarding'`.
4. **Verify path:** Call RapidAPI POST `/verify` with `{ license_number, state }`. Apply all existing TASK-0046 validation:
   - License number match, holder name required, `license_type` binding, status normalization, `data_match_passed`, `complete_license_verification()` RPC
   - On success → update `license_lookups.result = 'success'`, `completed_at = now()` → advance `onboarding_step` to `'confirm'` → return `{ result: 'success', next_step: 'confirm' }`
   - On failure → update `license_lookups.result = 'failed'`, `completed_at = now()` → stay at `'license'` → return failure with `message_code`
5. **Search path:** Call RapidAPI POST `/search` with `{ state, query: profile.last_name }`.
   - Zero results → `license_lookups.result = 'not_found'` → stay at `'license'` → return `{ result: 'not_found', message_code: 'license_not_found' }`
   - One result → resolve via POST `/verify` using the returned `license_number` → apply full TASK-0046 validation → on success advance to `'confirm'`; on failure stay at `'license'`
   - Multiple results → `license_lookups.result = 'needs_selection'`, store `candidate_data` (allowlisted: license_number, license_type, holder name — no discipline/DOB) → advance `onboarding_step` to `'license_checking'` → return `{ result: 'needs_selection', candidates: [...] }`
   - Provider error/timeout → `license_lookups.result = 'provider_error'` → stay at `'license'`

**Candidate allowlist for `candidate_data` (no PII beyond what nurse submitted or provider returns for name match):**
```json
[
  {
    "license_number": "751234",
    "license_type": "RN",
    "holder_name_display": "J. SMITH"  // first initial + last name only
  }
]
```
Full name is NOT returned to Lovable in the candidate list. Nurse selects by license number. Name is used server-side only for match.

**Write audit events** for: lookup_started, lookup_success, lookup_failed, search_needs_selection, search_resolved, provider_error, not_found.

### `license-lookup-select` (search candidate selection)

**Purpose:** Nurse selects a candidate from the multi-match search result. Backend resolves via `/verify` and binds.

**verify_jwt:** true  
**Method:** POST  
**Gate:** `onboarding_step = 'license_checking'`, ID.me verified, IAL2, account active

**Request body:**
```json
{
  "lookup_id":      "<uuid>",
  "license_number": "751234"
}
```

**Behavior:**
1. Validate that `lookup_id` belongs to this profile and is in `needs_selection` state.
2. Call RapidAPI POST `/verify` with the selected `license_number` + `state` from the original lookup record.
3. Apply full TASK-0046 validation (license number match, holder name, license_type binding, status, data_match_passed).
4. On success → update `license_lookups.result = 'success'`, `completed_at` → advance `onboarding_step` to `'confirm'` → return `{ result: 'success', next_step: 'confirm' }`
5. On failure → update `license_lookups.result = 'failed'`, advance `onboarding_step` back to `'license'` → return failure with `message_code`

### `confirm-info-complete`

**Purpose:** Nurse confirms their info on `/confirm-info`. Backend records the confirmation and advances step.

**verify_jwt:** true  
**Method:** POST  
**Gate:** `onboarding_step = 'confirm'`, ID.me verified, IAL2, account active, `data_match_passed = true`

**Request body:**
```json
{
  "confirmed_phone": "+15551234567"   // phone nurse wants to use for OTP
}
```

**Behavior:**
1. Validate phone format (E.164).
2. Validate all gates.
3. Store `confirmed_phone` temporarily on profile (NOT as verified phone — that happens after Twilio OTP).
4. Advance `onboarding_step` to `'phone'`.
5. Write audit event `confirm.info_confirmed`.
6. Return `{ success: true, next_step: 'phone' }`.

**Why a separate endpoint instead of updating `confirm-info-status`:** The confirm action writes state (phone intent + step advance). A GET should not write state.

## RapidAPI Behavior Summary

| Input | RapidAPI call | Resolution |
|---|---|---|
| License number provided | POST `/verify` with `{license_number, state}` | Synchronous — full TASK-0046 validation |
| No license number, single search result | POST `/search` with `{state, query: last_name}` → then POST `/verify` | Two calls; same TASK-0046 validation |
| No license number, multiple results | POST `/search` → return `needs_selection` to nurse | Nurse selects → `license-lookup-select` → POST `/verify` |
| No license number, zero results | POST `/search` → not_found | Stay at `license` |
| Provider timeout/error (either call) | Fail closed | Stay at `license`; audit event; no step advance |

**Never treat a search result as sufficient for binding without resolving through `/verify`.**  
Name-only search can return near-matches, maiden names, or credentials from another state. Only a confirmed license number match via `/verify` may produce `data_match_passed = true`.

### What Was Already Present

All five Edge Functions (and `migration_j`) were previously written in a prior session under the label TASK-0054. They are functionally correct but were blocked from deployment due to two critical schema issues (see below).

### Critical Issues Found and Fixed

**Issue 1: `complete_license_verification()` RPC advances to `'phone'`, not `'confirm'`**

Migration H set the target step to `'phone'`. The prior function implementations patched this with a post-RPC override:
```javascript
// override: RPC sets 'phone', correct to 'confirm'
.update({ onboarding_step: 'confirm' })
.in('onboarding_step', ['phone', 'confirm', 'license_checking'])
```
This is fragile and creates a race window. Migration J (Part 3) now updates the RPC to advance to `'confirm'` directly and accepts `'license'` OR `'license_checking'` as valid source steps. The post-RPC overrides in the Edge Functions become harmless no-ops.

**Issue 2: `onboarding_step` CHECK constraint missing `'license_checking'` and `'confirm'`**

The v4 baseline CHECK was: `('identity', 'phone', 'license', 'pass', 'complete')`. Any write of `'license_checking'` or `'confirm'` would fail with a constraint violation. Migration J (Part 2) drops and recreates the constraint with the full step set including TASK-0047 values (`plan`, `payment`, `selfie`) which were also absent from the original CHECK.

### Migration J (Updated)

Three-part migration — apply before any function deployment:

1. **Part 1:** Extend `license_lookups` with `submitted_license_number`, `submitted_state`, `submitted_license_type`, `search_mode`, `candidate_data`, `completed_at`. Fix stale `lookup_source` default.
2. **Part 2:** Drop/recreate `profiles_onboarding_step_check` with full step set.
3. **Part 3:** Update `complete_license_verification()` RPC — new source steps, new target step, preserved security grants.
