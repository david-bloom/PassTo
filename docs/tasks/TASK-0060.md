## Product Goal

Close the remaining TASK-0040 pre-production gap by proving a real nurse/profile can complete Stripe test checkout through Lovable and that Supabase state updates correctly from Stripe webhook metadata.

## Technical Scope

- Use Stripe test mode only.
- Exercise the Lovable payment route or an approved test path that reaches `onboarding_step = payment`.
- Complete Stripe Checkout with test card `4242 4242 4242 4242`.
- Verify `checkout.session.completed` creates or updates the correct `subscriptions` row.
- Verify `payments` records `subscription_start`.
- Verify `stripe_events` idempotency for the real checkout event.
- Verify profile payment step advances only after webhook-confirmed payment.
- Verify no Stripe secret or sensitive webhook data is exposed to Lovable.
- Document evidence, deviations, and any remaining pre-production gaps.

## Out of Scope

- Stripe live-mode cutover.
- Live Stripe product or price changes.
- Annual plans, coupons, trials, institutional billing, employer billing.
- Additional license purchase flow.
- Production launch.

## Security/RLS Impact

Security Impact: High

Paid entitlement must only activate from server-confirmed Stripe webhook state. Client navigation, success URLs, or query parameters must not activate entitlement.

## Acceptance Criteria

- [ ] A real test checkout through Lovable or approved test path completes with Stripe test card.
- [ ] `subscriptions.profile_id` is populated from Stripe subscription metadata.
- [ ] `subscriptions.status`, `plan_name`, and `license_entitlement_count` match the selected plan.
- [ ] `payments` records the subscription start event.
- [ ] `stripe_events` records the checkout event and remains idempotent on resend.
- [ ] Profile/onboarding advancement happens only after webhook-confirmed payment.
- [ ] Lovable receives only safe checkout/status payloads, not Stripe secrets.
- [ ] Claude documents evidence, tests run, deviations, and risks.

## QA Plan

- Run one Standard checkout and one Premier checkout if test personas are available.
- Resend the real checkout event and verify idempotency.
- Verify pending state before webhook confirmation.
- Verify payment failure or canceled checkout leaves entitlement inactive.
- Confirm no client-side route activates paid entitlement.
