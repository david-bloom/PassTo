## Design Decision - 2026-06-04: Stripe Integration Pattern

**Approved Approach:** Redirect-based Checkout (not embedded)

**Lovable Integration Pattern:**
```javascript
// /payment route: "Subscribe & Continue" handler
const { checkout_url } = await fetch('/.../stripe-checkout-create', {
  method: 'POST',
  headers: { Authorization: `Bearer ${jwt}` }
}).then(r => r.json());

window.location.href = checkout_url; // → Stripe Checkout
// User returns to: enroll.passtodigital.com/upload-selfie (success_url)
// Webhook auto-advances: onboarding_step = 'selfie'
```

**Why This Approach:**
- Backend (`stripe-checkout-create`) is deployed and protected by Supabase Auth.
- Webhook confirmation remains the payment source of truth.
- Redirect-based Checkout is the simplest approved pattern for TASK-0060 test-mode evidence.

**Future Enhancement:**
- After TASK-0060 closes, consider Lovable's embedded checkout (iframe, `enable_stripe_payments`)
- Better UX (no redirect), same webhook backend
- Defer to Phase 7+ pending product prioritization

**Lovable Implementation Guidance:**
1. Replace mock payment buttons in `/payment` route
2. On selection, call `stripe-checkout-create` with Authorization header (JWT)
3. Use returned `checkout_url` for redirect
4. On return from Stripe, frontend navigates to `/upload-selfie`
5. Webhook confirms payment; backend advances profile state

❌ **Lovable JWT integration issue discovered:**
- User is authenticated at page level
- "Subscribe & Continue" click triggers stripe-checkout-create call
- Edge Function returns 401 Unauthorized
- **Root cause:** JWT from magic-link session is not being passed in Authorization header to Edge Function

### The Issue

When Lovable calls `passtoSupabase.functions.invoke("stripe-checkout-create", ...)`:
1. User has valid Supabase Auth session from magic link
2. Lovable frontend checks `supabase.auth.getSession()` — returns valid session
3. Lovable should pass session JWT in Authorization header as `Bearer {jwt}`
4. **Instead:** Authorization header is not being set, or JWT is missing/malformed
5. Edge Function receives request without valid Authorization header → returns 401

### Evidence

- Frontend error: "You must be signed in to subscribe"
- Edge Function logs show 401 responses for checkout attempts
- Same error occurred in previous testing attempts with other auth methods

### Critical Discovery

The **Lovable integration with Supabase client-side function invocation needs review**. Specifically:
- How does Lovable pass JWT to `passtoSupabase.functions.invoke()`?
- Does `passtoSupabase` client have correct Supabase configuration?
- Is there a Lovable bug in Authorization header construction?
- Does the magic-link session set the right auth state in Lovable's Supabase client?

### Remaining Work for TASK-0060

Before executing real Stripe checkout, **Lovable JWT passing must be fixed**. Options:
