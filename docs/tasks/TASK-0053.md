## Product Goal

Verify that Phase 4 safely creates credentials, issues or tracks wallet passes, records durable provider state, and exposes honest success/status UX before downstream dashboard and verifier work depends on it.

## Technical Scope

- Review GitHub task docs and implementation notes.
- Review deployed Supabase Edge Functions and/or Vercel wallet routes.
- Verify credential creation gate behavior.
- Verify wallet signing contract and provider-state persistence.
- Verify `/success` / PassReady status behavior.
- Verify audit/event and ops visibility records.
- Verify RLS/direct-write boundaries.
- Record pass/block findings in GitHub task docs, activity log, and relevant issue/comment thread.

## Out of Scope

- Implementing fixes.
- Approving production launch.
- Approving live wallet provider/certificate changes.
- Phase 5 dashboard/share-link QA.
- Stripe QA beyond confirming Phase 4 uses server-confirmed entitlement state.

## Security/RLS Impact

Security Impact: High

Codex must explicitly test or review that client-side writes cannot create credentials, alter wallet provider truth, or bypass issuance gates.

## Design System Impact

Design System Impact: Medium  
Design Source: `/docs/design_system/README.md`

Codex should review whether `/success` status states are clear, non-misleading, and consistent with the product/status UI.

## Acceptance Criteria

- [ ] Credential creation refuses every missing/failed trust gate.
- [ ] Successful credential creation writes correct canonical state.
- [ ] Wallet signing contract keeps secrets backend-only.
- [ ] Wallet provider state is durable in `wallet_passes`.
- [ ] Wallet failure/pending states are visible and do not misrepresent status.
- [ ] `/success` direct navigation cannot skip gates.
- [ ] Audit/event records exist for success and failure.
- [ ] RLS/direct client access cannot mutate credential/wallet trust state.
- [ ] Codex records findings and next required action in GitHub.

## QA Plan

- Source review of implementation files and task docs.
- Live deployment/version check for relevant Edge Functions/routes.
- SQL/RLS review for credential and wallet tables.
- Manual or simulated test cases for all-gates-pass, missing gate, provider failure, pending wallet, and retry.
- User-facing review of `/success` states if Lovable UI is available.
