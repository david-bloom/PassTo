## Remediation Notes (2026-06-02)

### What happened

After the initial TASK-0065 implementation set Site URL to `https://app.passtodigital.com`, the user reported hitting a 404 at `https://app.passtodigital.com/id-verification` after login. Claude initially misdiagnosed this as the Lovable dashboard/verifier prompts having been applied to the wrong project, and began reverting the Supabase Auth URL Configuration toward `enroll.passtodigital.com`. Site URL was changed back to `https://enroll.passtodigital.com`; the three `app.passtodigital.com` redirect URLs were left in place, producing an inconsistent half-state.

David then clarified the 3-app architecture:

| Project | Lovable ID | Domain | Purpose |
|---|---|---|---|
| Marketing | `6c973fd1-2dcd-4377-8c98-4d2f0d68732e` | (TBD) | Landing |
| Enrollment | `d279ccd3-8397-4e7b-933c-8f5c8468d19e` | `enroll.passtodigital.com` | Sign-up, ID verification, license, wallet pass setup |
| App | `9a223cc4-ef58-43d4-929a-4c0424b586c2` | `app.passtodigital.com` | Nurse dashboard, share-link, `/v/:token`, password reset, account |

Under this architecture, the original `app.passtodigital.com` Site URL target was correct. The 404 root cause is **not** Supabase Site URL — it is the enrollment-app routing the user to a Site URL host that does not own the `/id-verification` route. The fix sits in Lovable `redirectTo` values, not in further Supabase URL config changes.

Claude paused all further changes, drafted a Codex review prompt summarizing the half-state and the proposed correction, and waited for David to relay Codex's verdict.
