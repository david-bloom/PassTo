## Product Goal

Define the MVP responsibility boundaries between Lovable, Supabase, and Vercel so PassTo implementation tasks do not drift across frontend, data, auth, RLS, orchestration, and sensitive backend/API concerns.

## Technical Scope

Create:

```text
/docs/architecture/LOVABLE_SUPABASE_VERCEL_RESPONSIBILITY_MAP.md
```

The map must define:

- What Lovable owns for MVP.
- What Supabase owns for MVP.
- What Vercel owns for MVP.
- Which responsibilities are explicitly not owned by each platform.
- Default routing rules for future task specs.
- MVP route ownership.
- Backend endpoint ownership.
- Integration ownership.
- Security and secret-handling rules.
- Open decisions and risks.

## Out of Scope

- App implementation.
- Lovable screen changes.
- Supabase migration application.
- RLS policy implementation.
- Vercel route implementation.
- Third-party integration setup.
- Final launch approval.

## Security/RLS Impact

Security Impact: Medium.

This task defines boundaries that will later inform security implementation:

- Lovable must not hold privileged secrets or perform trust decisions.
- Supabase RLS must protect product data.
- Service-role operations must run only in trusted backend contexts.
- Vercel is used only where a backend/API route is needed beyond Lovable or Supabase Edge Functions.

## Design System Impact

Design System Impact: Low.

This task does not create UI assets, but it maps which Lovable routes need future UI states.
