## Product Goal

Allow nurse onboarding to begin with ID.me verification instead of password-first account creation, while preserving backend trust boundaries, safe account/profile linking, and auditability.

## Technical Scope

- Update the ID.me backend workflow so `/id-verification` can start before a nurse has completed traditional account creation.
- Create or reuse a safe pre-account onboarding state, such as `onboarding_attempts`, if the current schema cannot safely represent this flow.
- Validate ID.me state and PKCE server-side.
- Exchange ID.me authorization code server-side only.
- Store verified identity result and safe contact fields needed for `/confirm-info`.
- Support profile/account creation or linking after ID.me verification and contact confirmation.
- Enforce one active profile/account linkage per ID.me subject unless Codex/David approve an exception.
- Record audit events for ID.me start, success, failure, expired/abandoned attempt, and account/profile link.
- Preserve the existing service-role boundary. Lovable must not receive secrets or make trust decisions.

## Claude Engineering Review — 2026-05-31

**Reviewer:** Claude / Senior Engineer  
**Review Status:** Complete — Awaiting David Approval and Codex Security Review before execution  
**Verdict:** Feasible — requires schema migration and `idme-exchange` rewrite before implementation

### 1. Technical Feasibility

The ID.me-first flow is technically feasible. The core challenge is bridging the gap between ID.me verification completing and a Supabase Auth account existing. The `profiles` table requires `auth_user_id NOT NULL`, so no profile row can exist before a Supabase Auth user is created. A pre-account state mechanism is required.

### 2. Schema Changes Required

**`onboarding_attempts` table is needed.** A JWT-only stateless approach is insufficient because:
- Server-side `UNIQUE(idme_subject)` prevents duplicate active enrollments from the same ID.me identity
- Resumability requires persisted state
- Abandoned/expired attempts need to be auditable

**Proposed Migration E SQL:**

```sql
CREATE TABLE public.onboarding_attempts (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  id_verification_level text,
  verified_first_name   text,
  verified_last_name    text,
  verified_email        text,
  verified_phone        text,
  pkce_state_hash       text NOT NULL,
  state                 text NOT NULL DEFAULT 'started'
                        CHECK (state IN ('started','id_verified','linked','abandoned','expired')),
  profile_id            uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  expires_at            timestamptz NOT NULL DEFAULT (now() + interval '30 minutes'),
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now()
);

-- Partial unique index: one active attempt per ID.me subject
CREATE UNIQUE INDEX onboarding_attempts_active_idme_subject_idx
  ON public.onboarding_attempts (id_me_subject)
  WHERE state NOT IN ('linked', 'abandoned', 'expired');

ALTER TABLE public.onboarding_attempts ENABLE ROW LEVEL SECURITY;
-- No authenticated client policies. Service-role only.
```

**Migration F also required:** Update `onboarding_step` CHECK constraint. New flow ordering is `identity → confirm → license → phone → plan → selfie → pass → complete`. Current constraint assumes `identity → phone → license`. This breaks `phone-send-otp` / `phone-verify-otp` which check `onboarding_step = 'phone'` before `license`.

### 3. Auth/Profile Linking Strategy

Recommended: service-role `admin.createUser()` called from a new `create-account` Edge Function after `/confirm-info`. The `handle_new_user()` trigger fires and creates the `profiles` row. The Edge Function immediately updates the row with verified fields. Magic link (not password) recommended for initial session — eliminates the password-first UX problem.

### 4. ID.me Subject Uniqueness

Two-layer approach:
- `onboarding_attempts` partial unique index (active attempts only) prevents simultaneous duplicate enrollments
- `profiles.id_me_subject UNIQUE` (already in v4) prevents final duplicate profile creation
- Conflict errors must not leak whether an email belongs to another user

### 5. Security/RLS Risks

Five risks identified:

| Risk | Severity | Mitigation |
|---|---|---|
| `idme-exchange` currently uses `verify_jwt: true` — incompatible with pre-account flow | High | New `idme-exchange-v2` with `verify_jwt: false` and custom PKCE+state validation. Requires Codex security review. |
| PKCE/state must be stored server-side, not just sessionStorage | High | `onboarding_attempts.pkce_state_hash` — hash of state stored at attempt creation |
| Nurse can change email at `/confirm-info`; collision with existing Supabase account | High | `create-account` Edge Function checks email existence without revealing it; writes audit event |
| `verified_phone` from ID.me must not be written to `profiles.phone` without Twilio OTP | High | `create-account` must not set `profiles.phone`; only `phone-verify-otp` may do so |
| `onboarding_attempts` must be service-role only — no authenticated client access | Medium | RLS enabled, no client policies defined |

### 6. Impact on TASK-0025

`idme-exchange` needs a rewrite. It currently:
- Requires `verify_jwt: true` (incompatible with pre-account flow)
- Writes directly to `profiles`
- Has no `onboarding_attempts` concept

Recommended: build new `idme-exchange-v2`. Bundle `IDME_ATTRIBUTES_URL` fix into v2. Deprecate existing function once v2 is validated.
