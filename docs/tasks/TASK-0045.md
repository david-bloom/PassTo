### C-1 — Approved validation pattern for unauthenticated ID.me callback

`idme-exchange-v2` may use `verify_jwt = false` only because the callback is pre-account. That setting removes Supabase's platform JWT gate, so the function must implement its own server-side validation before doing any token exchange, profile write, or Auth write.

Approved pattern:

```text
/id-verification/start
  creates onboarding_attempt
  generates state and PKCE verifier server-side
  stores only state_hash and encrypted/sealed PKCE verifier
  redirects to ID.me with state + S256 code_challenge

/auth/idme/callback
  verify_jwt = false
  accepts only ID.me code + state
  hashes received state and looks up one pending, unexpired attempt
  atomically marks attempt callback_received / locks attempt
  decrypts PKCE verifier server-side
  exchanges code with ID.me server-side
  validates ID.me response and assurance level
  stores verified identity result on the attempt
  redirects to /confirm-info with opaque attempt reference only
```

Required details:

- Use `state` as the primary CSRF/session correlation value.
- Store `state_hash`, not raw `state`.
- Use high-entropy random state; do not encode email, phone, name, or plan in state.
- Use PKCE with `code_challenge_method = S256`.
- The token exchange requires the raw `code_verifier`; therefore a hash-only verifier is not sufficient. Store the verifier encrypted/sealed with a server-only secret, or use an equivalent server-only sealed value. Do not expose the verifier to Lovable.
- Validate attempt status before token exchange: only `started` / `pending_callback` and not expired.
- Mark the attempt consumed/used atomically so callback replay cannot exchange or link twice.
- Reject missing, expired, reused, or mismatched state with a generic error page and audit event.
- Do not trust any caller-supplied profile/account identifiers on callback.
- Redirect only to approved first-party routes. Do not accept arbitrary `next` / redirect URLs from the browser.
- Do not include authorization code, state, access token, ID token, or provider details in user-facing redirects after callback handling.
- Do not write `profiles.id_verification_status`, create Auth users, or link profiles until the attempt has a validated ID.me result.

Implementation note: this may be one Edge Function with route branching or two Edge Functions. If split, only the provider callback must be `verify_jwt = false`; post-callback confirm/link actions should use an attempt-bound token plus server checks until a Supabase session exists.

### C-2 — `onboarding_attempts` schema requirements

The proposed `onboarding_attempts` model is the right direction, but a partial unique index for active attempts is not sufficient by itself.

Required fields:

```text
id uuid primary key
status text
state_hash text unique not null
code_verifier_ciphertext text not null
code_challenge_hash text
redirect_uri text not null
idme_subject text
id_verification_level text
email text
phone text
first_name text
last_name text
linked_auth_user_id uuid
linked_profile_id uuid
expires_at timestamptz not null
consumed_at timestamptz
created_at timestamptz not null
updated_at timestamptz not null
last_error_key text
attempt_ip inet
attempt_user_agent text
```

Required constraints/indexes:

- `state_hash` unique.
- Status check such as:

```text
started
callback_received
verified
linked
expired
failed
abandoned
```

- `expires_at > created_at`.
- `consumed_at` required when status is `verified` or `linked`.
- Partial unique index to prevent multiple active attempts for the same resolved ID.me subject after subject is known:

```text
unique (idme_subject)
where idme_subject is not null
  and status in ('verified', 'linked')
```

- Unique constraint or partial unique index preventing one ID.me subject from linking to multiple active profiles, either on `profiles.id_me_subject` or through an approved identity-link table.
- Partial index for cleanup/lookup:

```text
(expires_at) where status in ('started', 'callback_received', 'verified')
```

- No RLS policy that lets anonymous users read `onboarding_attempts`.
- No client-side direct insert/update to `onboarding_attempts`.
- Service-role only writes through Edge Functions.

Open design choice:

- Codex prefers an unexposed/server-only table. If it must be in `public`, RLS must be enabled with no anon/auth direct access except narrow server-controlled RPCs, but a private schema is cleaner if compatible with the existing Supabase project conventions.

Data minimization:

- Store only fields needed for confirmation and matching.
- Do not store raw provider access tokens unless absolutely required. If stored temporarily for debugging/provider follow-up, they must be encrypted, short-lived, and cleared after use.
- Store DOB or other sensitive matching attributes only if required for license matching, encrypted or otherwise minimized, and clear them after matching where possible.

### C-3 — Account creation mechanism

`supabase.auth.admin.createUser()` using the service-role client is approved for this ID.me-first flow, with constraints.

Rationale:

- The user is not authenticated yet, so browser-side `signUp()` does not fit the post-ID.me account creation moment cleanly.
- The account must be created or linked only after the backend has validated the ID.me attempt.
- The service-role key must remain server-side in the Edge Function.

Required constraints:

- Use `admin.createUser()` only after a validated ID.me attempt and contact confirmation.
- Do not use browser-supplied email/phone as trust source without comparing to the verified attempt and the nurse's confirmation step.
- Do not set `phone_confirm: true` from ID.me-returned phone. Phone possession still requires Twilio.
- `email_confirm: true` is allowed only if David/Codex explicitly accept ID.me email as sufficient for account email verification. Otherwise create the user and send a Supabase invite/magic-link/confirmation path.
- Do not set a user-chosen password in this pre-account flow unless a later approved UX requires it. Prefer passwordless/magic-link or generated recovery/invite flow.
- Store authorization/role/linking state in profile/application tables or app metadata where appropriate, not in user-editable metadata.
- Avoid account enumeration: existing-account conflicts must produce a generic recovery/linking message and support path.
- Profile creation/linking must be idempotent and protected by database uniqueness on `auth_user_id` and `id_me_subject`.

```text
        +

        +
server-held encrypted PKCE verifier
        +

        +
service-role admin.createUser/link after verified attempt
```

## Codex QA Review

### Codex Security QA — 2026-05-31

**Reviewer:** Codex  
**Scope:** `idme-exchange-v2`, `create-account`, Migration E/F live schema  
**Live Supabase Project Checked:** `wvzjfxacykgsaffskgtr`  
**Deployed Functions Checked:** `idme-exchange-v2` v2, `create-account` v2  
**Verdict:** Blocked

The two new functions are deployed with `verify_jwt = false`, as required for a pre-account flow, but the replacement validation is not sufficient yet. The current implementation does not meet the previously approved server-side state/PKCE pattern recorded in this task.

Live schema notes:

- `public.onboarding_attempts` exists with RLS enabled and no anon/auth policies. Service-role writes should bypass RLS.
- `public.onboarding_attempts` does not contain `state_hash`, `pkce_state_hash`, `code_verifier_ciphertext`, `consumed_at`, or server-side redirect/state binding fields.
- `public.onboarding_attempts` has a partial unique index on `idme_subject` where `state` is not `linked`, `abandoned`, or `expired`.
- `public.profiles.onboarding_step` CHECK includes `identity`, `confirm`, `license`, `phone`, `plan`, `selfie`, `pass`, and `complete`.
- `public.audit_events.actor_id` is nullable and references `profiles(id) ON DELETE SET NULL`; null actor audit rows are schema-compatible.

#### P1 — `idme-exchange-v2`: unauthenticated callback lacks server-side state validation

Pattern: `supabase/functions/idme-exchange-v2/index.ts` lines 42-73, 199-259

What is wrong:

- The function accepts only `code` and `code_verifier` from the browser.
- It does not accept, hash, or validate the OAuth `state` parameter server-side.
- It does not look up a pre-created pending `onboarding_attempt` before token exchange.
- It creates the `onboarding_attempts` row only after the provider token exchange and attributes call have already succeeded.
- PKCE format validation is not enough. Because the `code_verifier` is held in Lovable `sessionStorage`, the server cannot independently prove the callback belongs to a server-created enrollment attempt.

Attack surface opened by `verify_jwt = false`:

- Any internet caller can invoke the function directly; CORS only limits browser read access, not server-to-server invocation.
- The endpoint becomes a public ID.me code exchange endpoint for the PassTo client.
- CSRF/login-injection and callback-mixup protections depend on client-side state only.
- There is no server-side replay/consumption record for the authorization start event.

Required fix:

- Add a trusted `/id-verification/start` or equivalent backend branch that creates `onboarding_attempts` before redirecting to ID.me.
- Generate high-entropy `state` and PKCE verifier server-side.
- Store `state_hash` and an encrypted/sealed `code_verifier` or server-only equivalent.
- On callback, require `code + state`, hash `state`, find exactly one pending unexpired attempt, atomically mark it callback-received/consumed, then exchange the code with the stored verifier.
- Do not perform ID.me token exchange until server-side attempt/state validation passes.

#### P1 — `create-account`: account creation trusts browser-supplied email and pre-confirms it

Pattern: `supabase/functions/create-account/index.ts` lines 52-83, 86-90, 138-145

What is wrong:

- The function accepts `confirmed_email` from the browser and uses it as the Auth account email.
- The attempt lookup does not select `verified_email`, so the function never compares browser-submitted email to the ID.me-returned email.
- The function sets `email_confirm: true` for this browser-submitted email.
- A caller who completed ID.me for themselves could create a PassTo account under any unused email address and receive an immediate session token.

Required fix:

- Select `verified_email` from `onboarding_attempts`.
- If the nurse keeps the ID.me email, require exact normalized match before using `email_confirm: true`.
- If the nurse edits email, create the user without pre-confirming email and require Supabase email verification/magic-link possession before returning a session or advancing.
- Do not return a session for an email address that has not been verified or explicitly accepted as verified by the backend.

#### P1 — `create-account`: attempt consumption and profile linking are not atomic

Pattern: `supabase/functions/create-account/index.ts` lines 85-132, 138-228

What is wrong:

- The function reads an `id_verified` attempt, creates an Auth user, updates the profile, then marks the attempt `linked`.
- The attempt is not atomically claimed before account creation.
- Concurrent calls with the same `attempt_id` can both pass the initial state/profile checks before either marks the attempt linked.
- `profiles.id_me_subject UNIQUE` will stop final duplicate verified profiles, but only after Auth users and trigger-created profiles may already exist.
- The cleanup path uses best-effort `admin.deleteUser()` only. If the function times out, the network fails, or `deleteUser()` fails, partial Auth/profile state can persist.
- The attempt update to `linked` ignores `{ error }`, so the function can return a session even if the attempt remains active/stale.

Required fix:

- Atomically claim the attempt before `admin.createUser()`, for example `UPDATE onboarding_attempts SET state = 'account_creating' WHERE id = $1 AND state = 'id_verified' AND expires_at > now() RETURNING ...`.
- Add `account_creating` or equivalent to the state machine, with a recovery/expiry path.
- After Auth user creation, use a database RPC/transaction to update `profiles`, mark the attempt `linked`, and insert the success audit event together.
- Check every Supabase `{ error }` result.
- Add an operational remediation path for orphaned Auth/profile rows instead of relying only on best-effort immediate deletion.

#### P2 — `onboarding_attempts`: partial unique index has expiry and lifecycle gaps

Pattern: live index `onboarding_attempts_active_subject_idx`; `idme-exchange-v2` lines 206-259

What is wrong:

- The index excludes only rows whose `state` is `linked`, `abandoned`, or `expired`; it does not exclude rows whose `expires_at` is in the past.
- `idme-exchange-v2` checks whether an existing attempt is expired and then attempts to insert a new row, but the old expired-by-time row still participates in the unique index until its `state` is updated to `expired`.
- This can cause stale attempts to block new enrollment attempts for the same ID.me subject.
- The schema lacks `consumed_at` and database constraints tying terminal states to profile linkage/consumption.

Required fix:

- Before inserting a new attempt, mark expired active attempts for that subject as `expired`, or perform this through a single RPC.
- Add a cleanup job or function for expired attempts.
- Add lifecycle constraints such as `expires_at > created_at`, `consumed_at` for linked/consumed states, and a state transition model that distinguishes `id_verified`, `account_creating`, `linked`, `expired`, `failed`, and `abandoned`.
- Keep `profiles.id_me_subject UNIQUE`; it is still the final duplicate-profile backstop.

#### P2 — `create-account`: duplicate email detection relies on fragile message/status matching

Pattern: `supabase/functions/create-account/index.ts` lines 147-152

What is wrong:

- The code checks whether the error message contains `already` or `exists`, and treats any `status === 422` as duplicate.
- Supabase Auth exposes structured Auth error codes; the current duplicate email code is `email_exists`.
- Treating all `422` responses as duplicates can misclassify other validation/configuration failures and hide real setup issues.

Required fix:

- Detect duplicate email using the structured Auth error code:

```ts
const isDuplicate = createUserErr.code === "email_exists";
```

- If using a compatibility fallback, keep it narrow and log the full error server-side.
- Do not classify every `422` as a duplicate.

#### P2 — `create-account`: returned `hashed_token` is a bearer login credential

Pattern: `supabase/functions/create-account/index.ts` lines 240-263

What is wrong:

- `generateLink({ type: "magiclink" })` returns a token hash that can be exchanged with `verifyOtp()` for a Supabase session.
- Returning that token in a JSON body is effectively returning a one-time bearer login credential to browser JavaScript.
- HTTPS protects transport, but it does not protect against XSS, browser extension access, frontend logging, response caching, or misuse after `attempt_id` leakage.
- This is especially risky while `create-account` is gated only by an unbound `attempt_id`.

Required fix:

- Preferred: send the magic link to the verified email, or use a server-side `/auth/confirm` route that exchanges the token and sets secure, `HttpOnly`, `SameSite` cookies.
- If a SPA response-body token remains temporarily necessary, first fix server-side state/attempt binding and atomic consumption, then add `Cache-Control: no-store`, avoid logging the token, and immediately exchange it with `verifyOtp({ token_hash, type: "email" })`.
- Do not call the returned value a `session_token`; it is an OTP token hash, not a Supabase session.

#### P2 — `idme-exchange-v2`: provider response validation is not pinned enough for production

Pattern: `supabase/functions/idme-exchange-v2/index.ts` lines 121-140, 147-178

What is wrong:

- IAL/subject extraction is still heuristic across several possible fields.
- Diagnostic logging of provider response shape remains enabled.
- The function does not validate an ID.me issuer/audience-bound ID token or a pinned expected UserInfo/Attributes response contract.

Required fix:

- Pin the exact ID.me sandbox/production response fields before production.
- Validate the expected issuer/audience/subject source for the ID.me result, or document why the chosen ID.me endpoint is authoritative.
- Remove diagnostic provider-shape logging before production.
- Fail closed if the expected assurance/subject fields are absent or ambiguous.

#### P3 — both functions: CORS is useful but incomplete and must not be treated as authentication

Pattern: `supabase/functions/idme-exchange-v2/index.ts` lines 22-34; `supabase/functions/create-account/index.ts` lines 27-39

What is wrong:

- `Access-Control-Allow-Origin` is restricted to `https://enroll.passtodigital.com`, which is good for browser clients.
- The `OPTIONS` response does not include `Access-Control-Allow-Methods`.
- CORS does not prevent non-browser clients from invoking `verify_jwt = false` functions.

Required fix:

- Add at least:

```ts
"Access-Control-Allow-Methods": "POST, OPTIONS"
```

- Consider `Vary: Origin` and `Cache-Control: no-store`.
- Keep all real authorization in server-side state/attempt validation, not CORS.

#### P3 — both functions: Supabase insert/update errors are not consistently inspected

Pattern: `idme-exchange-v2` lines 263-276; `create-account` lines 220-234 and 281-288

What is wrong:

- Supabase client calls return `{ error }`; they do not necessarily throw.
- Audit writes and the `onboarding_attempts` linked update can fail silently.

Required fix:

- Inspect `{ error }` on every Supabase call.
- For success-path audit/linking, fail closed or return a recovery state if the write is part of the trust boundary.
- Keep non-blocking audit only for explicitly approved failure telemetry.

#### Validated Items

- `audit_events.actor_id = null` is schema-compatible. The column is nullable and the FK is `ON DELETE SET NULL`.
- Service-role inserts into `audit_events` and `onboarding_attempts` should bypass RLS. RLS is enabled and no anon/auth policies are present.
- Broad table grants to `anon` and `authenticated` exist on `onboarding_attempts` and `audit_events`, but RLS currently blocks access because there are no policies. Codex recommends revoking unnecessary anon/auth grants as defense in depth.

### Required Before Approval

- Implement server-side state + PKCE storage/validation before ID.me token exchange.
- Bind `create-account` to a server-validated, atomically claimed attempt.
- Stop pre-confirming browser-edited email.
- Replace fragile duplicate-email detection with `email_exists`.
- Resolve the session-establishment pattern and avoid returning bearer login material unless the attempt model is fixed and response hardening is added.
- Make profile update, attempt linking, and success audit atomic or explicitly recoverable.

**Verdict:** Feasible — requires schema migration and `idme-exchange` rewrite before implementation

### 1. Technical Feasibility

The ID.me-first flow is technically feasible. The core challenge is bridging the gap between ID.me verification completing and a Supabase Auth account existing. The `profiles` table requires `auth_user_id NOT NULL`, so no profile row can exist before a Supabase Auth user is created. A pre-account state mechanism is required.

### 2. Schema Changes Required

**`onboarding_attempts` table is needed.** A JWT-only stateless approach is insufficient because:
- Server-side `UNIQUE(idme_subject)` prevents duplicate active enrollments from the same ID.me identity
- Resumability requires persisted state
- Abandoned/expired attempts need to be auditable

**Proposed Migration E SQL:**

```sql
CREATE TABLE public.onboarding_attempts (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  id_verification_level text,
  verified_first_name   text,
  verified_last_name    text,
  verified_email        text,
  verified_phone        text,
  pkce_state_hash       text NOT NULL,
  state                 text NOT NULL DEFAULT 'started'
                        CHECK (state IN ('started','id_verified','linked','abandoned','expired')),
  profile_id            uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  expires_at            timestamptz NOT NULL DEFAULT (now() + interval '30 minutes'),
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now()
);

-- Partial unique index: one active attempt per ID.me subject
CREATE UNIQUE INDEX onboarding_attempts_active_idme_subject_idx
  ON public.onboarding_attempts (id_me_subject)
  WHERE state NOT IN ('linked', 'abandoned', 'expired');

ALTER TABLE public.onboarding_attempts ENABLE ROW LEVEL SECURITY;
-- No authenticated client policies. Service-role only.
```

**Migration F also required:** Update `onboarding_step` CHECK constraint. New flow ordering is `identity → confirm → license → phone → plan → selfie → pass → complete`. Current constraint assumes `identity → phone → license`. This breaks `phone-send-otp` / `phone-verify-otp` which check `onboarding_step = 'phone'` before `license`.

### 3. Auth/Profile Linking Strategy

Recommended: service-role `admin.createUser()` called from a new `create-account` Edge Function after `/confirm-info`. The `handle_new_user()` trigger fires and creates the `profiles` row. The Edge Function immediately updates the row with verified fields. Magic link (not password) recommended for initial session — eliminates the password-first UX problem.

### 4. ID.me Subject Uniqueness

Two-layer approach:
- `onboarding_attempts` partial unique index (active attempts only) prevents simultaneous duplicate enrollments
- `profiles.id_me_subject UNIQUE` (already in v4) prevents final duplicate profile creation
- Conflict errors must not leak whether an email belongs to another user

### 5. Security/RLS Risks

Five risks identified:

| Risk | Severity | Mitigation |
|---|---|---|
| `idme-exchange` currently uses `verify_jwt: true` — incompatible with pre-account flow | High | New `idme-exchange-v2` with `verify_jwt: false` and custom PKCE+state validation. Requires Codex security review. |
| PKCE/state must be stored server-side, not just sessionStorage | High | `onboarding_attempts.pkce_state_hash` — hash of state stored at attempt creation |
| Nurse can change email at `/confirm-info`; collision with existing Supabase account | High | `create-account` Edge Function checks email existence without revealing it; writes audit event |
| `verified_phone` from ID.me must not be written to `profiles.phone` without Twilio OTP | High | `create-account` must not set `profiles.phone`; only `phone-verify-otp` may do so |
| `onboarding_attempts` must be service-role only — no authenticated client access | Medium | RLS enabled, no client policies defined |
