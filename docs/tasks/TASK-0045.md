## Product Goal

Allow nurse onboarding to begin with ID.me verification instead of password-first account creation, while preserving backend trust boundaries, safe account/profile linking, and auditability.

## Technical Scope

- Update the ID.me backend workflow so `/id-verification` can start before a nurse has completed traditional account creation.
- Create or reuse a safe pre-account onboarding state, such as `onboarding_attempts`, if the current schema cannot safely represent this flow.
- Validate ID.me state and PKCE server-side.
- Exchange ID.me authorization code server-side only.
- Store verified identity result and safe contact fields needed for `/confirm-info`.
- Support profile/account creation or linking after ID.me verification and contact confirmation.
- Enforce one active profile/account linkage per ID.me subject unless Codex/David approve an exception.
- Record audit events for ID.me start, success, failure, expired/abandoned attempt, and account/profile link.
- Preserve the existing service-role boundary. Lovable must not receive secrets or make trust decisions.

### C-1 — Approved validation pattern for unauthenticated ID.me callback

`idme-exchange-v2` may use `verify_jwt = false` only because the callback is pre-account. That setting removes Supabase's platform JWT gate, so the function must implement its own server-side validation before doing any token exchange, profile write, or Auth write.

Approved pattern:

```text
/id-verification/start
  creates onboarding_attempt
  generates state and PKCE verifier server-side
  stores only state_hash and encrypted/sealed PKCE verifier
  redirects to ID.me with state + S256 code_challenge

/auth/idme/callback
  verify_jwt = false
  accepts only ID.me code + state
  hashes received state and looks up one pending, unexpired attempt
  atomically marks attempt callback_received / locks attempt
  decrypts PKCE verifier server-side
  exchanges code with ID.me server-side
  validates ID.me response and assurance level
  stores verified identity result on the attempt
  redirects to /confirm-info with opaque attempt reference only
```

Required details:

- Use `state` as the primary CSRF/session correlation value.
- Store `state_hash`, not raw `state`.
- Use high-entropy random state; do not encode email, phone, name, or plan in state.
- Use PKCE with `code_challenge_method = S256`.
- The token exchange requires the raw `code_verifier`; therefore a hash-only verifier is not sufficient. Store the verifier encrypted/sealed with a server-only secret, or use an equivalent server-only sealed value. Do not expose the verifier to Lovable.
- Validate attempt status before token exchange: only `started` / `pending_callback` and not expired.
- Mark the attempt consumed/used atomically so callback replay cannot exchange or link twice.
- Reject missing, expired, reused, or mismatched state with a generic error page and audit event.
- Do not trust any caller-supplied profile/account identifiers on callback.
- Redirect only to approved first-party routes. Do not accept arbitrary `next` / redirect URLs from the browser.
- Do not include authorization code, state, access token, ID token, or provider details in user-facing redirects after callback handling.
- Do not write `profiles.id_verification_status`, create Auth users, or link profiles until the attempt has a validated ID.me result.

Implementation note: this may be one Edge Function with route branching or two Edge Functions. If split, only the provider callback must be `verify_jwt = false`; post-callback confirm/link actions should use an attempt-bound token plus server checks until a Supabase session exists.

### C-2 — `onboarding_attempts` schema requirements

The proposed `onboarding_attempts` model is the right direction, but a partial unique index for active attempts is not sufficient by itself.

Required fields:

```text
id uuid primary key
status text
state_hash text unique not null
code_verifier_ciphertext text not null
code_challenge_hash text
redirect_uri text not null
idme_subject text
id_verification_level text
email text
phone text
first_name text
last_name text
linked_auth_user_id uuid
linked_profile_id uuid
expires_at timestamptz not null
consumed_at timestamptz
created_at timestamptz not null
updated_at timestamptz not null
last_error_key text
attempt_ip inet
attempt_user_agent text
```

Required constraints/indexes:

- `state_hash` unique.
- Status check such as:

```text
started
callback_received
verified
linked
expired
failed
abandoned
```

- `expires_at > created_at`.
- `consumed_at` required when status is `verified` or `linked`.
- Partial unique index to prevent multiple active attempts for the same resolved ID.me subject after subject is known:

```text
unique (idme_subject)
where idme_subject is not null
  and status in ('verified', 'linked')
```

- Unique constraint or partial unique index preventing one ID.me subject from linking to multiple active profiles, either on `profiles.id_me_subject` or through an approved identity-link table.
- Partial index for cleanup/lookup:

```text
(expires_at) where status in ('started', 'callback_received', 'verified')
```

- No RLS policy that lets anonymous users read `onboarding_attempts`.
- No client-side direct insert/update to `onboarding_attempts`.
- Service-role only writes through Edge Functions.

Open design choice:

- Codex prefers an unexposed/server-only table. If it must be in `public`, RLS must be enabled with no anon/auth direct access except narrow server-controlled RPCs, but a private schema is cleaner if compatible with the existing Supabase project conventions.

Data minimization:

- Store only fields needed for confirmation and matching.
- Do not store raw provider access tokens unless absolutely required. If stored temporarily for debugging/provider follow-up, they must be encrypted, short-lived, and cleared after use.
- Store DOB or other sensitive matching attributes only if required for license matching, encrypted or otherwise minimized, and clear them after matching where possible.

### C-3 — Account creation mechanism

`supabase.auth.admin.createUser()` using the service-role client is approved for this ID.me-first flow, with constraints.

Rationale:

- The user is not authenticated yet, so browser-side `signUp()` does not fit the post-ID.me account creation moment cleanly.
- The account must be created or linked only after the backend has validated the ID.me attempt.
- The service-role key must remain server-side in the Edge Function.

Required constraints:

- Use `admin.createUser()` only after a validated ID.me attempt and contact confirmation.
- Do not use browser-supplied email/phone as trust source without comparing to the verified attempt and the nurse's confirmation step.
- Do not set `phone_confirm: true` from ID.me-returned phone. Phone possession still requires Twilio.
- `email_confirm: true` is allowed only if David/Codex explicitly accept ID.me email as sufficient for account email verification. Otherwise create the user and send a Supabase invite/magic-link/confirmation path.
- Do not set a user-chosen password in this pre-account flow unless a later approved UX requires it. Prefer passwordless/magic-link or generated recovery/invite flow.
- Store authorization/role/linking state in profile/application tables or app metadata where appropriate, not in user-editable metadata.
- Avoid account enumeration: existing-account conflicts must produce a generic recovery/linking message and support path.
- Profile creation/linking must be idempotent and protected by database uniqueness on `auth_user_id` and `id_me_subject`.

## Claude Engineering Review — 2026-05-31

**Reviewer:** Claude / Senior Engineer  
**Review Status:** Complete — Awaiting David Approval and Codex Security Review before execution  
**Verdict:** Feasible — requires schema migration and `idme-exchange` rewrite before implementation

### 1. Technical Feasibility

The ID.me-first flow is technically feasible. The core challenge is bridging the gap between ID.me verification completing and a Supabase Auth account existing. The `profiles` table requires `auth_user_id NOT NULL`, so no profile row can exist before a Supabase Auth user is created. A pre-account state mechanism is required.

### 2. Schema Changes Required

**`onboarding_attempts` table is needed.** A JWT-only stateless approach is insufficient because:
- Server-side `UNIQUE(idme_subject)` prevents duplicate active enrollments from the same ID.me identity
- Resumability requires persisted state
- Abandoned/expired attempts need to be auditable

**Proposed Migration E SQL:**

```sql
CREATE TABLE public.onboarding_attempts (
  id                    uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  id_verification_level text,
  verified_first_name   text,
  verified_last_name    text,
  verified_email        text,
  verified_phone        text,
  pkce_state_hash       text NOT NULL,
  state                 text NOT NULL DEFAULT 'started'
                        CHECK (state IN ('started','id_verified','linked','abandoned','expired')),
  profile_id            uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  expires_at            timestamptz NOT NULL DEFAULT (now() + interval '30 minutes'),
  created_at            timestamptz NOT NULL DEFAULT now(),
  updated_at            timestamptz NOT NULL DEFAULT now()
);

-- Partial unique index: one active attempt per ID.me subject
CREATE UNIQUE INDEX onboarding_attempts_active_idme_subject_idx
  ON public.onboarding_attempts (id_me_subject)
  WHERE state NOT IN ('linked', 'abandoned', 'expired');

ALTER TABLE public.onboarding_attempts ENABLE ROW LEVEL SECURITY;
-- No authenticated client policies. Service-role only.
```

**Migration F also required:** Update `onboarding_step` CHECK constraint. New flow ordering is `identity → confirm → license → phone → plan → selfie → pass → complete`. Current constraint assumes `identity → phone → license`. This breaks `phone-send-otp` / `phone-verify-otp` which check `onboarding_step = 'phone'` before `license`.

### 3. Auth/Profile Linking Strategy

Recommended: service-role `admin.createUser()` called from a new `create-account` Edge Function after `/confirm-info`. The `handle_new_user()` trigger fires and creates the `profiles` row. The Edge Function immediately updates the row with verified fields. Magic link (not password) recommended for initial session — eliminates the password-first UX problem.

### 4. ID.me Subject Uniqueness

Two-layer approach:
- `onboarding_attempts` partial unique index (active attempts only) prevents simultaneous duplicate enrollments
- `profiles.id_me_subject UNIQUE` (already in v4) prevents final duplicate profile creation
- Conflict errors must not leak whether an email belongs to another user

### 5. Security/RLS Risks

Five risks identified:

| Risk | Severity | Mitigation |
|---|---|---|
| `idme-exchange` currently uses `verify_jwt: true` — incompatible with pre-account flow | High | New `idme-exchange-v2` with `verify_jwt: false` and custom PKCE+state validation. Requires Codex security review. |
| PKCE/state must be stored server-side, not just sessionStorage | High | `onboarding_attempts.pkce_state_hash` — hash of state stored at attempt creation |
| Nurse can change email at `/confirm-info`; collision with existing Supabase account | High | `create-account` Edge Function checks email existence without revealing it; writes audit event |
| `verified_phone` from ID.me must not be written to `profiles.phone` without Twilio OTP | High | `create-account` must not set `profiles.phone`; only `phone-verify-otp` may do so |
| `onboarding_attempts` must be service-role only — no authenticated client access | Medium | RLS enabled, no client policies defined |

### 6. Impact on TASK-0025

`idme-exchange` needs a rewrite. It currently:
- Requires `verify_jwt: true` (incompatible with pre-account flow)
- Writes directly to `profiles`
- Has no `onboarding_attempts` concept

Recommended: build new `idme-exchange-v2`. Bundle `IDME_ATTRIBUTES_URL` fix into v2. Deprecate existing function once v2 is validated.
