## Approved Storage Model

Bucket name:

```text
selfies
```

Path format:

```text
{auth_user_id}/selfie.jpg
```

Upload pattern:

```text
Lovable direct upload with authenticated user JWT, followed by backend Edge Function confirmation.
```

Access model:

- Nurse may upload, overwrite, and read only their own object path.
- Nurse may not delete selfie objects.
- Anonymous users have no access.
- Verifiers have no access.
- Backend Edge Functions and approved ops access use service-role contexts only.

## Migration SQL

Use a standalone migration name:

```text
v4_passto_mvp_selfie_storage
```

Required profile-field migration:

```sql
alter table public.profiles
  add column if not exists selfie_storage_path text,
  add column if not exists selfie_captured_at timestamptz;

comment on column public.profiles.selfie_storage_path
  is 'Supabase Storage path of the nurse selfie within the selfies bucket. Format: {auth_user_id}/selfie.jpg. Service-role only. Set by backend confirmation after upload validation, not by nurse client directly.';

comment on column public.profiles.selfie_captured_at
  is 'Timestamp when the selfie was successfully captured and confirmed by backend. Service-role only. Null until selfie step is confirmed.';
```

## Storage Policies

Apply these policies to `storage.objects` after the private `selfies` bucket exists.

```sql
create policy "nurse_upload_own_selfie"
  on storage.objects
  for insert
  to authenticated
  with check (
    bucket_id = 'selfies'
    and (storage.foldername(name))[1] = auth.uid()::text
  );

create policy "nurse_update_own_selfie"
  on storage.objects
  for update
  to authenticated
  using (
    bucket_id = 'selfies'
    and (storage.foldername(name))[1] = auth.uid()::text
  )
  with check (
    bucket_id = 'selfies'
    and (storage.foldername(name))[1] = auth.uid()::text
  );

create policy "nurse_select_own_selfie"
  on storage.objects
  for select
  to authenticated
  using (
    bucket_id = 'selfies'
    and (storage.foldername(name))[1] = auth.uid()::text
  );
```

No authenticated DELETE policy is approved for MVP.

## Risks / Guardrails

- Selfies are biometric PII. Bucket must remain private.
- Storage RLS only proves path ownership. It does not prove data-match completion. The future backend confirmation step must enforce onboarding state before confirming `selfie_storage_path`.
- The profile fields are trust-gate fields and must remain service-role write only.
- No long-lived signed URLs are approved for MVP.
- A future production purge process must delete selfie objects when retention rules allow account data purge.
