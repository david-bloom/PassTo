## Product Goal

Create the PassTo credential record only after all required trust, payment, and selfie gates are satisfied by backend-owned state.

## Technical Scope

- Implement or revise a backend credential creation operation.
- Enforce server-side gate checks before creating a credential:
  - ID.me verified at IAL2.
  - Account active.
  - License lookup succeeded with provider status/type/name safeguards.
  - ID.me identity and license record are bound with `data_match_passed = true`.
  - Phone possession verified through Twilio.
  - Plan/payment state satisfies the selected plan path.
  - Selfie gate is satisfied if selfie is required for MVP.
- Create the credential row with canonical profile/license relationship.
- Populate credential status, license status/current-as-of, and wallet pass treatment.
- Write audit/event records for success and failure.
- Fail closed on missing or ambiguous provider/license/status facts.

## Out of Scope

- Wallet pass signing or provider calls.
- Apple/Google wallet certificate setup.
- Share-link/verifier flow.
- Stripe checkout/webhook implementation beyond reading confirmed entitlement state.
- Lovable UI beyond invoking the approved backend operation.

## Acceptance Criteria

- [ ] Backend refuses credential creation unless all required gates pass.
- [ ] Credential row is created exactly once for the eligible profile/license path or fails idempotently.
- [ ] Credential status and current-as-of fields reflect canonical license state.
- [ ] Non-active, unknown, missing, or mismatched license status blocks issuance.
- [ ] Paid entitlement is read from server-confirmed subscription state, not client selection.
- [ ] Failure path writes audit/event state and does not partially issue a wallet pass.
- [ ] Lovable cannot bypass the backend gate through direct Supabase writes.
- [ ] Claude documents files changed, tests run, deviations, and risks.

## QA Plan

- Test all-gates-pass credential creation.
- Test each missing/failed gate blocks creation.
- Test duplicate/retry behavior does not create unsafe duplicate active credentials.
- Verify credential data, audit events, and failure records.
- Verify authenticated client cannot directly create/update credential trust fields.
- Verify wallet issuance is not triggered until TASK-0050/TASK-0051 contract is ready.
