## Product Goal

Create the PassTo credential record only after all required trust, payment, and selfie gates are satisfied by backend-owned state.

## Technical Scope

- Implement or revise a backend credential creation operation.
- Enforce server-side gate checks before creating a credential:
  - ID.me verified at IAL2.
  - Account active.
  - License lookup succeeded with provider status/type/name safeguards.
  - ID.me identity and license record are bound with `data_match_passed = true`.
  - Phone possession verified through Twilio.
  - Plan/payment state satisfies the selected plan path.
  - Selfie gate is satisfied if selfie is required for MVP.
- Create the credential row with canonical profile/license relationship.
- Populate credential status, license status/current-as-of, and wallet pass treatment.
- Write audit/event records for success and failure.
- Fail closed on missing or ambiguous provider/license/status facts.

**No migration required.** `credentials` table exists from v4 schema with all required columns.

### Function: `credential-create`

**verify_jwt:** true | **Method:** POST | **CORS:** `enroll.passtodigital.com`

**Gate sequence:**
1. Account active
2. ID.me verified at IAL2
3. `onboarding_step = 'pass'` (or `'complete'` → idempotent return)
4. `profiles.phone` set (written by `phone-verify-otp` after Twilio success)
5. Primary license: `data_match_passed = true`, `normalized_status = 'Active'`, `status_intent = 'credential_valid'`, `wallet_pass_treatment in ('valid', 'caution')`
6. Payment: `subscription_tier = 'free'` → no subscription required; paid plan → active `subscriptions` row required

**Idempotency:** Returns existing credential if already created for this profile/license pair. Does not create duplicates.

**`pass_template_data`** fields: `nurse_name`, `license_holder` (provider-verified), `license_type`, `license_state`, `license_number`, `normalized_status`, `expiration_date`, `credential_created`. No raw provider payload.

**Order of writes:**
1. Audit event (`credential.created`) — fail-closed, aborts if fails
2. Credential row insert (status = `pending`)
3. `onboarding_step` advance: `pass → complete` (fail-closed row check)

**Wallet/PassKit:** Not called. Credential status set to `pending`. PassKit issuance is TASK-0050.

**Files changed:**
- `supabase/functions/credential-create/index.ts` — created

### Deviations

- No RPC created for credential insert. Prior trust-critical writes used SECURITY DEFINER RPCs (`complete_license_verification`, `complete_phone_verification`). Credential creation uses service-role client writes with fail-closed error handling. An RPC would require a migration; direct service-role writes are consistent with the simpler functions and the insert is to a single table. Documented as deviation — Codex may recommend an RPC in QA.
- If the step advance fails after successful credential insert, the function returns 200 with `warning: step_advance_failed` rather than 500. The credential exists and the nurse is functionally complete; a 500 here would cause Lovable to retry and hit the idempotency path.

### Risks

- Credential row has no unique constraint on `(profile_id, license_id)` in the v4 schema. Idempotency is enforced in the Edge Function only. A race condition (two simultaneous calls) could create duplicate rows. For MVP with no concurrent nurse sessions this is low risk; a unique constraint migration would eliminate it.
- `pass_template_data` includes `license_number` in full. Lovable must not render this field directly in the UI without masking. The `/confirm-info` and `/success` routes should show masked or summarised data.
