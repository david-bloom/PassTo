## Product Goal

Give the nurse a launch-critical post-enrollment dashboard that shows credential, license, wallet, and subscription state clearly and exposes only approved MVP actions.

## Technical Scope

- Implement or revise the nurse dashboard status surface.
- Read credential, license, wallet, subscription, and entitlement state from backend-owned status endpoints or approved user-scoped reads.
- Show launch-critical states:
  - credential pending, active, suspended, revoked, expired, or blocked
  - license status and current-as-of timestamp
  - wallet issued, pending, failed, or not attempted
  - subscription/tier state and lapse state
  - share-link availability or disabled reason
- Route incomplete or blocked nurses to the correct recovery/status surface.
- Hide or clearly defer Show QR, PDF export, additional-license, and employer-dashboard controls.

## Out of Scope

- Share-link token creation implementation.
- Verifier `/v/{token}` flow.
- Show QR.
- PDF export.
- Additional license purchase/use.
- Employer dashboard.
- Lovable admin UI.

## Security/RLS Impact

Security Impact: Medium

Dashboard reads sensitive nurse-owned state. Lovable must not write trust, credential, wallet, payment, or verifier truth directly.

## Design System Impact

Design System Impact: Medium  
Design Source: `/docs/design_system/README.md`

The dashboard is an operational status surface. It should be concise, scannable, and avoid marketing-style layout.

## Acceptance Criteria

- [ ] Dashboard shows credential, license, wallet, and subscription state from backend-owned data.
- [ ] Dashboard does not mark credential, wallet, payment, or share-link status complete from client-only state.
- [ ] Share-link action is shown only when backend state says it is eligible or clearly disabled with a safe reason.
- [ ] Deferred Show QR, PDF export, additional-license, and employer-dashboard actions are hidden or marked deferred without blocking launch.
- [ ] Direct navigation to `/dashboard` before prerequisites routes safely.
- [ ] Lovable cannot directly mutate trust/payment/credential/wallet/verifier tables from dashboard UI.
- [ ] Claude documents files changed, tests run, deviations, and risks.

## Claude Implementation Notes

**Date:** 2026-06-02

**Deliverable:** `dashboard-status` Edge Function + Lovable prompt. No schema migrations required — reads existing tables only.

**Files created:**

| File | Purpose |
|---|---|
| `supabase/functions/dashboard-status/index.ts` | New GET status endpoint for `/dashboard` |
| `docs/tasks/LOVABLE_PROMPT_2026-06-02_TASK0055_DASHBOARD.md` | Lovable implementation instructions |

**Backend call from `/dashboard`:**

```
GET /functions/v1/dashboard-status
Authorization: Bearer <supabase session token>
```

Returns: `onboarding_step`, `account_status`, `subscription_tier`, `subscription_status`, `subscription_plan_name`, `license_type`, `license_state`, `license_normalized_status`, `license_status_intent`, `license_expiration_date`, `license_current_as_of`, `license_data_match_passed`, `credential_status`, `credential_issued_at`, `credential_expires_at`, `wallet.apple`, `wallet.google`, `wallet.any_issued`, `share_link_eligible`, `share_link_reason`, `can_add_license`.

**Tables read (read-only):**

- `profiles` — `onboarding_step`, `account_status`, `subscription_tier`
- `subscriptions` — `plan_name`, `status`, `license_entitlement_count` (active row only)
- `licenses` — `license_type`, `state`, `normalized_status`, `status_intent`, `expiration_date`, `status_checked_at`, `data_match_passed` (primary, most recent)
- `credentials` — `status`, `issued_at`, `expires_at` (most recent)
- `wallet_passes` — `provider`, `status`, `pass_url`, `external_pass_id` (by credential_id)

**Gate:** `onboarding_step` must be in `["pass", "complete"]`. All other steps return `403 onboarding_not_complete` with the current step for Lovable redirect.

**Share-link:** Returns `share_link_eligible: false, share_link_reason: "not_implemented"`. Share-link token creation is TASK-0056.

**Deviations from spec:**

- `payments` table not read — all needed state is in `subscriptions` and `profiles`. Reading `payments` directly would not add safe display value.
- `wallet_pass_treatment` not exposed in dashboard payload — `license_normalized_status` and `license_status_intent` are sufficient for status display.

**Tests run:** `dashboard-status` v1 deployed to Supabase (project wvzjfxacykgsaffskgtr). Manually testable with a valid nurse session at onboarding_step pass/complete. Codex QA required before production use.

**Risks / open items:**

- `subscriptions` table columns assumed from prior function reads (`plan_name`, `status`, `license_entitlement_count`). Codex should verify column names match the applied migration.
- Lovable must not call any write endpoint from `/dashboard` — the prompt explicitly prohibits this.

**Design system compliance:** Lovable prompt specifies all PassTo color tokens, fonts, and layout conventions per `/docs/design_system/README.md`.
