## Product Goal

Validate verifier share-link tokens in a backend-controlled flow and return only safe credential display data after the verifier submits required information and accepts Terms.

## Technical Scope

- Implement or revise a backend verifier token validation operation.
- Accept raw token from `/v/{token}` route.
- Hash token server-side and find matching `verification_tokens` row.
- Reject missing, malformed, expired, used, revoked, wrong-type, or non-shareable tokens.
- Require verifier name, verifier email, and Terms acceptance before returning credential data.
- Create or associate a `verifiers` row for the token.
- Write `verification_events` for form submitted, credential viewed, and failure paths.
- Mark token used after successful verified access.
- Return safe credential/license/current-as-of display data only.

## Acceptance Criteria

- [ ] Valid unused `share_link` token validates only before expiry and before first successful use.
- [ ] Expired, used, revoked, malformed, unknown, wrong-type, or blocked tokens return safe failure payloads.
- [ ] Verifier name, verifier email, and Terms acceptance are required before credential data is returned.
- [ ] Successful verification creates/verifies a `verifiers` row and writes `verification_events`.
- [ ] Successful verification marks token used or otherwise prevents second successful use.
- [ ] Returned credential data excludes private nurse/internal data and raw provider payloads.
- [ ] Verifier cannot browse records outside the tokenized backend flow.
- [ ] Claude documents files changed, tests run, deviations, and risks.
