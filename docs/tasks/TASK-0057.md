## Product Goal

Validate verifier share-link tokens in a backend-controlled flow and return only safe credential display data after the verifier submits required information and accepts Terms.

## Technical Scope

- Implement or revise a backend verifier token validation operation.
- Accept raw token from `/v/{token}` route.
- Hash token server-side and find matching `verification_tokens` row.
- Reject missing, malformed, expired, used, revoked, wrong-type, or non-shareable tokens.
- Require verifier name, verifier email, and Terms acceptance before returning credential data.
- Create or associate a `verifiers` row for the token.
- Write `verification_events` for form submitted, credential viewed, and failure paths.
- Mark token used after successful verified access.
- Return safe credential/license/current-as-of display data only.

## Claude Implementation Notes

**Date:** 2026-06-02

**Deliverable:** `token-verify` Edge Function (new) — anonymous POST, token-gated verifier validation.

**Files created:**

| File | Purpose |
|---|---|
| `supabase/functions/token-verify/index.ts` | Anonymous POST — validates share_link token, returns safe credential display data |

**Flow:**
1. Parse + validate body: `token`, `verifier_name`, `verifier_email`, `terms_accepted`
2. SHA-256 hash raw token server-side; look up `verification_tokens` by hash
3. Reject: not found, wrong type, used, revoked, not active, expired
4. Load credential (must still be `active`) + license (must still be `Active`) at time of verification
5. Atomically mark token `used` — `UPDATE WHERE status='active'`; 0 rows → race condition, return `token_used`
6. Insert `verifiers` row: `token_id`, `name`, `email`, `terms_accepted_at`
7. Write `verification_events`: `session_started` + `credential_viewed`
8. Write `audit_events`: `verification.completed` (best-effort, non-fatal)
9. Return safe display payload

**Tables written:**
- `verification_tokens` — status → `used`, `used_at`
- `verifiers` — `token_id`, `name`, `email`, `terms_accepted_at`, `status`
- `verification_events` — `session_started`, `credential_viewed`
- `audit_events` — `verification.completed` (best-effort)

**Safe display payload returned:**
- `credential.status`, `credential.issued_at`, `credential.expires_at`
- `license.type`, `license.state`, `license.normalized_status`, `license.expiration_date`, `license.current_as_of`
- `token_expires_at`, `verified_at`, `verifier_id`

**Excluded from payload (never returned):**
- Nurse name, email, phone, DOB
- Raw license number
- Internal profile/credential/license IDs
- Raw provider API payloads (`lookup_response`)
- Subscription/payment details

**CORS:** `*` — public anonymous endpoint. Token hash is the auth mechanism. No credential cookies sent.

**Security notes:**
- Double-use prevention: atomic UPDATE WHERE status='active' (single winner in any race)
- Credential + license status re-checked at redemption time, not just at token creation time
- All DB ops via service-role only — verifier cannot browse tables directly

**Deviations from spec:**
- `wallet_passes` not read — safe display payload adequately covers credential/license state without wallet status; wallet status is not relevant to verifier identity check.
- `verifier_id` included in response — needed by TASK-0058 Lovable flow for follow-on session_completed event if implemented.

**Tests run:** Deployed to Supabase (project wvzjfxacykgsaffskgtr). Source in GitHub before deployment. End-to-end test (token creation → validation chain) requires TASK-0056 Codex QA to pass first (issue #11).

**Open items:**

- `session_completed` event not written — TASK-0058 Lovable flow can write it via a separate endpoint if needed.
- First-use enforcement is the atomic mark-used in step 5; additional TTL enforcement handled by expires_at check in step 3.
