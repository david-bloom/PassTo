### ID.me OAuth — Architecture Violation

```
redirect_uri = https://enroll.passtodigital.com/auth/idme/callback
```

Hardcoded in `src/pages/IdVerification.tsx:7`. The OAuth callback is routed to the Lovable frontend (`IdmeCallback.tsx`), not the Vercel `api/idme-callback.js` endpoint.

**Violation:** Per DECISION-0013, Lovable must not perform credential/identity trust decisions. The ID.me code exchange belongs in a server-side Supabase Edge Function or Vercel route. The Vercel `api/idme-callback.js` exists for this purpose but appears bypassed.

**Risk:** If `IdmeCallback.tsx` is exchanging the authorization code for tokens client-side, the ID.me `client_secret` may be exposed in browser code or the exchange may be happening insecurely.

### Assessment

P2 is the most critical project from a migration standpoint. Make.com is still the active backend for the enrollment pipeline. ID.me callback is handled in Lovable in violation of platform ownership rules. `create-airtable-record` edge function is dead but still deployed. Until Make is replaced and ID.me callback is moved server-side, this project cannot be considered migrated.

## CRITICAL FINDINGS

### CF-1 — Three Separate Supabase Instances

| Project | Supabase Project ID Prefix |
|---|---|
| P1 PassTo Website | `zektkbhvmbbmhvthwah` |
| P2 PassTo Enroll | `ofpxczstptysqxoruiox` |
| P3 PassTo App | `vvefeasvpmdsqvwkkzsj` |

**Violation:** DECISION-0005 / DECISION-0013 establish Supabase as the single system of record. Three isolated databases mean enrollment data (P2) cannot flow to the dashboard (P3), and neither talks to the website (P1). There is no single source of truth.

**Required action:** David must confirm which Supabase project is the intended canonical project before migration SQL can be targeted. All three projects must eventually point to the same Supabase instance.

### CF-2 — Make.com Still Active in Production (P2)

Make webhook `https://hook.us2.make.com/6c1ylx67nwua8ksk3segplftob5iu9m8` is hardcoded and called live from P2's enrollment flow. DECISION-0012 states Make is abandoned but this project has not been migrated. The `submit-enrollment` edge function is a partial migration — it calls Supabase AND Make, meaning the enrollment pipeline is split across both platforms.

**Required action:** Make webhook calls must be removed from `IdmeCallback.tsx` (browser) and `submit-enrollment` (edge function) before this project is considered migrated.

### CF-3 — ID.me OAuth Callback in Lovable Frontend (P2)

`/auth/idme/callback` is a Lovable frontend route handled by `IdmeCallback.tsx`. The `redirect_uri` in `IdVerification.tsx` points to `enroll.passtodigital.com/auth/idme/callback`, bypassing the Vercel `api/idme-callback.js`.

**Violation:** Per DECISION-0013, Lovable must not perform credential or identity trust decisions. ID.me code exchange must be server-side (Supabase Edge Function or Vercel route).

**Required action:** Clarify whether `IdmeCallback.tsx` is doing the code exchange in the browser or deferring to a backend. If client-side, must be moved server-side before launch. The existing Vercel `api/idme-callback.js` or a new Supabase Edge Function should receive the redirect.

### CF-4 — Dead Airtable Edge Functions Still Deployed (P1, P2)

| Project | Function | Status |
|---|---|---|
| P1 | `sync-airtable` | Deployed and called from 2 files |
| P2 | `create-airtable-record` | Deployed and called from 1 file |

These call Airtable APIs that no longer exist as part of the approved stack. They should be removed from both the frontend call sites and the deployed edge functions.

### CF-5 — Wallet Pass Signing Not Yet Wired (All Projects)

The Vercel routes `api/sign-apple.js` and `api/sign-google.js` are not referenced in any of the three Lovable projects. Wallet pass generation has not been implemented in the current Lovable codebase. The `/pass-ready` route exists in P1 and P2 but does not call the signing endpoints.

**Implication:** Wallet credential issuance is a core MVP requirement (PRD Section 1) but is not yet wired into any Lovable project.
