## Proposed CORS Policy Per Function

### `dashboard-status`

Called by the App project's `/dashboard` only.

**Recommended:** lock to App domain.

```ts
const corsHeaders = {
  "Access-Control-Allow-Origin": "https://app.passtodigital.com",
  "Access-Control-Allow-Headers":
    "authorization, x-client-info, apikey, content-type",
  "Access-Control-Allow-Methods": "GET, OPTIONS",
};
```

### `share-link-create`

Called by the App project's share-link UI only.

**Recommended:** lock to App domain.

```ts
const corsHeaders = {
  "Access-Control-Allow-Origin": "https://app.passtodigital.com",
  "Access-Control-Allow-Headers":
    "authorization, x-client-info, apikey, content-type",
  "Access-Control-Allow-Methods": "POST, OPTIONS",
};
```

### `token-verify`

Called by the App project's `/v/:token` verifier surface — which is **public-facing** and may be opened by employers, schools, recruiters, or hospital staff from any device or context (work email, personal browser, mobile).

**Decision required from David — Option A vs Option B:**

| Option | Header value | Tradeoff |
|---|---|---|
| **A — lock to App domain** | `Access-Control-Allow-Origin: https://app.passtodigital.com` | Forces verifiers to view the verifier UI on `app.passtodigital.com` in the same browser session that makes the API call. Closes off any third-party embedding or wrapped page. Matches the assumption that `/v/:token` is the only legitimate consumer. |
| **B — keep `*` (current)** | `Access-Control-Allow-Origin: *` | Continues to allow any origin to call the endpoint. Required if PassTo intends third parties (employer dashboards, integrations) to call `token-verify` directly. Tokens are still gated by hash + first-use, so authorization is not loosened, only origin. |

**Claude's recommendation:** Option A (lock to App domain). The current MVP scope is the in-browser `/v/:token` surface on `app.passtodigital.com`. Public-API access for third parties is out of MVP scope, so Option A reduces attack surface without losing required functionality. Option B should be revisited only when a documented integration partner needs direct cross-origin access.

## Security / RLS Impact

Security Impact: Medium

CORS does not authorize requests — it only declares which web origins the browser will allow to read responses. Server-side authorization (JWT, hash, RLS, fail-closed gates) remains the security boundary. Tightening CORS reduces *browser-mediated* cross-origin abuse vectors, including CSRF-style misuse by malicious sites that try to read responses on behalf of an authenticated user. Loosening CORS does not bypass server-side gates but does enable any web page to script calls in a logged-in browser.
