## Product Goal

Add the launch-critical phone possession gate so credential issuance remains blocked until the nurse completes Twilio SMS verification.

## Technical Scope

- Implement backend send-code and verify-code operations using Twilio.
- Wire Lovable phone entry and code entry UI to the backend operations.
- Persist phone verification state on the nurse profile or approved verification state table.
- Write audit/event records for send, verify, failure, retry, and lockout behavior where supported.
- Provide clear retry/failure routing.

## Out of Scope

- Twilio A2P 10DLC production approval.
- License lookup, data matching, selfie, credential, wallet, or Stripe implementation.
- Production launch fallback approval.

## Acceptance Criteria

- [ ] Authenticated nurse can request an SMS verification code.
- [ ] Authenticated nurse can submit the code for verification.
- [ ] Phone verification success advances to license lookup.
- [ ] Failed/expired/too-many-attempts states do not advance the journey.
- [ ] Credential issuance remains blocked until phone verification passes.
- [ ] Claude documents tests, files changed, deviations, and risks.

## QA Plan

- Test send-code success and provider failure behavior.
- Test correct code, incorrect code, expired code, and retry path.
- Verify no Twilio secrets are exposed to Lovable.

### Findings

#### P2 — Twilio send failures do not write `notification_events` failure rows

TASK-0026 requires Twilio send errors to write `notification_events` with `status = 'failed'` before returning `{ sent: false, error: 'provider_error' }`.

Current `phone-send-otp` only writes `audit_events` on Twilio 429/provider errors and returns before any `notification_events` row is inserted.

**Impact:** Ops visibility misses failed outbound phone verification attempts, which weakens the Phase 3.2 failure-state and admin/ops trail.

**Required remediation:**

- On Twilio send 429, insert `notification_events` with:
  - `profile_id`
  - `related_entity_type = 'phone_verification'`
  - `channel = 'sms'`
  - `provider = 'twilio'`
  - `direction = 'outbound'`
  - `status = 'failed'`
  - `error_message` or equivalent safe non-secret provider summary
- On other Twilio send failures or thrown send exceptions, insert the same failed notification row.
- Do not include the phone number in notification/audit logs.

#### P2 — `phone-verify-otp` collapses Twilio terminal statuses into `invalid_code`

Current `phone-verify-otp` treats only `approved` and `canceled` specially. Any other non-approved status falls through to `invalid_code`.

Twilio's Verification Check API documents additional possible statuses including `expired`, `max_attempts_reached`, `deleted`, and `failed`, besides `pending`, `approved`, and `canceled`.

**Impact:** Expired/max-attempt states may be shown to the nurse as a simple incorrect code, which conflicts with TASK-0026's expired/max-attempt retry/resend behavior and can trap the user in the wrong UX path.

**Required remediation:**

- Map `expired`, `max_attempts_reached`, and `canceled` to `{ verified: false, error: 'code_expired' }` and write `phone.otp_expired`.
- Map `pending` to `{ verified: false, error: 'invalid_code' }` and write `phone.otp_failed`.
- Map `failed`, `deleted`, missing status, or any unexpected status to `{ verified: false, error: 'provider_error' }` and write `phone.otp_failed`.
- Keep `approved` as the only status that calls `complete_phone_verification()`.

#### P3 — Supabase insert error handling logs only thrown exceptions, not returned errors

Both functions use `try/catch` around Supabase `.insert()` calls, but Supabase client writes generally return `{ error }` rather than throwing. As written, `notification_events` and helper `writeAudit()` can silently fail without logging the returned error.

**Impact:** This does not create a credential bypass, but it makes QA and ops visibility weaker if an event/audit insert fails due to schema, constraint, or permission issues.

**Required remediation:**

- Capture `{ error }` from `notification_events` inserts and `audit_events` inserts.
- Log returned errors with enough context to debug, without logging phone numbers or secrets.
- Keep failure-path audit writes non-blocking except where the approved atomic RPC already handles `phone.verified`.
