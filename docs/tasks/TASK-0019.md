## 1. Bucket Name

**Recommended bucket name:** `selfies`

This is already named in the approved P2 enrollment pipeline spec (`docs/architecture/P2_ENROLLMENT_PIPELINE_MIGRATION_SPEC.md`, Step 6):

```js
supabase.storage.from("selfies").upload(...)
```

Using this name keeps the implementation consistent with the existing spec. It is descriptive, short, and unambiguous.

---

## 2. Public / Private Decision

**Decision: Private bucket.**

Selfies are biometric PII. A public bucket would expose biometric images to any caller who can guess or obtain a storage path — an unacceptable security failure for a trust/credentialing product.

Rationale by requirement:
- PRD Section 5.9: "Selfie storage must not be public/unrestricted."
- PRD Section 4.7: Lovable may read/write only records that RLS explicitly allows for the authenticated user.
- PRD Section 6.4: Service-role key must never be exposed to the client; selfie access follows the same boundary.
- Security Model: Biometric assets must not be broadly accessible. PassTo owns this selfie; no third party (including ID.me) owns it.

Supabase Storage private bucket behavior: objects are only accessible by callers whose JWT satisfies a matching storage policy, or by service-role callers who bypass RLS entirely.

## 3. File Path / Key Structure

**Recommended path:** `{auth_user_id}/selfie.jpg`

Where `auth_user_id` is `auth.uid()` — the Supabase Auth user UUID, equivalent to `profiles.auth_user_id`.

Examples:
```
a1b2c3d4-e5f6-7890-abcd-ef1234567890/selfie.jpg
```

**Rationale for using `auth.uid()` as the prefix (not `profiles.id`):**
Supabase Storage RLS policies have direct access to `auth.uid()` in the policy expression without a database join. Using `profiles.id` (a different UUID) would require a subquery against `public.profiles` in every policy expression — slower and more complex. Using `auth.uid()` keeps policies simple, fast, and consistent with how Lovable JS code accesses the user identity.

**Rationale for single file per nurse (overwrite on retry):**
The pipeline spec uses `selfie.jpg` as a fixed filename per user. On retry, Lovable calls `.upload()` with `upsert: true`, overwriting the previous attempt. This:
- Keeps storage clean (no accumulating old selfies)
- Reduces privacy risk (only the current selfie is retained)
- Simplifies the path model
- Is consistent with MVP scope (one credential, one primary license, one selfie)

**Rationale for `.jpg` extension:**
Mobile selfie cameras produce JPEG output by default. MIME type validation is enforced separately (see Section 9 — Security Risks). The `.jpg` filename is a convention, not a MIME enforcement mechanism.

**Alternative path not used: `{auth_user_id}/{timestamp}-selfie.jpg`**
Versioned paths accumulate old selfies, creating ongoing PII storage growth and no clear purge process. For MVP, overwrite-on-retry is cleaner and sufficient.

## 4. Upload Ownership: Lovable Direct Upload

**Decision: Lovable direct upload using authenticated user JWT, followed by a backend Edge Function confirmation step.**

This is the two-step pattern already specified in `P2_ENROLLMENT_PIPELINE_MIGRATION_SPEC.md`:

```js
// Step A — Lovable uploads directly to Storage (user JWT authorizes via RLS)
const { error: uploadError } = await supabase.storage
  .from("selfies")
  .upload(`${user.id}/selfie.jpg`, file, { upsert: true });

if (uploadError) { /* show retry UI */ return; }

// Step B — Backend Edge Function confirms upload and updates profile state
const { error: enrollError } = await supabase.functions.invoke("submit-enrollment", {
  body: { step: "selfie", selfie_storage_path: `${user.id}/selfie.jpg` }
});
```

**Why direct upload (not backend-mediated):**
- Supabase Storage is designed for authenticated client uploads; it is a core product feature, not a workaround.
- Backend-mediated upload (nurse sends file to an Edge Function, which forwards to Storage using service role) doubles the transfer cost, doubles the latency, and requires the Edge Function to handle raw binary payloads — a pattern Supabase Edge Functions handle less cleanly than direct storage SDK calls.
- RLS gives the same security boundary as a backend proxy would, with less complexity.
- PRD Section 4.7 explicitly permits Lovable to read/write records that RLS allows for the current authenticated user. This is exactly that case.

**Why the backend confirmation step is required:**
The Lovable upload alone is insufficient. After the file is in Storage, a backend Edge Function must:
1. Verify the JWT and confirm `auth.uid()` matches the path prefix.
2. Confirm the file actually exists at the declared path in Storage (prevent a nurse from claiming `selfie_storage_path` without uploading).
3. Validate the file size and MIME type server-side (belt-and-suspenders beyond client-side validation).
4. Update `profiles.selfie_storage_path` and `profiles.selfie_captured_at` using service role (these fields are not nurse-writable via RLS or the `update_own_profile_basic()` RPC).
5. Advance `profiles.onboarding_step` from `'license'` to `'pass'` as appropriate.
6. Write an `audit_events` record: `action = 'selfie.captured'`.

The nurse client must **not** update `profiles.selfie_storage_path` directly — that would allow a nurse to claim a selfie was captured without the file actually being validated by the backend.

## 5. Read / Access Rules

| Caller | Read Access | Write Access | Delete Access | How |
|---|---|---|---|---|
| Authenticated nurse (own selfie) | ✅ SELECT own path only | ✅ INSERT + UPDATE own path only | ❌ No policy (service-role only) | Storage RLS, user JWT |
| Authenticated nurse (another nurse's selfie) | ❌ RLS blocks | ❌ RLS blocks | ❌ RLS blocks | Storage RLS |
| Anonymous / unauthenticated | ❌ No policy | ❌ No policy | ❌ No policy | Private bucket |
| Backend Edge Functions (service role) | ✅ All objects | ✅ All objects | ✅ All objects | Service-role bypasses RLS |
| Ops (Supabase dashboard) | ✅ All objects | ✅ All objects | ✅ All objects | Supabase dashboard runs as service role |
| Verifier | ❌ No access | ❌ No access | ❌ No access | Verifiers are anonymous users; private bucket blocks |
| Vercel routes | ✅ If using service-role key in Vercel env | N/A | N/A | Server-to-server only; never from browser |

**No signed URL pattern for selfies in MVP:**
Generating presigned/signed URLs for selfies and serving them to clients (e.g., to display the selfie on the nurse's profile) is deferred. For MVP, selfie access from Lovable is limited to the upload flow. If a display case is needed later, short-TTL signed URLs (max 60 seconds) generated by a backend Edge Function would be the approved pattern — never direct public URLs or long-lived signed URLs.

## 6. Required Storage Policies

These are the four Storage RLS policies to create on the `storage.objects` table for the `selfies` bucket:

```sql
-- Policy 1: Nurse may upload (INSERT) their own selfie
-- Exact-path match enforces the single-object-per-nurse MVP model
create policy "nurse_upload_own_selfie"
  on storage.objects
  for insert
  to authenticated
  with check (
    bucket_id = 'selfies'
    and name = auth.uid()::text || '/selfie.jpg'
  );

-- Policy 2: Nurse may overwrite (UPDATE) their own selfie for retry
-- Both USING and WITH CHECK required for UPDATE; exact-path match
create policy "nurse_update_own_selfie"
  on storage.objects
  for update
  to authenticated
  using (
    bucket_id = 'selfies'
    and name = auth.uid()::text || '/selfie.jpg'
  )
  with check (
    bucket_id = 'selfies'
    and name = auth.uid()::text || '/selfie.jpg'
  );

-- Policy 3: Nurse may read their own selfie
-- Exact-path match; limits read to the single canonical object
create policy "nurse_select_own_selfie"
  on storage.objects
  for select
  to authenticated
  using (
    bucket_id = 'selfies'
    and name = auth.uid()::text || '/selfie.jpg'
  );

-- No DELETE policy for authenticated users.
-- Selfie deletion only via service-role (ops purge or support process).
-- Prevents nurses from removing audit-relevant biometric records.
```

**Storage bucket configuration (applied via Supabase dashboard or API when creating the bucket):**

```
Bucket name:         selfies
Public:              false
File size limit:     10 MB
Allowed MIME types:  image/jpeg, image/png, image/webp
```

**Why 10 MB:** Typical mobile camera selfie is 2–5 MB. 10 MB provides headroom for high-resolution devices without being unrestricted. Files above 10 MB are likely not genuine selfie photos.

**Why `image/jpeg, image/png, image/webp`:** These are the three dominant formats for mobile selfies and camera captures across iOS and Android. HEIC/HEIF is excluded because Supabase Storage may not serve it reliably and most browsers cannot display it without conversion.

## 7. Required Database Fields

The v4 migration schema (`docs/architecture/V4_MIGRATION_SQL.md`, base migration) does not include selfie fields on `profiles`. Two new fields are required before Phase 3.5 (selfie capture) can be built:

```sql
-- Required schema addition (v4_passto_mvp_selfie_fields migration)
-- Apply as part of TASK-0020 before Phase 3.5 implementation

alter table public.profiles
  add column if not exists selfie_storage_path  text,
  add column if not exists selfie_captured_at   timestamptz;

comment on column public.profiles.selfie_storage_path
  is 'Supabase Storage path of the nurse selfie within the selfies bucket. Format: {auth_user_id}/selfie.jpg. Service-role only. Set by submit-enrollment Edge Function after upload confirmation, not by nurse client directly.';

comment on column public.profiles.selfie_captured_at
  is 'Timestamp when the selfie was successfully captured and confirmed by backend. Set by service-role. Null until selfie step is confirmed.';
```

**Access rules for these fields:**
- `selfie_storage_path`: service-role write only (Edge Function). Nurse SELECT via the existing `nurse_select_own_profile` RLS policy (they can read it, which lets Lovable know whether a selfie is on file).
- `selfie_captured_at`: service-role write only. Nurse SELECT same.
- Neither field is included in `update_own_profile_basic()` — these are trust/gate fields, not self-service fields.

**No separate `selfies` database table needed for MVP:**
A one-per-nurse Storage path reference on `profiles` is sufficient for the MVP single-selfie model. If multiple credential attempts or selfie refresh become a requirement post-MVP, a `selfie_records` table can be added at that time.

**Audit trail:**
Selfie capture is audited via `audit_events` with `action = 'selfie.captured'`, not via a dedicated database table. The Storage object itself serves as the primary record.

## 8. Failure States and Retry / Support Behavior

| Failure | User-Visible Behavior | Backend Behavior | Recovery |
|---|---|---|---|
| Network failure during upload | Lovable shows upload error + retry button | Upload not received; no state change | Nurse retries; upsert overwrites partial upload |
| File too large (> 10 MB) | Client-side validation shows error before upload | Upload rejected by Storage; `selfie_storage_path` unchanged | Nurse takes a new photo at lower quality |
| Invalid MIME type | Client-side validation shows error before upload | Upload rejected by Storage MIME policy | Nurse uses a different camera app or format |
| Upload succeeds but Edge Function call fails | Lovable shows "Something went wrong" + retry prompt | File exists in Storage; `selfie_storage_path` is NOT yet updated | Nurse retries enrollment step; Edge Function re-validates the already-uploaded file and updates profile |
| Edge Function cannot find the file at the declared path | Edge Function returns error | `selfie_storage_path` not updated; audit event written | Nurse is asked to retry the selfie capture |
| Edge Function detects path prefix mismatch (wrong user's path) | Edge Function returns error | `selfie_storage_path` not updated; security audit event written | Support investigation required |
| Supabase Storage unavailable | Lovable shows service unavailable + retry | Edge Function confirms Storage unavailability; writes event | Nurse retries when Storage recovers; ops alert for extended outage |
| Duplicate upload (same path, nurse restarts enrollment) | Upload overwrites previous selfie silently | `upsert: true` replaces prior file; Edge Function updates `selfie_captured_at` | No action needed; the latest selfie is canonical |

**Credential issuance gate:**
Credential creation must remain blocked if `profiles.selfie_storage_path IS NULL` or if the Edge Function has not confirmed the upload. The backend credential-issue function must check this field before proceeding.

**Ops visibility:**
`profiles.selfie_storage_path` and `profiles.selfie_captured_at` are visible via the Supabase dashboard. Ops can identify whether a nurse's selfie step is complete or stuck.

## 9. Security Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| **Nurse guesses another nurse's storage path** (paths are `{uuid}/selfie.jpg` — UUIDs are non-guessable but theoretically knowable) | Medium | RLS Policy 3 (`nurse_select_own_selfie`) ensures a nurse can only `SELECT` objects where the path prefix matches their own `auth.uid()`. Knowing the UUID is not enough — the JWT must match. |
| **Nurse sets `selfie_storage_path` to an arbitrary value via direct API** | High | `profiles.selfie_storage_path` has no nurse UPDATE policy. Only `update_own_profile_basic()` RPC is callable, and this field is not in that RPC. Any client-side attempt to SET this column will be blocked by Supabase RLS (no UPDATE policy for authenticated role on `profiles`). |
| **Nurse uploads a non-selfie file (document, malware, etc.)** | Medium | MIME type restriction at the bucket level blocks non-image uploads. Client-side validation provides UX clarity before the attempt. Server-side: Edge Function should validate the file exists and is an image type before confirming in `profiles`. |
| **service_role key exposure** | Critical | If `service_role` is ever leaked, all selfies are readable. This key must never appear in Lovable code, client-side env vars, or browser contexts. Stored in Supabase Edge Function secrets only. |
| **Long-lived signed URLs for selfies are created and shared** | High | No signed URL generation is approved for MVP selfies. If a signed URL is ever needed (ops support), it must be generated server-side with a maximum TTL of 60 seconds. Never generate permanent or long-lived signed URLs for biometric assets. |
| **Selfie retained beyond the 7-year retention window** | Low (future) | `profiles.deleted_at` soft-delete is the account closure mechanism. A future purge process must also delete the Storage object at the nurse's path. This is out of MVP scope but must be documented before production launch. |
| **Upload BEFORE data matching completes (nurse bypasses the gate)** | Medium | The `selfies` bucket INSERT policy allows an authenticated nurse to upload at any time — Storage RLS cannot enforce "data match must be complete." This gate must be enforced by the backend: the `submit-enrollment` Edge Function for the selfie step must verify that `profiles.onboarding_step = 'license'` (or equivalent — data match passed) before confirming the selfie. A premature upload without backend confirmation does not advance the nurse's state. |
| **Cross-tenant path collision if bucket is ever shared across projects** | Low | Not a concern for MVP — one Supabase project, one bucket, one auth space. Document clearly that `selfies` bucket belongs to `wvzjfxacykgsaffskgtr` (PassTo Dev) only. |

**Verdict:** Approved with Required Changes — P2 policy fix incorporated; TASK-0020 unblocked pending David approval.

### P2 — Storage policies allowed any file under nurse's folder (FIXED above)

The original folder-prefix check `(storage.foldername(name))[1] = auth.uid()::text` allowed a nurse to upload objects at any path under their UUID prefix (e.g., `{uuid}/anything.jpg`, `{uuid}/nested/file.webp`). This did not break cross-user isolation but weakened the single-current-selfie MVP model.

**Fix applied:** All three Storage RLS policies now use exact-path matching: `name = auth.uid()::text || '/selfie.jpg'`. A nurse JWT can only INSERT, UPDATE, or SELECT the single canonical object at `{auth_user_id}/selfie.jpg`.

### P3 — HEIC/HEIF exclusion acceptable; Phase 3.5 must handle iOS UX

The MIME allowlist (`image/jpeg, image/png, image/webp`) is correct for MVP. HEIC/HEIF exclusion is reasonable given uneven browser support. **Phase 3.5 implementation requirement:** the Lovable capture flow must either convert to JPEG client-side before upload, or display a clear, actionable retry message if the device provides an unsupported format. Add to Phase 3.5 acceptance criteria.

### P3 — Pipeline spec has stale language re: `profiles.id = auth.uid()`

`docs/architecture/P2_ENROLLMENT_PIPELINE_MIGRATION_SPEC.md` contains stale language implying `profiles.id = auth.uid()`. The live v4 schema has `profiles.id` as its own UUID and `profiles.auth_user_id` referencing `auth.users(id)`. TASK-0019 correctly uses `{auth_user_id}/selfie.jpg`. **Edge Function implementation requirement:** the `submit-enrollment` confirmation step must look up the nurse profile using `profiles.auth_user_id = auth.uid()`, not assume `profiles.id = auth.uid()`.
