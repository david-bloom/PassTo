## Product Goal

Let a verifier open a nurse-sent share link, submit required minimal information, accept Terms, and view safe credential status without a PassTo account.

## Technical Scope

- Implement or revise Lovable `/v/{token}` route behavior.
- Show safe loading, invalid, expired, used, revoked, blocked, and success states.
- Collect only verifier name, verifier email, and Terms acceptance for MVP.
- Call the approved backend token validation function.
- Display safe credential and license status fields returned by backend.
- Avoid exposing raw source payloads, internal identifiers, DOB, phone, nurse email, payment/subscription details, or audit internals.
- Keep Show QR, PDF export, employer dashboard, and verifier account creation out of scope.

## Out of Scope

- Backend token creation.
- Backend token validation implementation.
- Verifier account creation.
- Employer dashboard.
- Show QR.
- PDF export.
- Legal drafting of final Terms text unless David provides approved copy.

## Routes Affected

- `/v/{token}`

## Components Affected

- Lovable verifier route/page.
- Verifier form component.
- Safe credential status display component.

## Database/Tables Affected

- None directly from Lovable. Backend function owns writes to `verifiers` and `verification_events`.

## Integrations Affected

- Supabase Edge Functions or approved backend route for token validation.

## Security/RLS Impact

Security Impact: Medium

The verifier route is public/tokenized. It must call backend validation and must not use Supabase direct table reads for verifier access.

## Design System Impact

Design System Impact: Medium  
Design Source: `/docs/design_system/README.md`

The verifier page should feel clear, trustworthy, and operational. It should not imply employment clearance, background-check status, or broader endorsement beyond the returned credential facts.

## Acceptance Criteria

- [ ] `/v/{token}` loads without requiring a PassTo account.
- [ ] Verifier must submit name, email, and Terms acceptance before credential facts are displayed.
- [ ] Invalid, expired, used, revoked, blocked, and unknown token states display safely without leaking token or nurse data.
- [ ] Successful state displays only backend-returned safe credential/license facts and current-as-of status.
- [ ] UI does not show Show QR, PDF export, employer dashboard, or verifier account controls.
- [ ] Lovable does not directly read or write `verification_tokens`, `verifiers`, `verification_events`, `credentials`, or `licenses`.
- [ ] Claude/Lovable documents files changed, tests run, deviations, and risks.

**Backend call from `/v/:token`:**

```
POST /functions/v1/token-verify
Content-Type: application/json
(no Authorization header)

{
  "token": "<from URL>",
  "verifier_name": "...",
  "verifier_email": "...",
  "terms_accepted": true,
  "marketing_consent": false
}
```

**All response states handled in Lovable prompt:**

| State | Trigger |
|---|---|
| Verified | `verified: true` (200) |
| Token expired | `token_expired` (403) |
| Token used | `token_used` (403) |
| Token not valid | `token_not_found` / `token_wrong_type` / `token_revoked` / `token_not_active` |
| Credential inactive | `credential_not_shareable` (403) |
| License changed | `license_not_active` (403) |
| Form validation errors | 400 with field-specific codes |
| Backend error | `backend_read_error` / 503 / network |

**Safe display payload shown to verifier:**
- `license.type`, `license.state`, `license.normalized_status`, `license.expiration_date`, `license.current_as_of`
- `credential.status`, `credential.issued_at`
- Disclaimer with `license.current_as_of` timestamp

**Deviations from VERIFIER_CREDENTIAL_VIEW.md:**
- Nurse full name excluded — not returned by `token-verify` (intentional; nurse name is private data per task spec). Flow doc predates the implemented backend contract.
- Raw license number excluded — same reason.
- No pre-flight token check before form display — `token-verify` is a single POST (validate + collect + mark used). Invalid/expired/used errors surface on form submission.
- Refresh after success shows form again → resubmission returns `token_used` — documented in prompt.
