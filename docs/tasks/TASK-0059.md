## Security/RLS Impact

Security Impact: High

Codex must explicitly verify that client-side writes cannot create verifier access, validate verifier tokens, mutate credential truth, or expose verifier identity.

## Design System Impact

Design System Impact: Medium  
Design Source: `/docs/design_system/README.md`

Codex should review dashboard/verifier surfaces for clarity, safe claims, and absence of deferred launch blockers.

## Acceptance Criteria

- [ ] Dashboard reads backend-owned state and does not expose deferred features as launch blockers.
- [ ] Share-link creation is ownership-checked, entitlement-gated, one-time, short-lived, and raw-token-safe.
- [ ] Verifier validation rejects invalid/expired/used/revoked/malformed tokens safely.
- [ ] Verifier flow requires name, email, and Terms acceptance before credential facts are shown.
- [ ] Verifier display payload excludes private nurse/internal/provider/payment data.
- [ ] Backend writes verifier and verification event records correctly.
- [ ] RLS/direct client access cannot mutate or browse protected verifier/share/credential state.
- [ ] Codex records findings and next required action in GitHub.
